## v1.27.0

**Release-Typ:** Patch Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.27.00/openapi-v01.27.00.json)

### Settings-Felder jetzt in Antworten dokumentiert

Dieses Release dokumentiert Settings-Felder, die die API bereits zurückgibt. Alle Änderungen sind additiv und abwärtskompatibel – am API-Verhalten hat sich nichts geändert, es wurde nur die OpenAPI-Spezifikation an das angepasst, was die Endpunkte bereits senden.

#### Signing-Request-Einstellungen

Das von den Signing-Request-Endpunkten (`POST /signing-requests`, `POST /signing-requests/create-and-send`, `GET /signing-requests`, `GET /signing-requests/{id}`, `PUT /signing-requests/{id}`, `PATCH /signing-requests/{id}`) zurückgegebene Settings-Objekt dokumentiert nun vier zusätzliche Felder:

| Feld | Typ | Beschreibung |
|-------|------|-------------|
| `show_qr_code` | boolean oder null | Zeigt einen QR-Code auf der Signaturseite, sodass Unterzeichner auf ihrem Smartphone fortfahren können. Erbt bei `null` von der Arbeitsbereichs- oder Unternehmenseinstellung. |
| `allow_presigning_download` | boolean oder null | Erlaubt Unterzeichnern, das Originaldokument vor dem Signieren herunterzuladen. Erbt bei `null` von der Arbeitsbereichs- oder Unternehmenseinstellung. |
| `identity_editable_fields` | string[] oder null | Identitätsfelder, die Unterzeichner vor dem Signieren bearbeiten dürfen (z. B. `["name", "company"]`). `null` deaktiviert die Identitätsbearbeitung. |
| `notify_identity_change_email` | boolean | E-Mail-Benachrichtigung senden, wenn ein Unterzeichner seine Identität ändert. |

#### Vorlageneinstellungen

Das von den Vorlagenendpunkten (`POST /templates`, `PUT /templates/{id}`, `PATCH /templates/{id}`, `GET /templates`, `GET /templates/{id}`, `POST /templates/{id}/replace-document`) zurückgegebene Settings-Objekt dokumentiert nun das Feld `show_qr_code`. Vorlageneinstellungen decken dieselben Optionen wie Signing-Request-Einstellungen ab, mit Ausnahme der Signer-Identity-Felder (`identity_editable_fields`, `notify_identity_change_email`), die nur für Signaturanfragen gelten.

#### Cancel- und Resend-Antworten

| Endpunkt | Neues Antwortfeld | Typ | Beschreibung |
|----------|--------------------|------|-------------|
| `POST /signing-requests/{id}/cancel` | `emails_sent` | integer | Anzahl der an Unterzeichner gesendeten Abbruchbenachrichtigungs-E-Mails |
| `POST /signing-requests/{id}/resend` | `recipients_failed` | integer | Anzahl der Empfänger, bei denen der Resend-Benachrichtigungs-Versand fehlgeschlagen ist |

### Neue Schemas

Zwei neue Schemas trennen die Settings-Formen, die sich zwischen Endpunkten unterscheiden, sodass jede Antwort genau die Felder dokumentiert, die sie zurückgibt:

| Schema | Beschreibung |
|--------|-------------|
| `TemplateSettings` | Von den Vorlagenendpunkten zurückgegebene Einstellungen. Entspricht `SigningRequestSettings` ohne die Signer-Identity-Felder (`identity_editable_fields`, `notify_identity_change_email`), die nur für Signaturanfragen gelten. |
| `LegacyDocumentSettings` | Reduzierter Einstellungssatz, der vom veralteten Endpunkt `POST /documents` zurückgegeben wird. Neue Integrationen sollten die Signing-Request-Endpunkte verwenden, die `SigningRequestSettings` zurückgeben. |

### Migrationsleitfaden

Keine Breaking Changes. Es handelt sich um reine Dokumentationsergänzungen, die Felder beschreiben, die die API bereits zurückgibt. Bestehende Integrationen sind nicht betroffen. Wenn Sie Settings-Objekte parsen, können Sie sich jetzt auf die dokumentierten Felder `show_qr_code`, `allow_presigning_download`, `identity_editable_fields` und `notify_identity_change_email` sowie auf `emails_sent` / `recipients_failed` in Cancel- und Resend-Antworten verlassen.

## v1.25.2

**Release-Typ:** Patch Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.25.02/openapi-v01.25.02.json)

### Strengere Validierung für Vorlagen-Feld-Overrides

Beim Erstellen einer Signaturanfrage aus einer Vorlage wird jedes übergebene Feld, das keinem Feld in dieser Vorlage entspricht, jetzt mit **`400` `VALIDATION_ERROR`** abgelehnt, anstatt stillschweigend ignoriert zu werden. Zuvor wurde ein solches Feld verworfen und die Anfrage war weiterhin erfolgreich – eine falsch geschriebene Kennung führte also dazu, dass das Feld ohne Fehlermeldung einfach nicht im Dokument erschien.

Ein übergebenes Feld entspricht einem Vorlagenfeld anhand von `template_field_id`, `variable_name` oder `variable_defined_name`. Der Fehler benennt das/die betreffende(n) Feld(er):

```json
{
  "error": "The following provided fields do not match any field in the template and cannot be applied: full_name_x. Each provided field must reference an existing template field by 'template_field_id' or 'variable_name'.",
  "code": "VALIDATION_ERROR"
}
```

Gilt für `POST /signing-requests` und `POST /signing-requests/create-and-send`.

**Migration:** Stellen Sie sicher, dass jedes Feld, das Sie bei einer vorlagenbasierten Erstellung senden, auf ein existierendes Vorlagenfeld verweist. Wenn Sie sich darauf verlassen haben, dass zusätzliche/nicht zuordenbare Felder ignoriert werden, entfernen Sie diese.

### Case-insensitive Feldtypen

Feld-`type`-Werte werden nun case-insensitiv normalisiert. `"Initials"`, `"Signature"`, `"TextArea"` usw. werden in ihren kanonischen Typ aufgelöst (`initial`, `signature`, `text_area`), anstatt als ungültig abgelehnt zu werden. Kleinschreibungen bleiben unverändert.

## v1.25.1

**Release-Typ:** Patch Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.25.01/openapi-v01.25.01.json)

### Präzise Fehlerantworten für create-and-send

`POST /signing-requests/create-and-send` lieferte zuvor ein generisches **500**, wenn der Send-Schritt eine Anfrage ablehnte, und verbarg damit den wahren Grund. Es liefert nun den **für den Client umsetzbaren Status** (`400`, `402` oder `422`) mit einem maschinenlesbaren `code` und den relevanten Details:

- **`400` `VALIDATION_ERROR`** — Request- oder Send-Validierung fehlgeschlagen (z. B. ein Unterzeichner ohne erforderliche Informationen); die Antwort enthält `validation_errors`, `field_errors` oder `details`, die angeben, was zu beheben ist.
- **`402` `INSUFFICIENT_CREDITS`** — nicht genug Guthaben zum Senden; `current_credits` gibt den Kontostand an.
- **`422` `RECIPIENT_EMAIL_SUPPRESSED`** *(neu)* — die E-Mail-Adresse eines Empfängers ist unterdrückt (zuvor gebounced oder als Spam markiert) und es kann nicht an sie gesendet werden.

Echte serverseitige Fehler liefern weiterhin `500`. Wenn der Send-Schritt fehlschlägt, wird die teilweise erstellte Signaturanfrage zurückgerollt, sodass kein verwaister Entwurf zurückbleibt.

**Migrationsleitfaden:** Keine Aktion erforderlich für Clients, die jede Nicht-2xx-Antwort als Fehler behandeln. Wenn Sie zuvor das undurchsichtige `500` dieses Endpunkts sonderbehandelt haben, wechseln Sie zur Verzweigung nach HTTP-Status/`code`: `400` und `422` bedeuten „Request korrigieren, dann erneut versuchen", `402` bedeutet „Guthaben aufladen".

## v1.25.0

**Release-Typ:** Minor Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.25.00/openapi-v01.25.00.json)

### Editor-Markenfarben

Die Appearance-Einstellungen umfassen jetzt fünf zusätzliche Markenfarben, die die **eingebetteten Editoren** (die Vorlagen- und Signing-Request-Editoren, die Sie auf Ihren eigenen Seiten einbetten) gestalten:

- **`color_accent`** — Akzentfarbe für Hervorhebungen, aktive Tabs, Switches und primäre Bedienelemente im Editor-Chrome.
- **`color_accent_fg`** — Vordergrundfarbe (Text/Symbole) auf Akzentflächen.
- **`color_canvas`** — Umgebungsfarbe hinter der Dokumentseite.
- **`color_muted`** — Gedämpfte Flächenfarbe (Trennlinien, Hover-Zustände, Switch-Tracks).
- **`color_muted_fg`** — Gedämpfte Textfarbe.

Diese ergänzen die bestehenden sechs Farbfelder, sodass Settings-Endpunkte jetzt elf `color_*`-Felder umfassen:

- **`GET`/`PUT /company/settings`** und **`GET`/`PUT /workspace/{workspace_id}/settings`** akzeptieren und liefern alle fünf neuen Felder. Jedes ist ein nullbarer `#rrggbb`-Hex-String; `null` bedeutet Erben (Arbeitsbereich → Unternehmen → Firma-Standard).
- Das aufgelöste **`color_palette`**-Objekt, das in Payloads eingebetteter Editoren zurückgegeben wird, enthält jetzt elf Schlüssel (die sechs bestehenden plus `accent`, `accent_fg`, `canvas`, `muted`, `muted_fg`). Wenn nicht gesetzt, wird `muted` standardmäßig auf die aufgelöste `card`-Farbe gesetzt und `muted_fg` wird zur Lesbarkeit abgeleitet, sodass die Palette immer vollständig gefüllt ist.

Arbeitsbereichs-Werte überschreiben Unternehmenswerte; nicht gesetzte Werte fallen auf die Firma-Standards zurück, sodass bestehende Integrationen unverändert dargestellt werden.

**Migrationsleitfaden:** Keine Aktion erforderlich. Alle fünf Felder sind additiv und nullbar. Um die eingebetteten Editoren zu branden, senden Sie eines der neuen `color_*`-Felder (gültiges `#rrggbb`-Hex oder `null` zum Löschen) an den Company- oder Workspace-Settings-Endpunkt.

## v1.24.0

**Release-Typ:** Minor Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.24.00/openapi-v01.24.00.json)

### Arbeitsbereichs-Test-API-Schlüssel

Arbeitsbereiche legen jetzt neben dem bestehenden Live-Schlüssel einen **Test-Modus-API-Schlüssel** offen. Mit einem Test-Schlüssel authentifizierte Anfragen verbrauchen kein Guthaben und erzeugen als Test markierte, mit einem Wasserzeichen versehene Signaturanfragen.

- **`POST /workspaces`** provisioniert jetzt ein Live+Test-Schlüsselpaar und liefert sowohl `api_key` (live) als auch das neue Feld `test_api_key`.
- **`GET /workspaces/{id}`** und **`GET /workspaces`** liefern jetzt sowohl `api_key` als auch `test_api_key`.
- **`POST /workspaces/{id}/api-key/regenerate`** und **`/api-key/expire`** akzeptieren einen optionalen `key_type` (`"live"` oder `"test"`, Standard `"live"`). Das Regenerieren eines Schlüsseltyps hat keine Auswirkung auf den anderen.
- Über die API erstellte bestehende Arbeitsbereiche werden mit einem Test-Schlüssel nachgezogen.

Alle Änderungen sind additiv und abwärtskompatibel: Konsumenten, die `api_key` lesen, funktionieren unverändert weiter.

**Migrationsleitfaden:** Keine Aktion erforderlich. Um den Testmodus zu nutzen, lesen Sie `test_api_key` aus einer beliebigen Workspace-Create/Get-Antwort und senden Sie ihn als `Authorization`-Schlüssel für Requests, die Sie im Testmodus ausführen möchten.

## v1.22.1

**Release-Typ:** Patch Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.22.01/openapi-v01.22.01.json)

### Löschen von E-Mail-Domains

`DELETE /company/domains/{id}` und `DELETE /workspace/{workspace_id}/domains/{id}` sind jetzt erfolgreich, wenn das Ziel die primäre Domain des Arbeitsbereichs oder seine einzige Domain ist. Zuvor lieferten diese `400 "Cannot delete primary domain"`. Nach dem Löschen fällt der ausgehende E-Mail-Versand auf den Standard-Absender des Unternehmens zurück oder auf den Plattform-Standard, wenn keiner gesetzt ist. Um wieder eine benutzerdefinierte Versanddomain zu verwenden, fügen Sie eine neue Domain hinzu, verifizieren Sie sie und legen Sie sie als primär fest.

### Migrationsleitfaden

Keine Breaking Changes. Die Antworten `400 "Cannot delete primary domain"` und „cannot delete the only domain" wurden aus beiden `DELETE`-Domain-Endpunkten entfernt; sie liefern in diesen Fällen jetzt `200`. Integrationen, die dieses `400` sonderbehandelt haben, können die Behandlung entfernen.

### Migrationsleitfaden

## v1.21.1

### Verwaltung von Arbeitsbereichs-Webhooks

Der Endpunkt `PATCH /workspaces/{id}` unterstützt jetzt zwei neue Felder zur Verwaltung von Arbeitsbereichs-Webhooks über die API:

- `webhook_enabled` (boolean) - Aktiviert oder deaktiviert Webhooks auf Arbeitsbereichsebene. Wenn erstmals auf `true` gesetzt, wird automatisch ein Webhook-Secret erzeugt.
- `ignore_company_webhooks` (boolean) - Wenn `true`, werden Unternehmens-Webhooks für Ereignisse in diesem Arbeitsbereich nicht ausgelöst.

Zuvor konnten diese Einstellungen nur über die Dashboard-UI konfiguriert werden.

### Schemaänderungen

- **PATCH /workspaces/{id}** Request-Body: `webhook_enabled` (boolean, optional) und `ignore_company_webhooks` (boolean, optional) hinzugefügt
- **PATCH /workspaces/{id}** Antwort: enthält jetzt `webhook_enabled`, `webhook_secret`, `webhook_secret_rotated_at`, `webhook_secret_created_at` und `ignore_company_webhooks`

### Migrationsleitfaden

Keine Breaking Changes. Bestehende Arbeitsbereichs-Webhook-Konfigurationen sind nicht betroffen. Verwenden Sie `PATCH /workspaces/{id}` mit `webhook_enabled: true`, um Arbeitsbereichs-Webhooks statt über das Dashboard über die API zu aktivieren.

## v1.21.0

**Release-Typ:** Minor Release

[OpenAPI-Spezifikation herunterladen](/api-reference/v01.21.00/openapi-v01.21.00.json)

### Neue Einstellungen

- `email_local_part` (string, nullbar) - Passt den Local Part (vor dem @) ausgehender Signatur-E-Mails an, wenn eine verifizierte benutzerdefinierte Domain verwendet wird. Auf zwei Ebenen verfügbar:

**Release-Datum:** 5. Mai 2026

### Neue Funktionen

#### Bearbeitbare Identitätsfelder

Signaturanfragen unterstützen jetzt pro-Feld- und pro-Signer-Bearbeitung der Identität. Wenn konfiguriert, sehen Unterzeichner vor dem Signieren einen Bestätigungsdialog, in dem sie festgelegte Identitätsfelder (Name, Firma, Titel, Telefon, Adresse) überprüfen und bearbeiten können.

Neue Einstellungs-Eigenschaft:

| Eigenschaft | Typ | Standard | Beschreibung |
|----------|------|---------|-------------|
| `identity_editable_fields` | `string[]` oder `null` | `null` | Array von Identitätsfeldschlüsseln, die Unterzeichner bearbeiten dürfen. `null` = deaktiviert. |
| `notify_identity_change_email` | `boolean` | `false` | E-Mail senden, wenn ein Unterzeichner seine Identität ändert |

Gültige Feldschlüssel: `name`, `company`, `title`, `phone`, `address`

Empfängerspezifischer Override über `identity_editable_fields` am Recipient-Objekt:
- `null` - Request-Standard verwenden
- `[]` - für diesen Signer deaktiviert
- `["name", "company"]` - benutzerdefinierte Felder für diesen Signer

Beispiel:

```json
{
  "settings": {
    "identity_editable_fields": ["name", "company"]
  },
  "recipients": [
    {
      "name": "John Doe",
      "email": "john@example.com",
      "designation": "signer",
      "order": 1,
      "identity_editable_fields": null
    },
    {
      "name": "Jane Smith",
      "email": "jane@example.com",
      "designation": "signer",
      "order": 2,
      "identity_editable_fields": []
    }
  ]
}
```

In diesem Beispiel erbt John den Request-Standard (Name + Company bearbeitbar), während bei Jane die Identitätsbearbeitung explizit deaktiviert ist.

Neues Webhook-Ereignis: `signing_request.recipient.identity_changed` ist jetzt als Standard-Webhook-Ereignis verfügbar. Abonnieren Sie es in Ihrer Webhook-Konfiguration, um Benachrichtigungen zu erhalten, wenn ein Signer seine Identität während des Signierens ändert.

#### Bearbeitbare vorausgefüllte Datenfelder

Vorausgefüllte Datenfelder können jetzt optional durch Unterzeichner bearbeitet werden. Zuvor waren alle Felder mit gesetztem `prefilled_data` automatisch schreibgeschützt. Eine neue Eigenschaft `prefilled_editable` steuert dieses Verhalten.

Wenn `prefilled_editable` `true` ist, wird der Feldwert weiterhin automatisch aus Empfängerdaten befüllt, aber der Signer kann ihn während des Signierprozesses ändern.

| Eigenschaft | Typ | Standard | Beschreibung |
|----------|------|---------|-------------|
| `prefilled_editable` | `boolean` | `false` | Wenn true, können Unterzeichner vorausgefüllte Werte bearbeiten |

Verfügbar bei:
- Vorlagenfeldern (`PATCH /templates/{id}`, `PUT /templates/{id}`)
- Signing-Request-Feldern (`POST /signing-requests`, `POST /signing-requests/create-and-send`)
- Feld-Response-Objekten (`GET /signing-requests/{id}`)

Beispiel – Signaturanfrage mit bearbeitbarem vorausgefüllten Feld erstellen:

```json
{
  "fields": [
    {
      "type": "text",
      "recipient_id": "temp_1",
      "page_number": 1,
      "x": 10,
      "y": 20,
      "width": 30,
      "height": 5,
      "format_rules": {
        "prefilledData": "full_name",
        "prefilledEditable": true
      }
    }
  ]
}
```

#### Validierung der Feldwertquelle

Die API validiert nun, dass Felder immer nur eine Wertquelle gleichzeitig verwenden. Requests, die `read_only_value`, `prefilled_data` und expliziten `value` auf demselben Feld kombinieren, liefern einen `400 VALIDATION_ERROR`. Diese Durchsetzung gilt für:

- `POST /signing-requests`
- `POST /signing-requests/create-and-send`
- `PATCH /templates/{id}` (Felderstellung und -aktualisierung)
- `PUT /templates/{id}` (Feld-Upsert)

#### Wertpräzedenz

Wenn `prefilled_editable` `true` ist und sowohl ein expliziter `value` als auch `prefilled_data` auf einem Feld gesetzt sind, hat der explizite Wert Vorrang vor den automatisch befüllten Empfängerdaten.

### Migrationsleitfaden

Keine Breaking Changes. Neue Funktionen sind opt-in:
- Identitätsbearbeitung ist standardmäßig deaktiviert (`identity_editable_fields: null`). Setzen Sie sie auf ein Array von Feldschlüsseln, um sie zu aktivieren.
- Bestehende vorausgefüllte Datenfelder bleiben standardmäßig schreibgeschützt. Setzen Sie `prefilled_editable: true` in `format_rules`, um sie bearbeitbar zu machen.

### Neue Funktionen

#### Signaturanfrage-Dokument herunterladen

Ein neuer Endpunkt liefert eine Download-URL für das Dokument einer Signaturanfrage. Bei abgeschlossenen Signaturanfragen zeigt die URL auf das finale signierte PDF. Bei in Bearbeitung befindlichen, abgebrochenen, abgelehnten oder abgelaufenen Anfragen zeigt die URL auf ein Snapshot-PDF, das den Dokumentzustand zum Zeitpunkt der letzten Signer-Aktion erfasst.

**Endpunkt:** `GET /signing-requests/{id}/download`

**Antwortfelder:**

| Feld | Typ | Beschreibung |
|-------|------|-------------|
| `status` | `string` | Status der Signaturanfrage: `finished`, `in_progress`, `cancelled`, `declined` oder `expired` |
| `is_partial` | `boolean` | `true`, wenn das Dokument ein Teil-Snapshot ist (nicht alle Signer haben abgeschlossen) |
| `download_url` | `string` | Pre-signed URL zum PDF-Dokument |
| `generated_at` | `date-time \| null` | Wann das Dokument zuletzt erzeugt wurde |
| `expires_at` | `date-time` | Wann die `download_url` abläuft |

**Verhalten je nach Status:**

| Zustand der Signaturanfrage | Antwort |
|-----------------------|----------|
| Noch nicht gesendet | `409 Conflict` — Download nicht verfügbar |
| Gesendet, Generierung läuft | `503 Service Unavailable` mit `Retry-After`-Header |
| Gesendet, mindestens ein Signer hat abgeschlossen | `200` mit `is_partial: true` |
| Abgebrochen / Abgelehnt / Abgelaufen | `200` mit `is_partial: true` (Snapshot des letzten bekannten Zustands) |
| Abgeschlossen (alle Signer fertig) | `200` mit `is_partial: false` |

Teil-Snapshots werden auf Anfrage erzeugt und zwischengespeichert. Der erste Request kann `503` mit einem `Retry-After`-Header liefern, während das PDF erzeugt wird — versuchen Sie es nach dem angegebenen Intervall erneut.

**URL-Ablauf:** Die `download_url` ist eine Pre-signed URL. Rufen Sie den Endpunkt nach `expires_at` erneut auf, um eine neue URL zu erhalten.

#### Wasserzeichen-Einstellung für Teil-Downloads

Eine neue Arbeitsbereichseinstellung `show_partial_watermark` steuert, ob PDF-Teil-Downloads ein diagonales Wasserzeichen „IN PROGRESS — NOT YET EXECUTED" enthalten. Standardmäßig deaktiviert. Kann auf Unternehmensebene (Standard für alle Arbeitsbereiche) gesetzt oder pro Arbeitsbereich überschrieben werden.

**Endpunkt:** `PUT /workspace/{workspace_id}/settings`

| Feld | Typ | Beschreibung |
|-------|------|-------------|
| `show_partial_watermark` | `boolean \| null` | `true` aktiviert das Wasserzeichen, `false` deaktiviert es, `null` erbt den Unternehmensstandard |

### Migrationsleitfaden

**Von v1.15.3 zu v1.16.0:**

Dies ist eine nicht-breaking, additive Änderung. Bestehende Integrationen funktionieren unverändert weiter.

- Der Download-Endpunkt ist rein opt-in. Am bestehenden Verhalten ändert sich nichts.
- Rufen Sie `GET /signing-requests/{id}/download` auf — behandeln Sie `503` mit `Retry-After`, bis das Dokument bereit ist.

- `POST /templates/{id}/replace-document`

- `GET /signing-requests/{id}/fields`
- `GET /documents` (Legacy)

```json
{

}
```

#### Bedingte Feldlogik

Felder unterstützen jetzt bedingte Regeln sowohl für den **required**-Status als auch für die **Sichtbarkeit**. Zwei neue nullbare Eigenschaften sind an Feldobjekten verfügbar:

- `required_conditions` — Wenn gesetzt, überschreibt sie das statische `required`-Flag. Das Feld ist nur erforderlich, wenn die Bedingungen basierend auf anderen Feldwerten zu true ausgewertet werden.
- `visibility_conditions` — Wenn gesetzt, ist das Feld ausgeblendet, es sei denn, die Bedingungen werden zu true ausgewertet. Ausgeblendete Felder werden beim Absenden nicht validiert.

Bedingungen verwenden eine verschachtelte Gruppenstruktur mit logischen Operatoren:

| Schema | Beschreibung |
|--------|-------------|
| `ConditionSet` | Container der obersten Ebene mit `logic`-Operator (`and`/`or`) und einem Array von `groups` |
| `ConditionGroup` | Enthält ein Array von `conditions`, verknüpft mit der Umkehrung des logic-Operators der Eltern-Ebene |
| `Condition` | Bewertet ein einzelnes Feld anhand von `field_id`, `operator` und optionalem `value` |

**Unterstützte Operatoren:** `is_filled`, `is_empty`, `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `less_than`, `greater_than_or_equal`, `less_than_or_equal`

**Beispiel — Ein Feld nur dann erforderlich machen, wenn ein anderes Feld ausgefüllt ist:**
```json
{
  "required_conditions": {
    "logic": "and",
    "groups": [
      {
        "conditions": [
          {
            "field_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
            "operator": "is_filled"
          }
        ]
      }
    ]
  }
}
```

- Webhook-CRUD: 60 Anfragen/Minute
- Webhook-Test: 10 Anfragen/Minute
- API-Key-Operationen: 1 Anfrage/Minute
- Secret-Rotation: 1 Anfrage/Minute

#### Kern-Schemas

##### Company
- `id`, `company_name`, `account_owner`, `account_owner_email`
- `website`, `icon_url`, `credits` (schreibgeschützt)
- `date_created`, `date_changed`

##### Workspace
- `id`, `name`, `protected`, `api_key`
- `date_created`, `date_changed`

##### Template
- `id`, `name`, `description`
- `document_url` (pre-signed, läuft ab)
- `document_url_expires_at`
- `date_created`, `date_changed`

##### SigningRequest
- `id`, `name`, `description`
- `document_url`, `document_url_expires_at`

- `settings`, `template_id`
- Datumsfelder: `date_created`, `date_sent`, `date_finished`, `date_cancelled`, `expires_at`
- `certificate`-Objekt mit Erzeugungsstatus
- `final_document_download_url`, `final_document_download_error`

##### Recipient
- Kern: `id`, `first_name`, `last_name`, `email`, `designation`, `order`
- Adresse: `street_address`, `city`, `state_province`, `postal_code`, `country`
- Berufliches: `phone_number`, `title`, `company`
- `custom_fields` für zusätzliche Daten
- Unterstützung für temporäre IDs bei dokumentbasierter Erstellung

##### Field
- `id`, `type`, `position`, `page_number`, `required`
- `recipient_id`, `variable_name`
- Typspezifisch: `dropdown_options`, `date_default`, `date_signing_default`
- Schreibgeschützt: `read_only`, `read_only_value`, `prefilled_data`
- Formatierung: `format_rules`, `validation_rules`

##### Webhook
- `id`, `url`, `events`, `enabled`
- `date_created`, `date_changed`
