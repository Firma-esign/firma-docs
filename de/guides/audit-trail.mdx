Jede Signaturanfrage in Firma verfügt über einen vollständigen Audit-Trail. Jede Dokumentansicht, Feldinteraktion, Signatur, Ablehnung und Admin-Aktion wird mit Zeitstempeln, Akteursidentität und IP-Adressen protokolliert.

Häufige Anwendungsfälle:

- **Compliance und rechtliche Beweise** – nachweisen, wer was, wann und von wo unterzeichnet hat
- **Fehlerbehebung bei Unterzeichnern** – genau nachvollziehen, was ein Empfänger während seiner Signatursitzung getan hat
- **Interne Berichterstattung** – Admin-Aktionen wie Erstellung, Bearbeitung und Stornierung nachverfolgen

Der Audit-Trail erfasst zwei Ereigniskategorien:

- **Unterzeichner-Ereignisse** – Dokumentansichten, Feldinteraktionen, Unterzeichnen, Ablehnen, Zertifikat-Downloads
- **Admin-Ereignisse** – Erstellung, Bearbeitungen, Versand, Stornierung, erneutes Senden (über Dashboard oder API)

Jedes Ereignis enthält einen Zeitstempel, eine für Menschen lesbare Beschreibung, die Akteursidentität und (bei Unterzeichner-Ereignissen) die IP-Adresse des Unterzeichners.

## Audit-Trail im Dashboard anzeigen

Öffnen Sie eine beliebige Signaturanfrage im Firma-Dashboard und klicken Sie auf den Tab **Audit Trail**. Sie sehen eine chronologische Zeitleiste aller Ereignisse, wobei Aktionen von Unterzeichnern und Admins visuell unterschieden werden. Aufeinanderfolgende sich wiederholende Ereignisse (wie Scrollen oder wiederholte Feldinteraktionen) werden automatisch mit einer Zählanzeige zusammengefasst.

## Audit-Trail über die API abrufen

Verwenden Sie den Audit-Endpunkt, um den vollständigen Ereignisverlauf einer Signaturanfrage abzurufen:

```
GET /signing-requests/{id}/audit
```

### curl

```bash
curl -X GET "https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit" \
  -H "Authorization: YOUR_API_KEY"
```

### Node.js

```js
const response = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequestId}/audit`,
  {
    headers: {
      'Authorization': process.env.FIRMA_API_KEY,
    },
  }
);

const { results } = await response.json();
console.log(results);
```

### Python

```python
import os
import requests

response = requests.get(
    f"https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit",
    headers={
        "Authorization": os.environ["FIRMA_API_KEY"],
    },
)

results = response.json()["results"]
```

### Antwortfelder

| Feld | Typ | Beschreibung |
|------|-----|--------------|
| `id` | `uuid` | Ereignis-ID |
| `timestamp` | `datetime` | Wann das Ereignis auftrat |
| `source` | `string` | `"signer"` oder `"admin"` |
| `event` | `string` | Ereignistyp-Bezeichner (siehe [Ereignisreferenz](#event-reference) unten) |
| `description` | `string` | Für Menschen lesbare Ereignisbeschreibung |
| `actor` | `object \| null` | Wer die Aktion ausgeführt hat (siehe unten) |
| `ip_address` | `string \| null` | IP-Adresse des Unterzeichners (nur Unterzeichner-Ereignisse) |
| `details` | `object \| null` | Ereignisspezifische Metadaten |
| `condensed_count` | `integer` | Vorhanden, wenn aufeinanderfolgende identische Ereignisse desselben Akteurs gruppiert werden |

## Ereignisquellen verstehen

Das Feld `source` gibt an, ob ein Ereignis von einem Unterzeichner oder einem Admin stammt, und das Feld `actor` hat jeweils eine andere Struktur.

### Unterzeichner-Ereignisse

Ereignisse mit `source: "signer"` sind Aktionen, die Empfänger während des Signaturflusses ausführen. Das `actor`-Objekt enthält `name` und `email` des Unterzeichners. Das Feld `ip_address` ist gefüllt.

```json
{
  "source": "signer",
  "actor": { "name": "Alice Johnson", "email": "alice@example.com" },
  "ip_address": "203.0.113.42"
}
```

### Admin-Ereignisse

Ereignisse mit `source: "admin"` sind Aktionen, die über das Dashboard oder die API ausgeführt werden. Das `actor`-Objekt enthält ein Feld `type` (`"api_key"` oder `"user"`) und optional eine `user_id`. Das Feld `ip_address` ist `null`.

```json
{
  "source": "admin",
  "actor": { "type": "api_key" },
  "ip_address": null
}
```

```json
{
  "source": "admin",
  "actor": { "type": "user", "user_id": "usr_abc123" },
  "ip_address": null
}
```

## Ereignisreferenz

### Dokumentlebenszyklus

| Ereignis | Beschreibung | Hat `details` |
|----------|--------------|---------------|
| `document_opened` | Dokument geöffnet | Nein |
| `document_viewed` | Dokument angezeigt | Nein |
| `document_saved` | Feldwerte gespeichert | Nein |
| `document_finished` | Unterzeichnung abgeschlossen | Nein |
| `document_signed` | Dokument unterzeichnet | Nein |
| `document_hidden` | Seite verlassen | Nein |
| `document_visible` | Zur Seite zurückgekehrt | Nein |
| `document_closed` | Dokument geschlossen | Nein |

### Signatur und Verifizierung

| Ereignis | Beschreibung | Hat `details` |
|----------|--------------|---------------|
| `signature_finalized` | Signatur finalisiert | Nein |
| `signing_declined` | Unterzeichnung abgelehnt | Nein |
| `terms_accepted` | Geschäftsbedingungen akzeptiert | Nein |
| `otp_verified` | Identität per OTP verifiziert | Nein |
| `certificate_download` | Zertifikat heruntergeladen | Nein |

### Feldinteraktionen

Feldinteraktionen werden als `field_interaction`-Ereignisse erfasst. Das `details`-Objekt enthält die konkrete Aktion und Feld-Metadaten.

| Aktion | Beschreibung |
|--------|--------------|
| `field_focused` | Feld ausgewählt |
| `field_completed` | Feld erstmals ausgefüllt |
| `field_modified` | Vorhandenen Feldwert geändert |
| `field_cleared` | Feldwert gelöscht |
| `field_blur` | Bearbeitung eines Felds beendet |

Das `details`-Objekt für Feldinteraktionen enthält:

| Feld | Typ | Beschreibung |
|------|-----|--------------|
| `field_type` | `string` | Feldtyp (Signatur, Text, Datum usw.) |
| `action` | `string` | Die konkrete Aktion aus der obigen Tabelle |
| `interaction_count` | `integer` | Anzahl der Interaktionen mit diesem Feld |
| `value_length` | `integer` | Länge des Feldwerts (sofern zutreffend) |
| `time_spent_ms` | `integer` | Auf dem Feld verbrachte Zeit in Millisekunden (sofern zutreffend) |

### Navigation

| Ereignis | Beschreibung | Hat `details` |
|----------|--------------|---------------|
| `page_viewed` | Bestimmte Seite angezeigt | Ja – `page_number` |
| `document_scrolled` | Zu einer Seite gescrollt | Ja – `page_number` |
| `navigation_action` | Im Dokument navigiert | Ja – `action` (`next_page`, `prev_page`, `go_to_page`) |
| `zoom_changed` | Zoomstufe geändert | Nein |
| `modal_interaction` | Dialog geöffnet oder geschlossen | Ja – `modal_type`, `action` |

### Admin-Aktivität

Admin-Ereignisse verwenden `event: "activity"` mit einem für Menschen lesbaren `description`-Feld, das die durchgeführte Aktion beschreibt (z. B. "Created signing request via API", "Sent signing request", "Cancelled signing request").
