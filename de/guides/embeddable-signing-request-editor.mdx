# Einbettbarer Signaturanfragen-Editor

Betten Sie den Signaturanfragen-Editor von Firma in Ihre Anwendung ein und nutzen Sie JWT-Authentifizierung für sicheren, zeitlich begrenzten Zugriff. Ideal für White-Label-Integrationen und Multi-Tenant-Anwendungen.

## Anwendungsfälle

- **White-Label-Signaturabläufe**: Signaturanfragen unter Ihrer Marke verwalten
- **Multi-Tenant-Anwendungen**: Sicherer Zugriff pro Unterzeichner, ohne API-Schlüssel offenzulegen
- **Eingebettete Dokument-Workflows**: Nahtlose Verwaltung von Signaturanfragen in Ihrem Produkt
- **Zeitlich begrenzter Zugriff**: Tokens laufen aus Sicherheitsgründen automatisch ab (7 Tage Gültigkeit)

## So funktioniert es

1. Ihr Server fordert mit Ihrem API-Schlüssel ein JWT-Token von der Firma-API an
2. Firma gibt ein kurzlebiges JWT-Token mit der Signing-Request-ID zurück
3. Ihr Frontend bettet den Editor mit dem JWT-Token ein
4. Das Token läuft automatisch nach 7 Tagen ab

<Note>
**Rate-Limit**: JWT-Endpunkte unterstützen **100 Anfragen pro Minute pro API-Schlüssel** für Anwendungen mit hohem Volumen.
</Note>

## JWT-Authentifizierung

### JWT-Token generieren

Generieren Sie ein JWT-Token für eine bestimmte Signaturanfrage über den Endpunkt `/jwt/generate-signing-request`.

**Endpunkt**: `POST /jwt/generate-signing-request`

**Request-Body**:
```json
{
  "companies_workspaces_signing_requests_id": "123e4567-e89b-12d3-a456-426614174000"
}
```

**Antwort (200 OK)**:
```json
{
  "jwt": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "jwt_id": "jwt123-e89b-12d3-a456-426614174000",
  "expires_at": "2024-04-27T10:00:00Z",
  "signing_request_id": "123e4567-e89b-12d3-a456-426614174000",
  "created_at": "2024-04-20T10:00:00Z"
}
```

**Rate-Limit-Header**:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 99
X-RateLimit-Reset: 1729180800
```

## Implementierungsleitfaden

<Warning>
**Sicherheit**: Legen Sie Ihren API-Schlüssel niemals im clientseitigen Code offen. Generieren Sie JWT-Tokens stets von Ihrem sicheren Backend aus.
</Warning>

### Backend: JWT-Token generieren

Rufen Sie die Firma-API aus Ihrem Backend auf, um ein JWT-Token zu generieren. Ihr Backend-Endpunkt sollte eine Signing-Request-ID akzeptieren und das JWT an Ihr Frontend zurückgeben.

**Node.js-Beispiel**:
```js
// Example: Simple backend function to generate JWT
async function generateSigningRequestToken(signingRequestId) {
  const response = await fetch('https://api.firma.dev/functions/v1/signing-request-api/jwt/generate-signing-request', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      companies_workspaces_signing_requests_id: signingRequestId
    })
  })
  
  const data = await response.json()
  return data.jwt
}
```

**Python-Beispiel**:
```py
import os
import requests

def generate_signing_request_token(signing_request_id):
    response = requests.post(
        'https://api.firma.dev/functions/v1/signing-request-api/jwt/generate-signing-request',
        headers={
            'Authorization': f'Bearer {os.getenv("FIRMA_API_KEY")}',
            'Content-Type': 'application/json'
        },
        json={
            'companies_workspaces_signing_requests_id': signing_request_id
        }
    )
    data = response.json()
    return data['jwt']
```

### Frontend-Implementierung – HTML / Vanilla-JavaScript

```html
<!-- Load the Firma Signing Request Editor library -->
<script src="https://api.firma.dev/functions/v1/embed-proxy/signing-request-editor.js"></script>

<!-- Create a container for the editor -->
<div id="signing-request-editor-container"></div>

<script>
  // Assuming you've already fetched the JWT from your backend
  const jwt = 'eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...';
  const signingRequestId = '08f72e3b-79a8-4eac-a268-b3f9efaf6573';
  
  // Initialize the signing request editor
  const editor = new FirmaSigningRequestEditor({
    container: '#signing-request-editor-container',
    jwt: jwt,
    signingRequestId: signingRequestId,
    showCloseButton: true, // Show close button in header
    onSave: (data) => {
      console.log('Signing request saved:', data);
    },
    onSend: (data) => {
      console.log('Signing request sent:', data);
    },
    onClose: (data) => {
      if (data.hasUnsavedChanges) {
        // Prompt user to save before leaving
      }
      console.log('Editor closed, unsaved changes:', data.hasUnsavedChanges);
    },
    onError: (error) => {
      console.error('Editor error:', error);
    },
    onLoad: (signingRequest) => {
      console.log('Editor loaded successfully:', signingRequest);
    }
  });
</script>
```

### Frontend-Implementierung – React

```jsx
import { useEffect, useRef, useState } from 'react';

function SigningRequestEditorComponent({ signingRequestId, jwt }) {
  const containerRef = useRef<HTMLDivElement>(null);
  const editorRef = useRef<any>(null);
  const [isLoaded, setIsLoaded] = useState(false);

  // Load the Firma Signing Request Editor library
  useEffect(() => {
    const script = document.createElement('script');
    script.src = 'https://api.firma.dev/functions/v1/embed-proxy/signing-request-editor.js';
    script.async = true;
    
    script.onload = () => setIsLoaded(true);
    document.body.appendChild(script);
    
    return () => {
      if (document.body.contains(script)) {
        document.body.removeChild(script);
      }
    };
  }, []);

  // Initialize the editor when library and JWT are ready
  useEffect(() => {
    if (!isLoaded || !containerRef.current || !jwt) return;

    editorRef.current = new window.FirmaSigningRequestEditor({
      container: containerRef.current,
      jwt: jwt,
      signingRequestId: signingRequestId,
      showCloseButton: true, // Show close button in header
      onSave: (data) => console.log('Saved:', data),
      onSend: (data) => console.log('Sent:', data),
      onClose: (data) => {
        if (data.hasUnsavedChanges) {
          // Prompt user to save before leaving
        }
      },
      onError: (error) => console.error('Error:', error),
      onLoad: (signingRequest) => console.log('Loaded:', signingRequest)
    });

    return () => {
      if (editorRef.current?.destroy) {
        editorRef.current.destroy();
      }
    };
  }, [isLoaded, jwt, signingRequestId]);

  return <div ref={containerRef} className="w-full h-full" />;
}

export default SigningRequestEditorComponent;
```

### Konfigurationsoptionen

| Option | Typ | Beschreibung |
|--------|-----|--------------|
| `container` | HTMLElement | DOM-Element, in dem der Editor gemountet wird (erforderlich) |
| `jwt` | string | Authentifizierungstoken von Ihrem Backend (erforderlich) |
| `signingRequestId` | string | Kennung der Signaturanfrage (erforderlich) |
| `theme` | `'dark'` \| `'light'` | Editor-Theme (Standard: `'dark'`) |
| `readOnly` | boolean | Nur-Lese-Modus aktivieren (Standard: `false`) |
| `height` | string | Container-Höhe (Standard: `'100vh'`) |
| `width` | string | Container-Breite (Standard: `'100%'`) |
| `showCloseButton` | boolean | Schließen-Button im Editor-Header anzeigen (Standard: `false`) |
| `onSave` | function | Callback, wenn die Signaturanfrage gespeichert wird |
| `onSend` | function | Callback, wenn die Signaturanfrage gesendet wird |
| `onClose` | function | Callback, wenn der Schließen-Button geklickt oder `triggerClose()` aufgerufen wird. Erhält `{ hasUnsavedChanges: boolean }` |
| `onError` | function | Callback für Fehlerbehandlung |
| `onLoad` | function | Callback, wenn der Editor mit den Daten der Signaturanfrage geladen wird |

### Instanzmethoden

| Methode | Beschreibung |
|---------|--------------|
| `triggerClose()` | Löst den Schließen-Ablauf programmatisch aus. Prüft auf ungespeicherte Änderungen und ruft `onClose` mit `{ hasUnsavedChanges: boolean }` auf |

Verwenden Sie `triggerClose()`, wenn Ihre Host-Anwendung den Editor aus ihrer eigenen Oberfläche schließen muss – zum Beispiel bei einem übergeordneten Navigationsereignis oder einem „Zurück"-Button:

```js
// Close the editor from your app's UI
document.getElementById('my-back-button').addEventListener('click', () => {
  editor.triggerClose();
});
```

## postMessage-Ereignisse (Editor → Host)

Der Signaturanfragen-Editor von Firma sendet postMessage-Ereignisse für wichtige Lebenszyklus-Aktionen. Nachfolgend ein empfohlenes, minimales Ereignisschema, das Sie implementieren können, um auf Speichern- und Sende-Ereignisse des Editors zu reagieren.

<Tip>
Nutzen Sie diese Ereignisse, um die Editor-Aktivität zu verfolgen, ohne zusätzliche API-Aufrufe zu machen – so bleiben Sie innerhalb der Rate-Limits.
</Tip>

Ereignis-Envelope (window.postMessage-Payload):

```json
{
	"type": "editor.event",
	"event": "signing_request.saved", // or signing_request.sent, signing_request.closed
	"payload": {
		"signing_request_id": "sr_123",
		"updated_at": "2025-09-04T12:34:56Z",
		"status": "draft" // or "sent"
	}
}
```

### Beispiel: Client-Listener (Plain JS)

```js
window.addEventListener('message', (ev) => {
	// 1) Validate origin
	if (ev.origin !== 'https://app.firma.dev') return
	// 2) Validate shape
	const data = ev.data || {}
	if (data.type !== 'editor.event') return

	switch (data.event) {
		case 'signing_request.saved':
			console.log('Signing request saved', data.payload)
			// Optionally fetch the latest signing request via API
			break
		case 'signing_request.sent':
			console.log('Signing request sent', data.payload)
			// Trigger webhook or notification
			break
		case 'signing_request.closed':
			console.log('Editor closed')
			break
		default:
			break
	}
})
```

## Verwaltung des Token-Lebenszyklus

JWT-Tokens werden mit einer Gültigkeitsdauer von 7 Tagen für typische Bearbeitungs-Workflows generiert. Der Editor verarbeitet Token-Ablauf automatisch.

### Automatischer Ablauf

JWT-Tokens laufen basierend auf dem `expires_at`-Zeitstempel automatisch nach 7 Tagen ab. Nach Ablauf:
- Der eingebettete Editor lehnt das Token ab
- Ein neues Token muss angefordert werden, um fortzufahren
- Kein API-Aufruf nötig – Tokens laufen passiv ab

### Token-Erneuerung (optional)

Für lang andauernde Bearbeitungssitzungen können Sie das Token vor Ablauf mit der Methode `updateJWT()` erneuern:

```js
// Generate a new JWT from your backend
const newJwt = await fetch('/your-backend/generate-token').then(r => r.json());

// Update the editor with the new token
editor.updateJWT(newJwt.jwt);
```

## Rate-Limiting

JWT-Endpunkte unterstützen **100 Anfragen pro Minute pro API-Schlüssel**.

**Rate-Limit-Header**:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1729180800
```

Wenn Sie das Rate-Limit überschreiten (429-Antwort):
- Cachen Sie Tokens und verwenden Sie sie bis zum Ablauf wieder (7 Tage)
- Implementieren Sie exponentielles Backoff für Retries
- Überwachen Sie den Header `X-RateLimit-Remaining`
- Generieren Sie Tokens on-demand, nicht präventiv

## Sicherheits-Best-Practices

<Warning>
**Legen Sie Ihren API-Schlüssel niemals im clientseitigen Code offen.** Generieren Sie JWT-Tokens stets über einen sicheren Server-Endpunkt.
</Warning>

### ✅ Do's

- ✅ Tokens vom Backend-Server generieren
- ✅ Rate-Limits überwachen (100 Anfragen/Minute)
- ✅ HTTPS für alle API-Anfragen verwenden
- ✅ `readOnly: true` für reinen Ansichtszugriff verwenden

### ❌ Don'ts

- ❌ API-Schlüssel nicht im Frontend-Code offenlegen
- ❌ Tokens nicht über Sitzungen oder Unterzeichner hinweg wiederverwenden
- ❌ JWT-Tokens nicht loggen (Sicherheitsrisiko)
- ❌ Tokens nicht zwischen verschiedenen Signaturanfragen teilen

## Fehlerbehebung

### Fehler: Token abgelaufen

**Symptom**: Editor zeigt „Token expired" oder Authentifizierungsfehler an

**Lösung**: 
- Token-Erneuerung vor Ablauf implementieren (7-Tage-Fenster)
- Neues Token generieren und `editor.updateJWT(newToken)` aufrufen
- Synchronisation der Systemuhr prüfen

### 401 Unauthorized

**Symptom**: JWT-Generierung schlägt mit 401 fehl

**Mögliche Ursachen**:
- Ungültiger oder fehlender API-Schlüssel
- API-Schlüssel hat nicht die erforderlichen Berechtigungen
- API-Schlüssel ist deaktiviert

**Lösung**: API-Schlüssel im Dashboard überprüfen und Berechtigungen prüfen

### 404 Not Found

**Symptom**: Signaturanfrage nicht gefunden beim Generieren eines JWTs

**Mögliche Ursachen**:
- Signing-Request-ID existiert nicht
- Signaturanfrage gehört zu einem anderen Workspace
- Signaturanfrage wurde gelöscht

**Lösung**: Signing-Request-ID und Workspace-Zugriff überprüfen

### Rate-Limit überschritten

**Symptom**: 429 Too Many Requests

**Lösung**:
- Token-Caching implementieren (7-Tage-Ablauffenster ist großzügig)
- Auf Rate-Limit-Reset warten (Header `X-RateLimit-Reset` prüfen)
- Retry-Logik mit exponentiellem Backoff implementieren

### Editor lädt nicht

**Symptom**: Container ist leer oder zeigt dauerhaft einen Lade-Spinner

**Mögliche Ursachen**:
- Skript nicht geladen (`script.onload`-Ereignis prüfen)
- Ungültiges JWT oder Signing-Request-ID
- CORS-Probleme (Browser-Konsole prüfen)

**Lösung**:
- Prüfen, ob das Skript von `https://app.firma.dev/embed/signing-request-editor.js` geladen wird
- JWT auf Gültigkeit und nicht abgelaufen prüfen
- Sicherstellen, dass das Backend korrekte CORS-Header zurückgibt
