Firma bietet umfassende White-Labeling-Funktionen, mit denen Sie ein vollständig gebrandetes E-Signatur-Erlebnis schaffen können. Von benutzerdefinierten Logos und Farbthemen bis hin zu eingebetteten Oberflächen können Sie sicherstellen, dass Ihre Kunden während des gesamten Signaturprozesses mit Ihrer Marke interagieren.

## Übersicht

White-Labeling in Firma umfasst mehrere Komponenten:

1. **Benutzerdefiniertes Branding** – Logos hochladen, Farbthemen festlegen und Firma-Branding vollständig ausblenden
2. **Benutzerdefinierte Unterzeichner-Bedingungen** – Unterzeichner müssen Ihre eigenen Nutzungsbedingungen akzeptieren
3. **Beschriftungen der Signaturschaltflächen** – Passen Sie den Text auf Signaturschaltflächen pro Sprache an
4. **Benutzerdefinierte E-Mail-Domains** – Versenden Sie Signaturanfrage-E-Mails von Ihrer eigenen Domain
5. **Benutzerdefinierte Absender-E-Mail-Adresse** – Steuern Sie die „From"-Adresse auf allen ausgehenden E-Mails
6. **Benutzerdefinierte E-Mail-Vorlagen** – Personalisieren Sie Inhalt und Branding der Signaturbenachrichtigungs-E-Mails
7. **Firma-E-Mails deaktivieren** – Schalten Sie automatische E-Mails pro Signaturanfrage aus und versenden Sie eigene
8. **Eingebettete Erlebnisse** – Betten Sie Signatur- und Vorlagen-Editoren direkt in Ihre App ein

<Note>
  Alle API-Beispiele verwenden Ihren API-Schlüssel direkt im `Authorization`-Header. Verwenden Sie kein `Bearer`-Präfix.
</Note>

## Benutzerdefiniertes Branding

Passen Sie Logos, Farben und die Sichtbarkeit des Brandings sowohl auf Unternehmens- als auch auf Arbeitsbereichsebene an. Arbeitsbereichseinstellungen überschreiben Unternehmenseinstellungen, sodass Sie für jeden Ihrer Kunden unterschiedliche gebrandete Erlebnisse schaffen können.

### Unternehmenslogo

Laden Sie ein Logo für Ihr gesamtes Firma-Konto hoch. Dieses Logo erscheint in Signatur-E-Mails und im Signaturerlebnis.

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/logo.png"
```

**Anforderungen:**
- Format: nur PNG oder JPEG
- Maximale Größe: 2 MB

So entfernen Sie das Unternehmenslogo:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Arbeitsbereichslogo

Überschreiben Sie das Unternehmenslogo für einen bestimmten Arbeitsbereich:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/workspace-logo.png"
```

So entfernen Sie ein Arbeitsbereichslogo (fällt auf das Unternehmenslogo zurück):

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Farbthematisierung

Passen Sie die Farbpalette des Signaturerlebnisses mit Hex-Farbwerten an. Farben können sowohl auf Unternehmens- als auch auf Arbeitsbereichsebene gesetzt werden.

| Farbeinstellung | Beschreibung | Firma-Standard |
|---------------|-------------|---------------|
| `color_primary` | Primäre Akzentfarbe (Buttons, Links) | `#c285ff` |
| `color_primary_fg` | Textfarbe auf primären Elementen | `#ffffff` |
| `color_background` | Seitenhintergrund | `#1c1c21` |
| `color_foreground` | Primäre Textfarbe | `#ffffff` |
| `color_card` | Karten-/Panel-Hintergrund | `#22222a` |
| `color_border` | Rahmenfarbe | `#3b3b3b` |
| `color_accent` | Sekundäre Akzentfarbe | `#c285ff` |
| `color_accent_fg` | Textfarbe auf Akzentelementen | `#ffffff` |
| `color_canvas` | Dokument-Canvas-/Viewer-Hintergrund | `#1c1c21` |
| `color_muted` | Gedämpfte/dezente Hintergrundbereiche | `#22222a` |
| `color_muted_fg` | Textfarbe in gedämpften Bereichen | `#a1a1aa` |

**Farben auf Unternehmensebene setzen:**

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#0066cc",
    "color_primary_fg": "#ffffff",
    "color_background": "#f5f5f5",
    "color_foreground": "#1a1a1a",
    "color_card": "#ffffff",
    "color_border": "#e0e0e0"
  }'
```

**Farben auf Arbeitsbereichsebene setzen** (überschreibt Unternehmensfarben für diesen Arbeitsbereich):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#ff6600",
    "color_primary_fg": "#ffffff"
  }'
```

Farben müssen 6-stellige Hex-Werte sein (z. B. `#0066cc`). Setzen Sie eine Farbe auf `null`, um sie zu entfernen und auf die nächste Ebene in der Hierarchie zurückzufallen.

### Firma-Branding ausblenden

Entfernen Sie das gesamte Firma-Branding aus dem Signaturerlebnis, indem Sie `show_custom_branding_only` auf Unternehmensebene aktivieren:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "show_custom_branding_only": true
  }'
```

### Weitere Anzeigeeinstellungen

Feineinstellungen für das Signaturerlebnis, verfügbar sowohl auf Unternehmens- als auch auf Arbeitsbereichsebene:

| Einstellung | Beschreibung | Standard |
|---------|-------------|---------|
| `show_signature_frame` | Signatur-Zeichenrahmen anzeigen | `true` |
| `show_partial_watermark` | Wasserzeichen auf teilweise signierten Dokumenten anzeigen | `false` |
| `show_qr_code` | QR-Code mit Link zur Signaturseite in E-Mails einfügen | `false` |
| `require_otp_verification` | E-Mail-OTP-Verifikation vor dem Signieren erforderlich | `false` |
| `require_terms_acceptance` | Unterzeichner müssen Bedingungen vor dem Signieren akzeptieren | `true` |
| `disable_guided_navigation` | Schritt-für-Schritt-Feldnavigation deaktivieren | `false` |
| `allow_presigning_download` | Unterzeichnern erlauben, das Dokument vor dem Signieren herunterzuladen | `false` |

Setzen Sie diese auf Arbeitsbereichsebene auf `null`, um die Unternehmenseinstellung zu erben.

### Einstellungshierarchie

Branding-Einstellungen folgen einer kaskadierenden Hierarchie:

1. **Arbeitsbereichseinstellung** (höchste Priorität)
2. **Unternehmenseinstellung**
3. **Firma-Standard** (niedrigste Priorität)

So können Sie unternehmensweite Standards festlegen und diese pro Arbeitsbereich überschreiben. Auf Arbeitsbereichsebene bedeutet ein Wert von `null` „von Unternehmen erben".

Siehe die [API-Referenz zu Arbeitsbereichseinstellungen](/api-reference/v01.27.00/workspace-settings/get-workspace-settings) für die vollständige Liste konfigurierbarer Einstellungen.

## Benutzerdefinierte Unterzeichner-Bedingungen

Fordern Sie Unterzeichner auf, Ihre eigenen Nutzungsbedingungen oder rechtlichen Vereinbarungen zu akzeptieren, bevor sie unterschreiben. Bedingungen werden pro Sprache konfiguriert und folgen derselben Unternehmens-/Arbeitsbereichs-Kaskade wie andere Einstellungen.

### Funktionsweise

Wenn `require_terms_acceptance` aktiviert ist (Standard), sehen Unterzeichner ein Kontrollkästchen zur Zustimmung der Bedingungen, bevor sie unterschreiben können. Sie können den Erklärungstext anpassen und optional auf ein vollständiges Bedingungsdokument verlinken.

Bedingungen werden pro Sprache gespeichert, sodass Unterzeichner die Bedingungen in der Sprache ihres Arbeitsbereichs sehen und über die Kaskade zurückfallen:

1. **Arbeitsbereichsbedingungen** für die Sprache des Unterzeichners (höchste Priorität)
2. **Unternehmensbedingungen** für die Sprache des Unterzeichners
3. **Firma-Standardbedingungen** (niedrigste Priorität)

### Bedingungen auf Unternehmensebene setzen

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acme.com/terms\">Terms of Service</a> and acknowledge that my electronic signature is legally binding.",
    "terms_url": "https://acme.com/terms"
  }'
```

### Bedingungen auf Arbeitsbereichsebene setzen

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acmecorp.com/signing-terms\">Signing Terms</a>.",
    "terms_url": "https://acmecorp.com/signing-terms"
  }'
```

**Felder:**
- `statement_text` (erforderlich): HTML-Text, der neben dem Zustimmungs-Kontrollkästchen angezeigt wird. Grundlegende HTML-Tags sind erlaubt (`<a>`, `<b>`, `<i>`, `<br>`).
- `terms_url` (optional): URL zum vollständigen Bedingungsdokument.

### Bedingungen auflisten

Alle konfigurierten Bedingungen für ein Unternehmen oder einen Arbeitsbereich abrufen:

```bash
# Company terms
curl https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms \
  -H "Authorization: YOUR_API_KEY"

# Workspace terms
curl https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms \
  -H "Authorization: YOUR_API_KEY"
```

### Bedingungen löschen

Entfernen Sie benutzerdefinierte Bedingungen für eine bestimmte Sprache, um auf die nächste Ebene in der Kaskade zurückzufallen:

```bash
# Delete company terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"

# Delete workspace terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"
```

### Zustimmung zu Bedingungen deaktivieren

Um die Zustimmungsanforderung vollständig zu deaktivieren:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "require_terms_acceptance": false
  }'
```

Unterstützte Sprachen: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

Siehe die [API-Referenz zu Signer Terms](/api-reference/v01.27.00/signer-terms/list-company-signer-terms) für die vollständige Endpunktdokumentation.

## Benutzerdefinierte Beschriftungen für Signaturschaltflächen

Passen Sie den Text auf Signaturschaltflächen pro Arbeitsbereich und Sprache an. So können Sie die Beschriftungen an die Terminologie oder Lokalisierung Ihres Produkts anpassen.

### Beschriftungs-Overrides setzen

Schaltflächenbeschriftungen werden als JSONB-Objekt in den Arbeitsbereichseinstellungen gesetzt, indiziert nach Sprache und dann nach Schaltflächenschlüssel:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": {
      "en": {
        "sign": "Approve and Sign",
        "next": "Continue",
        "finish": "Complete Signing"
      },
      "es": {
        "sign": "Aprobar y Firmar",
        "next": "Continuar",
        "finish": "Completar Firma"
      }
    }
  }'
```

### Beschriftungs-Overrides löschen

Auf `null` setzen, um zu den Standardbeschriftungen von Firma zurückzukehren:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": null
  }'
```

**Validierung:**
- Jede Beschriftung muss ein String mit maximal 500 Zeichen sein
- Sprachschlüssel müssen gültige Sprachcodes sein (max. 10 Zeichen)
- HTML-Tags und Steuerzeichen werden automatisch entfernt

<Note>
  Beschriftungs-Overrides sind ausschließlich eine Einstellung auf Arbeitsbereichsebene. Es gibt keinen Override auf Unternehmensebene.
</Note>

## Benutzerdefinierte E-Mail-Domains

Standardmäßig werden Signaturanfrage-E-Mails von der Firma-Domain gesendet. Mit benutzerdefinierten E-Mail-Domains scheinen E-Mails direkt von Ihrem Unternehmen oder den Unternehmen Ihrer Kunden zu kommen.

### E-Mail-Domains auf Kontoebene (Unternehmen)

Richten Sie eine benutzerdefinierte E-Mail-Domain für Ihr gesamtes Firma-Konto ein. Alle Arbeitsbereiche verwenden diese Domain standardmäßig, sofern sie nicht überschrieben wird.

#### Schritt 1: Ihre Domain hinzufügen

Verwenden Sie die API, um eine benutzerdefinierte E-Mail-Domain hinzuzufügen:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "acme.com"
  }'
```

**Antwort (201 Created)**:

```json
{
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 0,
    "domain_status": 0,
    "is_primary": false,
    "verification_token": "firma-verify=abc123xyz",
    "date_created": "2024-01-15T10:30:00Z"
  },
  "verification_instructions": {
    "record_type": "TXT",
    "record_name": "_firma-verification.acme.com",
    "record_value": "firma-verify=abc123xyz",
    "next_step": "Add this TXT record to your DNS, then call POST /company/domains/{id}/verify-ownership"
  }
}
```

#### Schritt 2: TXT-Verifikationseintrag hinzufügen

Fügen Sie den Verifikations-TXT-Eintrag zu Ihrem DNS hinzu:

| Type | Name                           | Value                    |
| ---- | ------------------------------ | ------------------------ |
| TXT  | `_firma-verification.acme.com` | `firma-verify=abc123xyz` |

<Note>
  Die DNS-Propagation dauert typischerweise wenige Minuten, kann jedoch bis zu 48 Stunden dauern.
</Note>

#### Schritt 3: Domain-Eigentümerschaft verifizieren

Sobald der TXT-Eintrag hinzugefügt ist, verifizieren Sie die Eigentümerschaft:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-ownership \
  -H "Authorization: YOUR_API_KEY"
```

**Antwort**:

```json
{
  "message": "Domain ownership verified",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 1,
    "domain_status": 0
  },
  "next_step": "Call POST /company/domains/{id}/finalize to complete domain setup and receive DNS records for email sending"
}
```

#### Schritt 4: Domain-Einrichtung abschließen

Nachdem die Eigentümerschaft verifiziert ist, schließen Sie die Domain ab, um die DNS-Einträge für den E-Mail-Versand zu erhalten:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/finalize \
  -H "Authorization: YOUR_API_KEY"
```

**Antwort**:

```json
{
  "message": "Domain finalized. Add the following DNS records to enable email sending.",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 0
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "ttl": "Auto", "status": "pending" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "ttl": "Auto", "status": "pending" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "ttl": "Auto", "status": "pending" }
  ],
  "next_step": "Add these DNS records, then call POST /company/domains/{id}/verify-dns to complete verification"
}
```

#### Schritt 5: DNS-Einträge hinzufügen

Fügen Sie alle drei DNS-Einträge zu Ihrer Domain hinzu:

| Type  | Name                | Value                               |
| ----- | ------------------- | ----------------------------------- |
| TXT   | `@`                 | `v=spf1 include:amazonses.com ~all` |
| CNAME | `resend._domainkey` | `resend._domainkey.amazonses.com`   |
| TXT   | `_dmarc`            | `v=DMARC1; p=none;`                 |

#### Schritt 6: DNS-Einträge verifizieren

Sobald die DNS-Einträge hinzugefügt sind, verifizieren Sie sie:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-dns \
  -H "Authorization: YOUR_API_KEY"
```

**Antwort (alles verifiziert)**:

```json
{
  "verified": true,
  "message": "Domain is fully verified and ready to send emails",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 1,
    "is_primary": true
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "status": "verified" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "status": "verified" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "status": "verified" }
  ]
}
```

#### Schritt 7: Als primäre Domain festlegen (optional)

Wenn Sie mehrere Domains haben, legen Sie eine als Standard fest:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/set-primary \
  -H "Authorization: YOUR_API_KEY"
```

### E-Mail-Domains auf Arbeitsbereichsebene

Für mandantenfähige SaaS-Anwendungen können Sie unterschiedliche E-Mail-Domains pro Arbeitsbereich konfigurieren. Arbeitsbereichsdomains überschreiben die Domain auf Unternehmensebene.

Der Workflow ist mit Unternehmensdomains identisch, verwendet aber arbeitsbereichsbezogene Endpunkte:

| Aktion           | Endpunkt                                                       |
| ---------------- | -------------------------------------------------------------- |
| Domains auflisten     | `GET /workspace/{workspace_id}/domains`                        |
| Domain hinzufügen       | `POST /workspace/{workspace_id}/domains`                       |
| Domain abrufen       | `GET /workspace/{workspace_id}/domains/{id}`                   |
| Domain löschen    | `DELETE /workspace/{workspace_id}/domains/{id}`                |
| Eigentümerschaft verifizieren | `POST /workspace/{workspace_id}/domains/{id}/verify-ownership` |
| Einrichtung abschließen   | `POST /workspace/{workspace_id}/domains/{id}/finalize`         |
| DNS verifizieren       | `POST /workspace/{workspace_id}/domains/{id}/verify-dns`       |
| Als primär festlegen      | `POST /workspace/{workspace_id}/domains/{id}/set-primary`      |

**Beispiel: Arbeitsbereichsdomain hinzufügen**

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "sign.acmecorp.com"
  }'
```

Siehe die [API-Referenz zu E-Mail-Domains](/api-reference/v01.27.00/email-domains/list-company-domains) für die vollständige Endpunktdokumentation.

## Benutzerdefinierte Absender-E-Mail-Adresse

Steuern Sie die „From"-Adresse auf allen ausgehenden E-Mails, indem Sie eine benutzerdefinierte Domain mit einem benutzerdefinierten Local Part kombinieren.

### So werden E-Mail-Adressen aufgelöst

Wenn Firma eine E-Mail sendet, wird die Absenderadresse aus drei Komponenten aufgebaut:

```
{sender_name} <{local_part}@{domain}>
```

Jede Komponente wird über eine Fallback-Kette aufgelöst:

**Domain-Auflösung:**
1. Verifizierte Domain des Arbeitsbereichs (primär bevorzugt)
2. Verifizierte Domain des Unternehmens
3. `updates.firma.dev` (Standard)

**Local-Part-Auflösung** (gilt nur bei Verwendung einer benutzerdefinierten Domain):
1. Arbeitsbereichseinstellung `email_local_part`
2. Unternehmenseinstellung `email_local_part`
3. `support` (Standard)

**Sender-Name** ist der Name des Nutzers, der die Signaturanfrage gesendet hat.

Wenn Sie z. B. `email_local_part` auf `noreply` setzen und Ihre verifizierte Domain `sign.acmecorp.com` ist, werden E-Mails gesendet von:

```
Jane Smith <noreply@sign.acmecorp.com>
```

### Local Part der E-Mail setzen

**Unternehmensebene** (Standard für alle Arbeitsbereiche):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "noreply"
  }'
```

**Arbeitsbereichsebene** (überschreibt Unternehmenseinstellung):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "signing"
  }'
```

**Validierungsregeln:**
- 1–64 Zeichen, Kleinbuchstaben, Ziffern, Punkte, Unterstriche und Bindestriche
- Muss mit einem Buchstaben oder einer Ziffer beginnen und enden
- Keine aufeinanderfolgenden Punkte
- Reservierte Werte (`postmaster`, `abuse`, `mailer-daemon`) sind nicht erlaubt

Auf Arbeitsbereichsebene auf `null` setzen, um die Unternehmenseinstellung zu erben.

## Benutzerdefinierte E-Mail-Vorlagen

Passen Sie die Benachrichtigungs-E-Mails an, die Firma in Ihrem Namen sendet, sodass sie zu Ihrer Markenstimme und Ihrem Stil passen. Sie können Vorlagen sowohl auf Unternehmens- als auch auf Arbeitsbereichsebene festlegen, wobei Arbeitsbereichsvorlagen die Unternehmensstandards überschreiben.

### Anpassbare E-Mail-Typen

| E-Mail-Typ | Wann gesendet |
|------------|----------|
| `signing_invite` | Wenn eine Signaturanfrage an einen Empfänger gesendet wird |
| `next_signer` | Wenn der nächste Unterzeichner in einer Signaturreihenfolge an der Reihe ist |
| `resend_notification` | Wenn der Absender eine Signatur-Einladung manuell erneut versendet |
| `reminder_default` | Wenn eine automatische Erinnerung für eine ausstehende Signatur gesendet wird |
| `signing_expired` | Wenn eine Signaturanfrage abläuft |
| `signing_cancelled` | Wenn eine Signaturanfrage abgebrochen wird |
| `signing_declined` | Wenn ein Unterzeichner ablehnt (an andere Unterzeichner gesendet) |
| `signing_declined_admin` | Wenn ein Unterzeichner ablehnt (an den Absender gesendet, inklusive Ablehnungsgrund) |
| `signing_completed` | Wenn alle Unterzeichner das Signieren abgeschlossen haben |
| `signer_identity_changed` | Wenn ein Unterzeichner seinen Namen während des Signierens ändert |

### Verfügbare Platzhalter

Vorlagen unterstützen HTML-Bodies mit dynamischen Platzhaltern in der Syntax `{{placeholder}}`.

**Unterzeichner-Platzhalter:**

| Platzhalter | Beschreibung |
|-------------|-------------|
| `{{signer_first_name}}` | Vorname des Unterzeichners |
| `{{signer_last_name}}` | Nachname des Unterzeichners |
| `{{signer_name}}` | Vollständiger Name des Unterzeichners |
| `{{signer_email}}` | E-Mail-Adresse des Unterzeichners |
| `{{signer_title}}` | Jobtitel des Unterzeichners |
| `{{signer_company}}` | Firmenname des Unterzeichners |

**Dokument-Platzhalter:**

| Platzhalter | Beschreibung |
|-------------|-------------|
| `{{signing_request_name}}` | Name der Signaturanfrage |
| `{{signing_link}}` | URL, unter der der Unterzeichner unterschreibt |
| `{{expiration_date}}` | Wann die Signaturanfrage abläuft |
| `{{download_link}}` | Link zum Herunterladen des signierten Dokuments |
| `{{decliner_name}}` | Name des Unterzeichners, der abgelehnt hat |
| `{{decline_reason}}` | Beim Ablehnen angegebener Grund |
| `{{signing_qr_code}}` | QR-Code-Bild, das zur Signaturseite verlinkt |

**Team-Platzhalter:**

| Platzhalter | Beschreibung |
|-------------|-------------|
| `{{team_name}}` | Arbeitsbereichs-/Teamname |
| `{{team_email}}` | Kontakt-E-Mail des Arbeitsbereichs |
| `{{company_name}}` | Firmenname |
| `{{company_logo}}` | Firmenlogo-Bild |

Sie können diese Liste auch programmatisch abrufen:

```bash
curl https://api.firma.dev/functions/v1/signing-request-api/email-templates/placeholders \
  -H "Authorization: YOUR_API_KEY"
```

### Standardvorlagen ansehen

Rufen Sie die integrierten Standardvorlagen von Firma für eine unterstützte Sprache ab, um sie als Ausgangspunkt zu verwenden:

```bash
curl https://api.firma.dev/functions/v1/signing-request-api/email-templates/defaults/en \
  -H "Authorization: YOUR_API_KEY"
```

Unterstützte Sprachen: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

### E-Mail-Vorlage für Arbeitsbereich setzen

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subject": "Action required: Please sign {{signing_request_name}}",
    "body": "<p>Hi {{signer_name}},</p><p>Please review and sign {{signing_request_name}}.</p><p><a href=\"{{signing_link}}\">Sign now</a></p><p>- The {{team_name}} Team</p>"
  }'
```

**Validierung:**
- `subject`: erforderlich, max. 500 Zeichen
- `body`: erforderlich, max. 50.000 Zeichen

### Unternehmens-E-Mail-Vorlage setzen

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subject": "{{company_name}}: Please sign {{signing_request_name}}",
    "body": "<p>Hi {{signer_name}},</p><p>You have a document to sign.</p><p><a href=\"{{signing_link}}\">Sign now</a></p>"
  }'
```

### Benutzerdefinierte Vorlage löschen

Entfernen Sie eine benutzerdefinierte Vorlage, um zur nächsten Ebene in der Hierarchie zurückzukehren:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY"
```

### Vorlagenhierarchie

E-Mail-Vorlagen folgen einer kaskadierenden Hierarchie:

1. **Arbeitsbereichsvorlage** (höchste Priorität)
2. **Unternehmensvorlage**
3. **Eingebauter Standard** für die Sprache des Arbeitsbereichs (niedrigste Priorität)

So können Sie eine unternehmensweit gebrandete Vorlage festlegen und diese bei Bedarf für bestimmte Arbeitsbereiche überschreiben. Das Löschen einer Vorlage auf einer beliebigen Ebene führt zum Zurückfallen auf die nächste Ebene.

<Warning>
  Es wird eine Warnung zurückgegeben, wenn der Vorlagentext den Platzhalter `{{signing_link}}` nicht enthält, da Empfänger einen Link benötigen, um auf den Signaturablauf zuzugreifen.
</Warning>

Siehe die [API-Referenz zu E-Mail-Vorlagen](/api-reference/v01.27.00/email-templates/list-workspace-email-templates) für die vollständige Endpunktdokumentation.

## Firma-E-Mails deaktivieren

Für vollständige Kontrolle über die Kundenkommunikation können Sie die automatischen E-Mails von Firma pro Signaturanfrage deaktivieren. Dies ermöglicht Ihnen:

- Signaturanfrage-Links über Ihr eigenes E-Mail-System zu versenden
- Integration in bestehende Benachrichtigungsworkflows
- E-Mail-Zeitplanung und Follow-up-Sequenzen anzupassen
- Ihre eigene E-Mail-Zustellinfrastruktur zu verwenden

### E-Mail-Einstellungen bei Signaturanfragen

Jede Signaturanfrage hat Einstellungen, die steuern, welche E-Mails gesendet werden:

| Einstellung                   | Beschreibung                                              | Standard |
| ------------------------- | -------------------------------------------------------- | ------- |
| `send_signing_email`      | Signaturanfrage-Benachrichtigungs-E-Mails an Unterzeichner senden      | `true`  |
| `send_finish_email`       | Abschluss-E-Mail senden, wenn alle Unterzeichner fertig sind            | `true`  |
| `send_expiration_email`   | Ablaufbenachrichtigung senden, wenn die Anfrage abläuft        | `true`  |
| `send_cancellation_email` | Abbruchbenachrichtigung senden, wenn die Anfrage abgebrochen wird | `true`  |

### Signaturanfrage mit deaktivierten E-Mails erstellen

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/signing-requests \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace_id": "workspace_123",
    "template_id": "template_456",
    "settings": {
      "send_signing_email": false,
      "send_finish_email": false,
      "send_expiration_email": false,
      "send_cancellation_email": false
    },
    "recipients": [
      {
        "first_name": "Jane",
        "last_name": "Smith",
        "email": "jane@example.com",
        "designation": "Signer",
        "order": 1
      }
    ]
  }'
```

### Signatur-URLs für manuelle Verteilung abrufen

Wenn E-Mails deaktiviert sind, rufen Sie die Signatur-URLs über die API ab und senden sie über eigene Kanäle:

```js
// Create signing request with emails disabled
const response = await fetch(
  'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
  {
    method: 'POST',
    headers: {
      'Authorization': API_KEY,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      workspace_id: workspaceId,
      template_id: templateId,
      settings: {
        send_signing_email: false,
        send_finish_email: false
      },
      recipients: [
        { first_name: 'Jane', last_name: 'Smith', email: 'jane@example.com', designation: 'Signer', order: 1 }
      ]
    })
  }
)
const signingRequest = await response.json()

// Get signing URLs for each recipient
const usersResponse = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequest.id}/users`,
  {
    headers: { 'Authorization': API_KEY }
  }
)
const { results: users } = await usersResponse.json()

// Send emails through your own system
for (const user of users) {
  const signingUrl = `https://app.firma.dev/signing/${user.id}`
  await yourEmailService.send({
    to: user.email,
    subject: 'Please sign your document',
    body: `Click here to sign: ${signingUrl}`
  })
}
```

## Best Practices

### Branding

- Verwenden Sie eine einheitliche Farbpalette in Logos, E-Mail-Vorlagen und eingebetteten Erlebnissen
- Laden Sie Logos im PNG-Format mit transparentem Hintergrund hoch, um die besten Ergebnisse zu erzielen
- Aktivieren Sie `show_custom_branding_only`, um das Firma-Branding vollständig zu entfernen
- Legen Sie unternehmensweites Branding als Baseline fest und überschreiben Sie es bei Bedarf pro Arbeitsbereich
- Testen Sie den gesamten Signaturablauf aus Sicht Ihrer Kunden

### E-Mail-Domain-Konfiguration

- Verwenden Sie eine Subdomain (z. B. `sign.yourcompany.com`) statt Ihrer Hauptdomain
- Richten Sie alle DNS-Einträge (SPF, DKIM, DMARC) für optimale Zustellbarkeit ein
- Überwachen Sie E-Mail-Bounce-Raten und passen Sie bei Bedarf an
- Testen Sie die E-Mail-Zustellung, bevor Sie mit Kunden live gehen

### E-Mail-Vorlagen

- Verwenden Sie den [defaults-Endpunkt](#view-default-templates), um die eingebauten Vorlagen von Firma als Ausgangspunkt zu sehen
- Fügen Sie stets `{{signing_link}}` in Einladungs- und Erinnerungsvorlagen ein
- Legen Sie eine gebrandete Baseline auf Unternehmensebene fest und überschreiben Sie sie bei Bedarf pro Arbeitsbereich
- Verwenden Sie `{{company_logo}}` in Vorlagen, um das Arbeitsbereichs- oder Unternehmenslogo einzubinden

### Sicherheit

- Erzeugen Sie JWT-Token immer in Ihrem Backend
- Geben Sie API-Schlüssel niemals in Client-seitigem Code preis
- Verwenden Sie kurze Token-Ablaufzeiten (empfohlen: 1–4 Stunden)
- Validieren Sie postMessage-Ursprünge, wenn Sie Iframe-Ereignisse verarbeiten

## Fehlerbehebung

### Domain-Eigentümerschaftsverifikation fehlgeschlagen

**Mögliche Ursachen**:

- DNS-Einträge nicht propagiert (bis zu 48 Stunden warten)
- Falscher TXT-Eintragsname oder -wert
- TXT-Eintrag in falscher Zone hinzugefügt

**Lösung**: Prüfen Sie, ob Ihre DNS-Konfiguration mit den `verification_instructions` übereinstimmt, die beim Hinzufügen der Domain zurückgegeben wurden

### DNS-Einträge werden nicht verifiziert

**Mögliche Ursachen**:

- Einträge noch nicht propagiert
- Falsche Eintragswerte
- Fehlende Einträge

**Lösung**: Rufen Sie `GET /company/domains/{id}` oder `verify-dns` auf, um zu sehen, welche spezifischen Einträge ausstehend sind

### Signatur-Iframe lädt nicht

**Mögliche Ursachen**:

- Ungültige Signing-Request-User-ID
- Abgelaufene oder abgebrochene Signaturanfrage
- Content Security Policy blockiert das Iframe

**Lösung**: Prüfen Sie den Status der Signaturanfrage und die Browser-Konsole auf CSP-Fehler

### JWT-Token abgelaufen

**Mögliche Ursachen**:

- Token-TTL für den Anwendungsfall zu kurz
- Uhrzeitabweichung zwischen Servern

**Lösung**: Erzeugen Sie Token mit angemessener Ablaufzeit und ziehen Sie in Betracht, Token proaktiv zu erneuern

### Farben werden nicht angewendet

**Mögliche Ursachen**:

- Ungültiges Hex-Format (muss 6-stellig sein, z. B. `#0066cc`, nicht 3-stellig `#06c`)
- Einstellung auf Unternehmensebene angewendet, aber der Arbeitsbereich hat einen eigenen Override

**Lösung**: Prüfen Sie zuerst die Arbeitsbereichseinstellungen, dann die Unternehmenseinstellungen. Verwenden Sie `null` auf Arbeitsbereichsebene, um einen Override zu löschen.
