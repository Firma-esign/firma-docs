## v1.27.0

**Tipo de versión:** versión de parche

[Descargar especificación OpenAPI](/api-reference/v01.27.00/openapi-v01.27.00.json)

### Campos de configuración ahora documentados en las respuestas

Esta versión documenta los campos de configuración que la API ya devolvía. Todos los cambios son aditivos y compatibles con versiones anteriores: no cambia ningún comportamiento de la API, solo se ha actualizado la especificación OpenAPI para que coincida con lo que los endpoints ya envían.

#### Configuración de solicitud de firma

El objeto de configuración devuelto por los endpoints de solicitud de firma (`POST /signing-requests`, `POST /signing-requests/create-and-send`, `GET /signing-requests`, `GET /signing-requests/{id}`, `PUT /signing-requests/{id}`, `PATCH /signing-requests/{id}`) ahora documenta cuatro campos adicionales:

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `show_qr_code` | boolean o null | Muestra un código QR en la página de firma que permite a los firmantes continuar en su teléfono. Hereda de la configuración del espacio de trabajo o de la empresa cuando es `null`. |
| `allow_presigning_download` | boolean o null | Permite a los firmantes descargar el documento original antes de firmar. Hereda de la configuración del espacio de trabajo o de la empresa cuando es `null`. |
| `identity_editable_fields` | string[] o null | Campos de identidad que los firmantes pueden editar antes de firmar (por ejemplo, `["name", "company"]`). `null` desactiva la edición de identidad. |
| `notify_identity_change_email` | boolean | Envía una notificación por correo electrónico cuando un firmante cambia su identidad. |

#### Configuración de plantilla

El objeto de configuración devuelto por los endpoints de plantilla (`POST /templates`, `PUT /templates/{id}`, `PATCH /templates/{id}`, `GET /templates`, `GET /templates/{id}`, `POST /templates/{id}/replace-document`) ahora documenta el campo `show_qr_code`. La configuración de plantilla cubre las mismas opciones que la configuración de solicitud de firma, salvo los campos de identidad del firmante (`identity_editable_fields`, `notify_identity_change_email`), que solo se aplican a las solicitudes de firma.

#### Respuestas de cancelación y reenvío

| Endpoint | Nuevo campo de respuesta | Tipo | Descripción |
|----------|--------------------------|------|-------------|
| `POST /signing-requests/{id}/cancel` | `emails_sent` | integer | Número de correos de notificación de cancelación enviados a los firmantes |
| `POST /signing-requests/{id}/resend` | `recipients_failed` | integer | Número de destinatarios cuya notificación de reenvío no se pudo enviar |

### Nuevos esquemas

Dos nuevos esquemas dividen las formas de configuración que difieren entre endpoints, para que cada respuesta documente exactamente los campos que devuelve:

| Esquema | Descripción |
|---------|-------------|
| `TemplateSettings` | Configuración devuelta por los endpoints de plantilla. Coincide con `SigningRequestSettings` menos los campos de identidad del firmante (`identity_editable_fields`, `notify_identity_change_email`), que solo se aplican a las solicitudes de firma. |
| `LegacyDocumentSettings` | Conjunto reducido de configuración devuelto por el endpoint obsoleto `POST /documents`. Las integraciones nuevas deben usar los endpoints de solicitud de firma, que devuelven `SigningRequestSettings`. |

### Guía de migración

Sin cambios incompatibles. Estas son adiciones únicamente de documentación que describen campos que la API ya devuelve. Las integraciones existentes no se ven afectadas. Si analizas objetos de configuración, ahora puedes basarte en los campos documentados `show_qr_code`, `allow_presigning_download`, `identity_editable_fields` y `notify_identity_change_email`, y en `emails_sent` / `recipients_failed` en las respuestas de cancelación y reenvío.

### Configuración del espacio de trabajo: cuatro campos nuevos

Ahora hay cuatro campos disponibles en `GET /workspace/{workspace_id}/settings` y `PUT /workspace/{workspace_id}/settings`:

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `signing_button_label_overrides` | object o null | Etiquetas personalizadas por idioma para los botones de la vista de firma. Las claves son códigos de idioma, los valores son objetos que asocian claves de botón a texto personalizado. Establece `null` para usar los valores por defecto. |
| `show_qr_code` | boolean o null | Muestra un código QR en la página de firma para que los firmantes puedan continuar en su teléfono. Hereda de la configuración de la empresa cuando es `null`. |
| `allow_presigning_download` | boolean o null | Permite a los firmantes descargar el documento original antes de firmar. Hereda de la configuración de la empresa cuando es `null`. |
| `show_partial_watermark` | boolean o null | Muestra una marca de agua "IN PROGRESS" en las descargas parciales de PDF. Hereda de la configuración de la empresa cuando es `null`. |

#### Estructura de sobrescrituras de etiquetas de botón

```json
{
  "en": {
    "common.finish": "Confirm",
    "signing.decline.confirm": "I Refuse"
  },
  "de": {
    "common.finish": "Bestätigen"
  }
}
```

Claves de botón personalizables: `common.finish`, `signing.approveAndFinish`, `signing.nextRequiredField`, `signing.saveAndFinishLater`, `signing.decline.confirm`, `signing.decline.title`, `signing.decline.cancel`, `signing.finishAnyway`, `signing.goBack`.

Los valores se saneen en el lado del servidor (se elimina el HTML, máximo 200 caracteres). Los valores vacíos o solo con espacios en blanco se ignoran.

## v1.25.2

**Tipo de versión:** versión de parche

[Descargar especificación OpenAPI](/api-reference/v01.25.02/openapi-v01.25.02.json)

### Validación más estricta para sobrescrituras de campos de plantilla

Al crear una solicitud de firma a partir de una plantilla, cualquier campo proporcionado que no coincida con un campo de esa plantilla se rechaza ahora con **`400` `VALIDATION_ERROR`** en lugar de ignorarse en silencio. Anteriormente, ese campo se descartaba y la solicitud aún tenía éxito, por lo que un identificador mal escrito significaba que el campo simplemente no aparecía en el documento sin ningún error.

Un campo proporcionado coincide con un campo de plantilla por `template_field_id`, `variable_name` o `variable_defined_name`. El error nombra los campos infractores:

```json
{
  "error": "The following provided fields do not match any field in the template and cannot be applied: full_name_x. Each provided field must reference an existing template field by 'template_field_id' or 'variable_name'.",
  "code": "VALIDATION_ERROR"
}
```

Se aplica a `POST /signing-requests` y `POST /signing-requests/create-and-send`.

**Migración:** asegúrate de que cada campo que envíes en una creación basada en plantilla haga referencia a un campo de plantilla existente. Si dependías de que se ignoraran los campos adicionales o no coincidentes, elimínalos.

### Tipos de campo insensibles a mayúsculas y minúsculas

Los valores del `type` de campo ahora se normalizan sin distinción entre mayúsculas y minúsculas. `"Initials"`, `"Signature"`, `"TextArea"`, etc. se resuelven a su tipo canónico (`initial`, `signature`, `text_area`) en lugar de rechazarse como no válidos. Los valores en minúscula no cambian.

## v1.25.1

**Tipo de versión:** versión de parche

[Descargar especificación OpenAPI](/api-reference/v01.25.01/openapi-v01.25.01.json)

### Respuestas de error precisas para create-and-send

`POST /signing-requests/create-and-send` devolvía anteriormente un genérico **500** cada vez que el paso de envío rechazaba una solicitud, ocultando el motivo real. Ahora devuelve el **estado accionable por el cliente** (`400`, `402` o `422`) con un `code` legible por máquina y el detalle relevante:

- **`400` `VALIDATION_ERROR`**: la validación de la solicitud o del envío falló (p. ej., un firmante al que le falta información obligatoria); la respuesta incluye `validation_errors`, `field_errors` o `details` que identifican qué corregir.
- **`402` `INSUFFICIENT_CREDITS`**: no hay suficientes créditos para enviar; `current_credits` informa del saldo.
- **`422` `RECIPIENT_EMAIL_SUPPRESSED`** *(nuevo)*: la dirección de correo de un destinatario está suprimida (previamente ha rebotado o ha sido marcada como spam) y no se le puede enviar.

Los fallos genuinos del lado del servidor siguen devolviendo `500`. Cuando el paso de envío falla, la solicitud de firma creada parcialmente se revierte, por lo que no queda ningún borrador huérfano.

**Guía de migración:** no se requiere ninguna acción para clientes que ya tratan cualquier respuesta que no sea 2xx como un fallo. Si anteriormente tratabas de forma especial el `500` opaco de este endpoint, cambia a ramificar según el estado HTTP / `code`: `400` y `422` significan "corrige la solicitud y vuelve a intentarlo"; `402` significa "añade créditos".

## v1.25.0

**Tipo de versión:** versión menor

[Descargar especificación OpenAPI](/api-reference/v01.25.00/openapi-v01.25.00.json)

### Colores de marca del editor

La configuración de apariencia incluye ahora cinco colores de marca adicionales que dan tema a los **editores integrados** (los editores de plantillas y de solicitudes de firma que integras en tus propias páginas):

- **`color_accent`**: color de acento para resaltados, pestañas activas, interruptores y controles principales del chrome del editor.
- **`color_accent_fg`**: color de primer plano (texto/icono) que se muestra en las superficies de acento.
- **`color_canvas`**: color de fondo del lienzo del documento que rodea a la página.
- **`color_muted`**: color de superficie apagado (separadores, estados hover, pistas de interruptores).
- **`color_muted_fg`**: color de texto apagado.

Estos se suman a los seis campos de color existentes, de modo que los endpoints de configuración llevan ahora once campos `color_*`:

- **`GET`/`PUT /company/settings`** y **`GET`/`PUT /workspace/{workspace_id}/settings`** aceptan y devuelven los cinco nuevos campos. Cada uno es una cadena hexadecimal `#rrggbb` anulable; `null` significa heredar (espacio de trabajo → empresa → valor por defecto de Firma).
- El objeto **`color_palette`** resuelto devuelto en las cargas de los editores integrados contiene ahora once claves (las seis existentes más `accent`, `accent_fg`, `canvas`, `muted`, `muted_fg`). Cuando no están definidas, `muted` toma por defecto el color `card` resuelto y `muted_fg` se deriva para asegurar la legibilidad, de modo que la paleta siempre está completamente poblada.

Los valores del espacio de trabajo sobrescriben los valores de la empresa; los valores no definidos recurren a los valores por defecto de Firma, por lo que las integraciones existentes se renderizan sin cambios.

**Guía de migración:** no se requiere ninguna acción. Los cinco campos son aditivos y anulables. Para dar marca a los editores integrados, envía cualquiera de los nuevos campos `color_*` (hex `#rrggbb` válido, o `null` para limpiar) al endpoint de configuración de la empresa o del espacio de trabajo.

## v1.24.0

**Tipo de versión:** versión menor

[Descargar especificación OpenAPI](/api-reference/v01.24.00/openapi-v01.24.00.json)

### Claves de API de prueba de espacio de trabajo

Los espacios de trabajo ahora exponen una **clave de API en modo de prueba** junto a la clave en vivo existente. Las solicitudes autenticadas con una clave de prueba no consumen créditos y producen solicitudes de firma marcadas como de prueba y con marca de agua.

- **`POST /workspaces`** ahora aprovisiona un par de claves en vivo + de prueba y devuelve tanto `api_key` (en vivo) como el nuevo campo `test_api_key`.
- **`GET /workspaces/{id}`** y **`GET /workspaces`** ahora devuelven tanto `api_key` como `test_api_key`.
- **`POST /workspaces/{id}/api-key/regenerate`** y **`/api-key/expire`** aceptan un `key_type` opcional (`"live"` o `"test"`, predeterminado `"live"`). Regenerar un tipo de clave ya no afecta al otro.
- Los espacios de trabajo existentes creados a través de la API se rellenan retroactivamente con una clave de prueba.

Todos los cambios son aditivos y compatibles con versiones anteriores: los consumidores que leen `api_key` siguen funcionando sin cambios.

**Guía de migración:** no se requiere ninguna acción. Para usar el modo de prueba, lee `test_api_key` de cualquier respuesta de creación/obtención de espacio de trabajo y envíala como clave `Authorization` en las solicitudes que quieras ejecutar en modo de prueba.

## v1.22.1

**Tipo de versión:** versión de parche

[Descargar especificación OpenAPI](/api-reference/v01.22.01/openapi-v01.22.01.json)

### Eliminación de dominio de correo

`DELETE /company/domains/{id}` y `DELETE /workspace/{workspace_id}/domains/{id}` ahora tienen éxito cuando el objetivo es el dominio principal del espacio de trabajo o su único dominio. Anteriormente devolvían `400 "Cannot delete primary domain"`. Tras la eliminación, el correo saliente recurre al remitente por defecto de la empresa, o al predeterminado de la plataforma si no se ha establecido ninguno. Para volver a usar un dominio de envío personalizado, añade y verifica un nuevo dominio, y luego establécelo como principal.

### Guía de migración

Sin cambios incompatibles. Las respuestas `400 "Cannot delete primary domain"` y "cannot delete the only domain" se eliminan de ambos endpoints `DELETE` de dominios; ahora devuelven `200` en esos casos. Las integraciones que trataban de forma especial ese `400` pueden eliminar el manejo.

## v1.22.0

**Tipo de versión:** versión menor

[Descargar especificación OpenAPI](/api-reference/v01.22.00/openapi-v01.22.00.json)

### Endpoint para eliminar solicitud de firma

Nuevo endpoint `DELETE /signing-requests/{id}` para eliminar solicitudes de firma no enviadas (borrador). Este endpoint solo funciona con solicitudes de firma que aún no han sido enviadas. Para las solicitudes de firma ya enviadas, utiliza el endpoint de cancelación existente.

**Respuesta:**
- `200` con `signing_request_id` y `deleted_on` en caso de éxito
- `409 ALREADY_SENT` si la solicitud de firma ya ha sido enviada
- `404` si no se encuentra o ya ha sido eliminada

### Nuevo evento de webhook

- `signing_request.deleted` se dispara cuando se elimina una solicitud de firma a través de la API

### Guía de migración

Sin cambios incompatibles. El nuevo método `DELETE` en `/signing-requests/{id}` es puramente aditivo. Las integraciones existentes no se ven afectadas.

## v1.21.1

**Tipo de versión:** versión de parche

[Descargar especificación OpenAPI](/api-reference/v01.21.01/openapi-v01.21.01.json)

### Gestión de webhooks de espacio de trabajo

El endpoint `PATCH /workspaces/{id}` admite ahora dos nuevos campos para gestionar los webhooks a nivel de espacio de trabajo a través de la API:

- `webhook_enabled` (boolean): habilita o deshabilita los webhooks a nivel de espacio de trabajo. Cuando se establece en `true` por primera vez, se genera automáticamente un secreto de webhook.
- `ignore_company_webhooks` (boolean): si es `true`, los webhooks a nivel de empresa no se dispararán para los eventos de este espacio de trabajo.

Anteriormente, estas configuraciones solo podían configurarse a través de la interfaz del panel.

### Cambios de esquema

- Cuerpo de solicitud de **PATCH /workspaces/{id}**: añadidos `webhook_enabled` (boolean, opcional) e `ignore_company_webhooks` (boolean, opcional)
- Respuesta de **PATCH /workspaces/{id}**: incluye ahora `webhook_enabled`, `webhook_secret`, `webhook_secret_rotated_at`, `webhook_secret_created_at` e `ignore_company_webhooks`

### Guía de migración

Sin cambios incompatibles. Las configuraciones de webhook de espacio de trabajo existentes no se ven afectadas. Usa `PATCH /workspaces/{id}` con `webhook_enabled: true` para habilitar los webhooks de espacio de trabajo mediante la API en lugar del panel.

## v1.21.0

**Tipo de versión:** versión menor

[Descargar especificación OpenAPI](/api-reference/v01.21.00/openapi-v01.21.00.json)

### Nuevos ajustes

- `email_local_part` (string, anulable): personaliza la parte local (antes de la @) de los correos de firma salientes cuando se usa un dominio personalizado verificado. Disponible en dos niveles:
  - **Configuración de la empresa** (`GET/PATCH /company`): establece el valor por defecto para todos los espacios de trabajo. String con valor por defecto `"support"`.
  - **Configuración del espacio de trabajo** (`GET/PUT /workspace/{id}/settings`): sobrescribe el valor por defecto de la empresa. String anulable (null hereda de la empresa).

Los correos se envían desde `{email_local_part}@{your-custom-domain}`. Por ejemplo, establecer `email_local_part` en `"noreply"` hace que los correos se envíen desde `noreply@yourdomain.com` en lugar de `support@yourdomain.com`.

Validación: alfanumérico en minúsculas, puntos, guiones y guiones bajos. Debe empezar y terminar por un carácter alfanumérico. 1-64 caracteres. Patrón: `^[a-z0-9]([a-z0-9._-]*[a-z0-9])?$`

### Cambios de esquema

**Fecha de lanzamiento:** 5 de mayo de 2026

### Nuevas funciones

#### Campos editables de identidad

Las solicitudes de firma ahora admiten la edición de identidad por campo y por firmante. Cuando está configurada, los firmantes ven un diálogo de confirmación antes de firmar en el que pueden revisar y editar los campos de identidad especificados (nombre, empresa, cargo, teléfono, dirección).

Nuevas propiedades de configuración:

| Propiedad | Tipo | Por defecto | Descripción |
|-----------|------|-------------|-------------|
| `identity_editable_fields` | `string[]` o `null` | `null` | Array de claves de campos de identidad que los firmantes pueden editar. `null` = desactivado. |
| `notify_identity_change_email` | `boolean` | `false` | Envía un correo cuando un firmante cambia su identidad |

Claves de campo válidas: `name`, `company`, `title`, `phone`, `address`

Sobrescritura por destinatario a través de `identity_editable_fields` en el objeto del destinatario:
- `null`: usa el valor por defecto de la solicitud
- `[]`: desactivado para este firmante
- `["name", "company"]`: campos personalizados para este firmante

Ejemplo:

```json
{
  "settings": {
    "identity_editable_fields": ["name", "company"]
  },
  "recipients": [
    {
      "name": "John Doe",
      "email": "john@example.com",
      "designation": "signer",
      "order": 1,
      "identity_editable_fields": null
    },
    {
      "name": "Jane Smith",
      "email": "jane@example.com",
      "designation": "signer",
      "order": 2,
      "identity_editable_fields": []
    }
  ]
}
```

En este ejemplo, John hereda el valor por defecto de la solicitud (nombre + empresa editables), mientras que Jane tiene la edición de identidad explícitamente desactivada.

Nuevo evento de webhook: `signing_request.recipient.identity_changed` está ahora disponible como evento de webhook estándar. Suscríbete a él en tu configuración de webhooks para recibir notificaciones cuando un firmante modifique su identidad durante la firma.

#### Campos de datos precargados editables

Los campos de datos precargados pueden ahora ser opcionalmente editables por los firmantes. Antes, todos los campos con `prefilled_data` establecido se bloqueaban automáticamente como de solo lectura. Una nueva propiedad `prefilled_editable` te permite controlar este comportamiento.

Cuando `prefilled_editable` es `true`, el valor del campo se sigue autopoblando a partir de los datos del destinatario, pero el firmante puede modificarlo durante el proceso de firma.

| Propiedad | Tipo | Por defecto | Descripción |
|-----------|------|-------------|-------------|
| `prefilled_editable` | `boolean` | `false` | Cuando es true, los firmantes pueden editar los valores precargados |

Disponible en:
- Campos de plantilla (`PATCH /templates/{id}`, `PUT /templates/{id}`)
- Campos de solicitud de firma (`POST /signing-requests`, `POST /signing-requests/create-and-send`)
- Objetos de respuesta de campo (`GET /signing-requests/{id}`)

Ejemplo: crear una solicitud de firma con un campo precargado editable:

```json
{
  "fields": [
    {
      "type": "text",
      "recipient_id": "temp_1",
      "page_number": 1,
      "x": 10,
      "y": 20,
      "width": 30,
      "height": 5,
      "format_rules": {
        "prefilledData": "full_name",
        "prefilledEditable": true
      }
    }
  ]
}
```

#### Validación de la fuente del valor del campo

La API ahora valida que los campos utilicen solo una fuente de valor a la vez. Las solicitudes que combinen `read_only_value`, `prefilled_data` y `value` explícito en el mismo campo devolverán un `400 VALIDATION_ERROR`. Esta imposición se aplica a:

- `POST /signing-requests`
- `POST /signing-requests/create-and-send`
- `PATCH /templates/{id}` (creación y actualización de campos)
- `PUT /templates/{id}` (upsert de campos)

#### Precedencia de valores

Cuando `prefilled_editable` es `true` y se establecen tanto un `value` explícito como `prefilled_data` en un campo, el valor explícito tiene prioridad sobre los datos del destinatario autopoblados.

### Guía de migración

Sin cambios incompatibles. Las nuevas funciones son opt-in:
- La edición de identidad está desactivada por defecto (`identity_editable_fields: null`). Establécela en un array de claves de campo para activarla.
- Los campos de datos precargados existentes siguen bloqueados (solo lectura) por defecto. Establece `prefilled_editable: true` en `format_rules` para hacerlos editables.

### Nuevas funciones

#### Descargar documento de solicitud de firma

Un nuevo endpoint recupera una URL de descarga para el documento de una solicitud de firma. Para las solicitudes de firma completadas, la URL apunta al PDF final firmado. Para las solicitudes en curso, canceladas, rechazadas o expiradas, la URL apunta a un PDF de instantánea parcial que captura el estado del documento en el momento de la última acción del firmante.

**Endpoint:** `GET /signing-requests/{id}/download`

**Campos de respuesta:**

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `status` | `string` | Estado de la solicitud de firma: `finished`, `in_progress`, `cancelled`, `declined` o `expired` |
| `is_partial` | `boolean` | `true` cuando el documento es una instantánea parcial (no todos los firmantes han completado) |
| `download_url` | `string` | URL prefirmada al documento PDF |
| `generated_at` | `date-time \| null` | Cuándo se generó por última vez el documento |
| `expires_at` | `date-time` | Cuándo expira `download_url` |

**Comportamiento según el estado:**

| Estado de la solicitud de firma | Respuesta |
|--------------------------------|-----------|
| Aún no enviada | `409 Conflict`: descarga no disponible |
| Enviada, generación en curso | `503 Service Unavailable` con encabezado `Retry-After` |
| Enviada, al menos un firmante ha completado | `200` con `is_partial: true` |
| Cancelada / rechazada / expirada | `200` con `is_partial: true` (instantánea del último estado conocido) |
| Completada (todos los firmantes han terminado) | `200` con `is_partial: false` |

Las instantáneas parciales se generan bajo demanda y se almacenan en caché. La primera solicitud puede devolver `503` con un encabezado `Retry-After` mientras se genera el PDF; vuelve a intentarlo tras el intervalo indicado.

**Expiración de URL:** el `download_url` es una URL prefirmada. Vuelve a llamar al endpoint después de `expires_at` para obtener una URL nueva.

#### Ajuste de marca de agua para descargas parciales

Un nuevo ajuste de espacio de trabajo `show_partial_watermark` controla si las descargas parciales de PDF incluyen una marca de agua diagonal "IN PROGRESS — NOT YET EXECUTED". Desactivado por defecto. Se puede establecer a nivel de empresa (valor por defecto para todos los espacios de trabajo) o sobrescribirse por espacio de trabajo.

**Endpoint:** `PUT /workspace/{workspace_id}/settings`

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `show_partial_watermark` | `boolean \| null` | `true` para activar la marca de agua, `false` para desactivarla, `null` para heredar el valor por defecto de la empresa |

### Guía de migración

**De v1.15.3 a v1.16.0:**

- El endpoint de descarga es puramente opt-in. No cambia ningún comportamiento existente.

- Añadido `stamp` a los enums de tipos de campo en todos los esquemas relacionados con campos (plantillas, solicitudes de firma, create-and-send)
- Actualizadas las descripciones de los tipos de campo para documentar el comportamiento del sello

### Guía de migración

**De v01.12.01 a v01.13.00:**

Este es un cambio aditivo no incompatible. Las integraciones existentes no se ven afectadas.

- Si creas campos a través de la API, ahora puedes usar `"stamp"` como tipo de campo. Los campos de sello deben establecerse como `read_only: true` con la imagen del sello como URL de datos PNG en `read_only_value`.
- Los campos de sello también son compatibles en las anchor tags. Al usar anchor tags, la imagen del sello debe proporcionarse mediante `read_only_value` como URL de datos PNG.

## v01.12.01

**Fecha de lanzamiento:** 2 de abril de 2026

### Nuevas funciones

#### `variable_defined_name` en respuestas de campo

Todos los endpoints de la API que devuelven campos de plantilla o de solicitud de firma incluyen ahora una propiedad `variable_defined_name`. Este es el `field_name` legible por humanos de la definición de campo personalizado al que está vinculado el campo.

Anteriormente, los campos vinculados a definiciones de campos personalizados solo exponían `variable_name`, que contiene el identificador interno (por ejemplo, `custom_template_a1b2c3d4-...`). El nuevo `variable_defined_name` devuelve el nombre que definiste originalmente (por ejemplo, `artist_name`), lo que facilita mapear los campos en tu integración sin cruzar referencias con el endpoint de campos personalizados.

**Endpoints afectados:**
- `GET /templates` (lista y único)
- `GET /templates/{id}/fields`
- `POST /templates/{id}/replace-document`
- `GET /signing-requests` (lista)
- `GET /signing-requests/{id}/fields`
- `GET /documents` (legacy)

**Ejemplo de respuesta:**
```json
{
  "variable_name": "custom_template_a1b2c3d4-...",
  "variable_defined_name": "artist_name"
}
```

Para los campos no vinculados a una definición de campo personalizado, `variable_defined_name` es `null`.

#### `variable_defined_name` como entrada para la coincidencia de campos

Al crear una solicitud de firma desde una plantilla mediante `POST /signing-requests/create-and-send`, ahora puedes usar `variable_defined_name` para dirigirte a los campos para sobrescrituras en lugar del `variable_name` interno:

```json
{
  "fields": [
    { "variable_defined_name": "artist_name", "read_only_value": "John Smith" }
  ]
}
```

Se aceptan tanto `variable_name` como `variable_defined_name`. `variable_name` tiene prioridad cuando se proporcionan ambos.

### Sin cambios incompatibles

Esta es una versión de parche no incompatible. Todo el comportamiento existente de la API no cambia.

## v01.10.00

### Nuevas funciones

#### Lógica de campos condicional

Los campos admiten ahora reglas condicionales tanto para el estado **obligatorio** como para la **visibilidad**. Hay dos nuevas propiedades anulables disponibles en los objetos de campo:

- `required_conditions`: cuando se establece, sobrescribe la marca estática `required`. El campo solo es obligatorio cuando las condiciones se evalúan a true según los valores de otros campos.
- `visibility_conditions`: cuando se establece, el campo se oculta a menos que las condiciones se evalúen a true. Los campos ocultos no se validan en el envío.

Las condiciones utilizan una estructura de grupos anidados con operadores lógicos:

| Esquema | Descripción |
|---------|-------------|
| `ConditionSet` | Contenedor de nivel superior con un operador `logic` (`and`/`or`) y un array de `groups` |
| `ConditionGroup` | Contiene un array de `conditions`, combinadas usando el operador opuesto al `logic` del padre |
| `Condition` | Evalúa un único campo por `field_id`, `operator` y un `value` opcional |

**Operadores admitidos:** `is_filled`, `is_empty`, `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `less_than`, `greater_than_or_equal`, `less_than_or_equal`

**Ejemplo: hacer un campo obligatorio solo cuando otro campo esté relleno:**
```json
{
  "required_conditions": {
    "logic": "and",
    "groups": [
      {
        "conditions": [
          {
            "field_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
            "operator": "is_filled"
          }
        ]
      }
    ]
  }
}
```

Se han añadido tres nuevos tipos de campo:

```json
    "format_rules": { "urlDisplayText": "View Terms & Conditions" }
  }
}
```

#### Operaciones PATCH sobre campos

Tanto los endpoints PATCH de plantilla como de solicitud de firma admiten ahora operaciones sobre un único campo:

**PATCH de plantilla** (`PATCH /templates/{id}`):
- Ahora puede actualizar propiedades, un único usuario **o un único campo**
- Incluye el objeto `field` en el cuerpo de la solicitud
- Incluye `field.id` para actualizar un campo existente; omítelo para crear uno nuevo

**PATCH de solicitud de firma** (`PATCH /signing-requests/{id}`):
- Ahora puede actualizar propiedades, un único destinatario **o un único campo**
- Misma estructura de objeto de campo que las plantillas

Ejemplo: crear un nuevo campo:
```json
{
  "field": {
    "type": "text",
    "x": 100,
    "y": 200,
    "width": 200,
    "height": 30,
    "page": 1,
    "required": true,
    "assigned_to_user_id": "user-uuid"
  }
}
```

Ejemplo: actualizar un campo existente:
```json
{
  "field": {
    "id": "field-uuid",
    "x": 120,
    "y": 220
  }
}
```

#### Coincidencia de usuario de plantilla para destinatarios

Al crear solicitudes de firma a partir de plantillas, ahora puedes usar `template_user_id` para hacer coincidir explícitamente los destinatarios con los usuarios de la plantilla:

```json
{
  "template_id": "template-uuid",
  "recipients": [
    {
      "template_user_id": "template-user-uuid",
      "first_name": "Jane",
      "last_name": "Smith",
      "email": "jane@example.com"
    }
  ]
}
```

- `template_user_id` es el método preferido para la coincidencia
- `order` sigue disponible como alternativa

Actualizado de:
```
["text", "signature", "date", "checkbox", "initials", "dropdown", "radio"]
```

A:
```
["text", "signature", "date", "checkbox", "initials", "dropdown", "radio_buttons", "textarea", "url"]
```

- La creación de solicitudes de firma usa ahora la referencia al esquema compartido `Recipient`
