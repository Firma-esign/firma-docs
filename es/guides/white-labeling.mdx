Firma ofrece capacidades completas de marca blanca que te permiten crear una experiencia de firma electrónica con marca totalmente personalizada. Desde logotipos personalizados y temas de color hasta interfaces integradas, puedes asegurarte de que tus clientes interactúen con tu marca durante todo el proceso de firma.

## Descripción general

La marca blanca en Firma implica varios componentes:

1. **Marca personalizada**: sube logotipos, establece temas de color y oculta por completo la marca de Firma
2. **Términos personalizados del firmante**: obliga a los firmantes a aceptar tus propios términos de servicio
3. **Etiquetas de botones de firma**: personaliza el texto de los botones de firma por idioma
4. **Dominios de correo personalizados**: envía correos de solicitud de firma desde tu propio dominio
5. **Dirección de remitente de correo personalizada**: controla la dirección "from" en todos los correos salientes
6. **Plantillas de correo personalizadas**: personaliza el contenido y la marca de los correos de notificación de firma
7. **Desactivar los correos de Firma**: apaga los correos automáticos por solicitud de firma y envía los tuyos
8. **Experiencias integradas**: integra los editores de firma y de plantillas directamente en tu aplicación

<Note>
  Todos los ejemplos de API usan tu clave de API directamente en el encabezado `Authorization`. No uses el prefijo `Bearer`.
</Note>

## Marca personalizada

Personaliza logotipos, colores y visibilidad de la marca a nivel de empresa y de espacio de trabajo. Los ajustes del espacio de trabajo sobrescriben los de la empresa, lo que te permite crear experiencias con marca distintas para cada uno de tus clientes.

### Logotipo de la empresa

Sube un logotipo para toda tu cuenta de Firma. Este logotipo aparece en los correos de firma y en la experiencia de firma.

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/logo.png"
```

**Requisitos:**
- Formato: solo PNG o JPEG
- Tamaño máximo: 2 MB

Para eliminar el logotipo de la empresa:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Logotipo del espacio de trabajo

Sobrescribe el logotipo de la empresa para un espacio de trabajo específico:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/workspace-logo.png"
```

Para eliminar un logotipo de espacio de trabajo (recurrirá al logotipo de la empresa):

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Tematización de colores

Personaliza la paleta de colores de la experiencia de firma usando valores hexadecimales. Los colores pueden establecerse a nivel de empresa y de espacio de trabajo.

| Ajuste de color | Descripción | Predeterminado de Firma |
|-----------------|-------------|-------------------------|
| `color_primary` | Color de acento principal (botones, enlaces) | `#c285ff` |
| `color_primary_fg` | Color de texto sobre elementos primarios | `#ffffff` |
| `color_background` | Fondo de la página | `#1c1c21` |
| `color_foreground` | Color principal del texto | `#ffffff` |
| `color_card` | Fondo de tarjetas/paneles | `#22222a` |
| `color_border` | Color del borde | `#3b3b3b` |
| `color_accent` | Color de acento secundario | `#c285ff` |
| `color_accent_fg` | Color de texto sobre elementos de acento | `#ffffff` |
| `color_canvas` | Fondo del lienzo/visor del documento | `#1c1c21` |
| `color_muted` | Áreas de fondo apagadas/sutiles | `#22222a` |
| `color_muted_fg` | Color de texto sobre áreas apagadas | `#a1a1aa` |

**Establecer colores a nivel de empresa:**

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#0066cc",
    "color_primary_fg": "#ffffff",
    "color_background": "#f5f5f5",
    "color_foreground": "#1a1a1a",
    "color_card": "#ffffff",
    "color_border": "#e0e0e0"
  }'
```

**Establecer colores a nivel de espacio de trabajo** (sobrescribe los colores de la empresa para este espacio de trabajo):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#ff6600",
    "color_primary_fg": "#ffffff"
  }'
```

Los colores deben ser valores hexadecimales de 6 dígitos (por ejemplo, `#0066cc`). Establece un color en `null` para borrarlo y volver al siguiente nivel de la jerarquía.

### Ocultar la marca de Firma

Elimina toda la marca de Firma de la experiencia de firma habilitando `show_custom_branding_only` a nivel de empresa:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "show_custom_branding_only": true
  }'
```

### Ajustes de visualización adicionales

Afina la experiencia de firma con estos ajustes, disponibles a nivel de empresa y de espacio de trabajo:

| Ajuste | Descripción | Predeterminado |
|--------|-------------|----------------|
| `show_signature_frame` | Mostrar el marco/borde de dibujo de la firma | `true` |
| `show_partial_watermark` | Mostrar una marca de agua en los documentos parcialmente firmados | `false` |
| `show_qr_code` | Incluir un código QR con enlace a la página de firma en los correos | `false` |
| `require_otp_verification` | Requerir verificación OTP por correo antes de firmar | `false` |
| `require_terms_acceptance` | Requerir que los firmantes acepten los términos antes de firmar | `true` |
| `disable_guided_navigation` | Desactivar la navegación guiada paso a paso por los campos | `false` |
| `allow_presigning_download` | Permitir a los firmantes descargar el documento antes de firmar | `false` |

A nivel de espacio de trabajo, establece estos valores en `null` para heredar el ajuste de la empresa.

### Jerarquía de ajustes

Los ajustes de marca siguen una jerarquía en cascada:

1. **Ajuste del espacio de trabajo** (prioridad más alta)
2. **Ajuste de la empresa**
3. **Predeterminado de Firma** (prioridad más baja)

Esto te permite establecer valores por defecto a nivel de empresa y sobrescribirlos por espacio de trabajo. A nivel de espacio de trabajo, establecer un valor en `null` significa "heredar de la empresa".

Consulta la [referencia de la API de configuración del espacio de trabajo](/api-reference/v01.27.00/workspace-settings/get-workspace-settings) para ver la lista completa de ajustes configurables.

## Términos personalizados del firmante

Obliga a los firmantes a aceptar tus propios términos de servicio o acuerdo legal antes de firmar. Los términos se configuran por idioma y siguen la misma cascada empresa/espacio de trabajo que los demás ajustes.

### Cómo funciona

Cuando `require_terms_acceptance` está activado (el valor por defecto), los firmantes ven una casilla de aceptación de términos antes de poder firmar. Puedes personalizar el texto de la declaración y enlazar opcionalmente a un documento completo de términos.

Los términos se almacenan por idioma para que los firmantes vean los términos en el idioma de su espacio de trabajo, recurriendo a la cascada:

1. **Términos del espacio de trabajo** para el idioma del firmante (prioridad más alta)
2. **Términos de la empresa** para el idioma del firmante
3. **Términos predeterminados de Firma** (prioridad más baja)

### Establecer términos a nivel de empresa

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acme.com/terms\">Terms of Service</a> and acknowledge that my electronic signature is legally binding.",
    "terms_url": "https://acme.com/terms"
  }'
```

### Establecer términos a nivel de espacio de trabajo

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acmecorp.com/signing-terms\">Signing Terms</a>.",
    "terms_url": "https://acmecorp.com/signing-terms"
  }'
```

**Campos:**
- `statement_text` (obligatorio): texto HTML mostrado junto a la casilla de aceptación. Se permiten etiquetas HTML básicas (`<a>`, `<b>`, `<i>`, `<br>`).
- `terms_url` (opcional): URL al documento completo de términos.

### Listar términos

Obtén todos los términos configurados para una empresa o espacio de trabajo:

```bash
# Company terms
curl https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms \
  -H "Authorization: YOUR_API_KEY"

# Workspace terms
curl https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms \
  -H "Authorization: YOUR_API_KEY"
```

### Eliminar términos

Elimina términos personalizados para un idioma específico para recurrir al siguiente nivel de la cascada:

```bash
# Delete company terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"

# Delete workspace terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"
```

### Desactivar la aceptación de términos

Para desactivar por completo el requisito de aceptación de términos:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "require_terms_acceptance": false
  }'
```

Idiomas admitidos: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

Consulta la [referencia de la API de términos del firmante](/api-reference/v01.27.00/signer-terms/list-company-signer-terms) para conocer la documentación completa del endpoint.

## Etiquetas personalizadas de botones de firma

Personaliza el texto de los botones de firma por espacio de trabajo y por idioma. Esto te permite ajustar las etiquetas de los botones a la terminología o la localización de tu producto.

### Establecer sobrescrituras de etiquetas de botones

Las etiquetas de los botones se establecen como un objeto JSONB en la configuración del espacio de trabajo, indexado por idioma y luego por clave de botón:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": {
      "en": {
        "sign": "Approve and Sign",
        "next": "Continue",
        "finish": "Complete Signing"
      },
      "es": {
        "sign": "Aprobar y Firmar",
        "next": "Continuar",
        "finish": "Completar Firma"
      }
    }
  }'
```

### Borrar sobrescrituras de etiquetas de botones

Establece en `null` para revertir a las etiquetas de botón predeterminadas de Firma:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": null
  }'
```

**Validación:**
- Cada etiqueta debe ser una cadena, máximo 500 caracteres
- Las claves de idioma deben ser códigos de idioma válidos (máx. 10 caracteres)
- Las etiquetas HTML y los caracteres de control se eliminan automáticamente

<Note>
  Las sobrescrituras de etiquetas de botones son un ajuste solo a nivel de espacio de trabajo. No hay una sobrescritura a nivel de empresa.
</Note>

## Dominios de correo personalizados

Por defecto, los correos de solicitud de firma se envían desde el dominio de Firma. Con los dominios de correo personalizados, los correos aparecen como si vinieran directamente de tu empresa o de las empresas de tus clientes.

### Dominios de correo a nivel de cuenta (empresa)

Configura un dominio de correo personalizado para toda tu cuenta de Firma. Todos los espacios de trabajo usarán este dominio por defecto a menos que se sobrescriba.

#### Paso 1: añade tu dominio

Usa la API para añadir un dominio de correo personalizado:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "acme.com"
  }'
```

**Respuesta (201 Created)**:

```json
{
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 0,
    "domain_status": 0,
    "is_primary": false,
    "verification_token": "firma-verify=abc123xyz",
    "date_created": "2024-01-15T10:30:00Z"
  },
  "verification_instructions": {
    "record_type": "TXT",
    "record_name": "_firma-verification.acme.com",
    "record_value": "firma-verify=abc123xyz",
    "next_step": "Add this TXT record to your DNS, then call POST /company/domains/{id}/verify-ownership"
  }
}
```

#### Paso 2: añade el registro TXT de verificación

Añade el registro TXT de verificación a tu DNS:

| Tipo | Nombre                        | Valor                    |
| ---- | ----------------------------- | ------------------------ |
| TXT  | `_firma-verification.acme.com`| `firma-verify=abc123xyz` |

<Note>
  La propagación de DNS suele tardar unos minutos, pero puede tardar hasta 48 horas.
</Note>

#### Paso 3: verifica la propiedad del dominio

Una vez añadido el registro TXT, verifica la propiedad:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-ownership \
  -H "Authorization: YOUR_API_KEY"
```

**Respuesta**:

```json
{
  "message": "Domain ownership verified",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 1,
    "domain_status": 0
  },
  "next_step": "Call POST /company/domains/{id}/finalize to complete domain setup and receive DNS records for email sending"
}
```

#### Paso 4: finaliza la configuración del dominio

Tras verificar la propiedad, finaliza el dominio para recibir los registros DNS de envío de correo:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/finalize \
  -H "Authorization: YOUR_API_KEY"
```

**Respuesta**:

```json
{
  "message": "Domain finalized. Add the following DNS records to enable email sending.",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 0
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "ttl": "Auto", "status": "pending" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "ttl": "Auto", "status": "pending" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "ttl": "Auto", "status": "pending" }
  ],
  "next_step": "Add these DNS records, then call POST /company/domains/{id}/verify-dns to complete verification"
}
```

#### Paso 5: añade los registros DNS

Añade los tres registros DNS a tu dominio:

| Tipo  | Nombre              | Valor                               |
| ----- | ------------------- | ----------------------------------- |
| TXT   | `@`                 | `v=spf1 include:amazonses.com ~all` |
| CNAME | `resend._domainkey` | `resend._domainkey.amazonses.com`   |
| TXT   | `_dmarc`            | `v=DMARC1; p=none;`                 |

#### Paso 6: verifica los registros DNS

Una vez añadidos los registros DNS, verifícalos:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-dns \
  -H "Authorization: YOUR_API_KEY"
```

**Respuesta (todos verificados)**:

```json
{
  "verified": true,
  "message": "Domain is fully verified and ready to send emails",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 1,
    "is_primary": true
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "status": "verified" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "status": "verified" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "status": "verified" }
  ]
}
```

#### Paso 7: establecer como dominio principal (opcional)

Si tienes varios dominios, establece uno como predeterminado:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/set-primary \
  -H "Authorization: YOUR_API_KEY"
```

### Dominios de correo a nivel de espacio de trabajo

Para aplicaciones SaaS multitenant, puedes configurar distintos dominios de correo por espacio de trabajo. Los dominios del espacio de trabajo sobrescriben el dominio a nivel de empresa.

El flujo es idéntico al de los dominios de empresa, pero utiliza endpoints con ámbito de espacio de trabajo:

| Acción             | Endpoint                                                       |
| ------------------ | -------------------------------------------------------------- |
| Listar dominios    | `GET /workspace/{workspace_id}/domains`                        |
| Añadir dominio     | `POST /workspace/{workspace_id}/domains`                       |
| Obtener dominio    | `GET /workspace/{workspace_id}/domains/{id}`                   |
| Eliminar dominio   | `DELETE /workspace/{workspace_id}/domains/{id}`                |
| Verificar propiedad| `POST /workspace/{workspace_id}/domains/{id}/verify-ownership` |
| Finalizar config.  | `POST /workspace/{workspace_id}/domains/{id}/finalize`         |
| Verificar DNS      | `POST /workspace/{workspace_id}/domains/{id}/verify-dns`       |
| Establecer principal | `POST /workspace/{workspace_id}/domains/{id}/set-primary`   |

**Ejemplo: añadir un dominio de espacio de trabajo**

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "sign.acmecorp.com"
  }'
```

Consulta la [referencia de la API de dominios de correo](/api-reference/v01.27.00/email-domains/list-company-domains) para conocer la documentación completa del endpoint.

## Dirección de remitente de correo personalizada

Controla la dirección "from" en todos los correos salientes combinando un dominio personalizado con una parte local personalizada.

### Cómo se resuelven las direcciones de correo

Cuando Firma envía un correo, la dirección del remitente se construye a partir de tres componentes:

```
{sender_name} <{local_part}@{domain}>
```

Cada componente se resuelve mediante una cadena de fallback:

**Resolución del dominio:**
1. Dominio verificado del espacio de trabajo (se prefiere el principal)
2. Dominio verificado de la empresa
3. `updates.firma.dev` (predeterminado)

**Resolución de la parte local** (solo se aplica cuando se usa un dominio personalizado):
1. Ajuste `email_local_part` del espacio de trabajo
2. Ajuste `email_local_part` de la empresa
3. `support` (predeterminado)

**Sender name** es el nombre del usuario que envió la solicitud de firma.

Por ejemplo, si estableces `email_local_part` en `noreply` y tu dominio verificado es `sign.acmecorp.com`, los correos se enviarán desde:

```
Jane Smith <noreply@sign.acmecorp.com>
```

### Establecer la parte local del correo

**Nivel de empresa** (predeterminado para todos los espacios de trabajo):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "noreply"
  }'
```

**Nivel de espacio de trabajo** (sobrescribe el ajuste de la empresa):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "signing"
  }'
```

**Reglas de validación:**
- 1-64 caracteres, letras minúsculas, números, puntos, guiones bajos y guiones
- Debe empezar y terminar con una letra o número
- Sin puntos consecutivos
- Los valores reservados (`postmaster`, `abuse`, `mailer-daemon`) no están permitidos

Establece en `null` a nivel de espacio de trabajo para heredar del ajuste de la empresa.

## Plantillas de correo personalizadas

Personaliza los correos de notificación que Firma envía en tu nombre para que coincidan con la voz y el estilo de tu marca. Puedes establecer plantillas a nivel de empresa y de espacio de trabajo, y las plantillas del espacio de trabajo sobrescriben las predeterminadas de la empresa.

### Tipos de correo personalizables

| Tipo de correo | Cuándo se envía |
|----------------|-----------------|
| `signing_invite` | Cuando se envía una solicitud de firma a un destinatario |
| `next_signer` | Cuando le toca al siguiente firmante en un orden de firma |
| `resend_notification` | Cuando el remitente reenvía manualmente una invitación de firma |
| `reminder_default` | Cuando se envía un recordatorio automático para una firma pendiente |
| `signing_expired` | Cuando una solicitud de firma expira |
| `signing_cancelled` | Cuando se cancela una solicitud de firma |
| `signing_declined` | Cuando un firmante rechaza (se envía a los demás firmantes) |
| `signing_declined_admin` | Cuando un firmante rechaza (se envía al remitente, incluye el motivo del rechazo) |
| `signing_completed` | Cuando todos los firmantes han completado la firma |
| `signer_identity_changed` | Cuando un firmante cambia su nombre durante la firma |

### Marcadores disponibles

Las plantillas admiten cuerpos HTML con marcadores dinámicos usando la sintaxis `{{placeholder}}`.

**Marcadores del firmante:**

| Marcador | Descripción |
|----------|-------------|
| `{{signer_first_name}}` | Nombre del firmante |
| `{{signer_last_name}}` | Apellido del firmante |
| `{{signer_name}}` | Nombre completo del firmante |
| `{{signer_email}}` | Dirección de correo del firmante |
| `{{signer_title}}` | Cargo del firmante |
| `{{signer_company}}` | Nombre de la empresa del firmante |

**Marcadores del documento:**

| Marcador | Descripción |
|----------|-------------|
| `{{signing_request_name}}` | Nombre de la solicitud de firma |
| `{{signing_link}}` | URL para que el firmante firme |
| `{{expiration_date}}` | Cuándo expira la solicitud de firma |
| `{{download_link}}` | Enlace para descargar el documento firmado |
| `{{decliner_name}}` | Nombre del firmante que rechazó |
| `{{decline_reason}}` | Motivo proporcionado al rechazar |
| `{{signing_qr_code}}` | Imagen de código QR que enlaza a la página de firma |

**Marcadores del equipo:**

| Marcador | Descripción |
|----------|-------------|
| `{{team_name}}` | Nombre del espacio de trabajo/equipo |
| `{{team_email}}` | Correo de contacto del espacio de trabajo |
| `{{company_name}}` | Nombre de la empresa |
| `{{company_logo}}` | Imagen del logotipo de la empresa |

También puedes obtener esta lista de forma programática:

```bash
curl https://api.firma.dev/functions/v1/signing-request-api/email-templates/placeholders \
  -H "Authorization: YOUR_API_KEY"
```

### Ver las plantillas predeterminadas

Obtén las plantillas integradas predeterminadas de Firma para cualquier idioma admitido y úsalas como punto de partida:

```bash
curl https://api.firma.dev/functions/v1/signing-request-api/email-templates/defaults/en \
  -H "Authorization: YOUR_API_KEY"
```

Idiomas admitidos: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

### Establecer una plantilla de correo del espacio de trabajo

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subject": "Action required: Please sign {{signing_request_name}}",
    "body": "<p>Hi {{signer_name}},</p><p>Please review and sign {{signing_request_name}}.</p><p><a href=\"{{signing_link}}\">Sign now</a></p><p>- The {{team_name}} Team</p>"
  }'
```

**Validación:**
- `subject`: obligatorio, máx. 500 caracteres
- `body`: obligatorio, máx. 50 000 caracteres

### Establecer una plantilla de correo de la empresa

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subject": "{{company_name}}: Please sign {{signing_request_name}}",
    "body": "<p>Hi {{signer_name}},</p><p>You have a document to sign.</p><p><a href=\"{{signing_link}}\">Sign now</a></p>"
  }'
```

### Eliminar una plantilla personalizada

Elimina una plantilla personalizada para volver al siguiente nivel de la jerarquía:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY"
```

### Jerarquía de plantillas

Las plantillas de correo siguen una jerarquía en cascada:

1. **Plantilla del espacio de trabajo** (prioridad más alta)
2. **Plantilla de la empresa**
3. **Predeterminada integrada** para el idioma del espacio de trabajo (prioridad más baja)

Esto te permite establecer una plantilla con marca a nivel de empresa y sobrescribirla para espacios de trabajo específicos cuando sea necesario. Eliminar una plantilla en cualquier nivel hará que recurra al siguiente nivel.

<Warning>
  Se devuelve una advertencia si el cuerpo de una plantilla no contiene el marcador `{{signing_link}}`, ya que los destinatarios necesitan un enlace para acceder al flujo de firma.
</Warning>

Consulta la [referencia de la API de plantillas de correo](/api-reference/v01.27.00/email-templates/list-workspace-email-templates) para conocer la documentación completa del endpoint.

## Desactivar los correos de Firma

Para un control total sobre las comunicaciones con los clientes, puedes desactivar los correos automáticos de Firma por solicitud de firma. Esto te permite:

- Enviar los enlaces de solicitud de firma a través de tu propio sistema de correo
- Integrarte con tus flujos de trabajo de notificaciones existentes
- Personalizar los tiempos y las secuencias de seguimiento del correo
- Usar tu propia infraestructura de entrega de correo

### Ajustes de correo en las solicitudes de firma

Cada solicitud de firma tiene ajustes que controlan qué correos se envían:

| Ajuste                    | Descripción                                              | Predeterminado |
| ------------------------- | -------------------------------------------------------- | -------------- |
| `send_signing_email`      | Enviar correos de notificación de solicitud de firma a los firmantes | `true`  |
| `send_finish_email`       | Enviar correo de finalización cuando todos los firmantes terminen | `true`  |
| `send_expiration_email`   | Enviar notificación de expiración cuando la solicitud expire | `true`  |
| `send_cancellation_email` | Enviar notificación de cancelación cuando la solicitud se cancele | `true`  |

### Crear una solicitud de firma con los correos desactivados

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/signing-requests \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace_id": "workspace_123",
    "template_id": "template_456",
    "settings": {
      "send_signing_email": false,
      "send_finish_email": false,
      "send_expiration_email": false,
      "send_cancellation_email": false
    },
    "recipients": [
      {
        "first_name": "Jane",
        "last_name": "Smith",
        "email": "jane@example.com",
        "designation": "Signer",
        "order": 1
      }
    ]
  }'
```

### Obtener las URLs de firma para distribución manual

Cuando los correos están desactivados, obtén las URLs de firma desde la API y envíalas por tus propios canales:

```js
// Create signing request with emails disabled
const response = await fetch(
  'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
  {
    method: 'POST',
    headers: {
      'Authorization': API_KEY,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      workspace_id: workspaceId,
      template_id: templateId,
      settings: {
        send_signing_email: false,
        send_finish_email: false
      },
      recipients: [
        { first_name: 'Jane', last_name: 'Smith', email: 'jane@example.com', designation: 'Signer', order: 1 }
      ]
    })
  }
)
const signingRequest = await response.json()

// Get signing URLs for each recipient
const usersResponse = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequest.id}/users`,
  {
    headers: { 'Authorization': API_KEY }
  }
)
const { results: users } = await usersResponse.json()

// Send emails through your own system
for (const user of users) {
  const signingUrl = `https://app.firma.dev/signing/${user.id}`
  await yourEmailService.send({
    to: user.email,
    subject: 'Please sign your document',
    body: `Click here to sign: ${signingUrl}`
  })
}
```

## Mejores prácticas

### Marca

- Usa una paleta de colores consistente en logotipos, plantillas de correo y experiencias integradas
- Sube los logotipos en formato PNG con fondo transparente para obtener los mejores resultados
- Habilita `show_custom_branding_only` para eliminar por completo la marca de Firma
- Establece una marca a nivel de empresa como base y sobrescríbela por espacio de trabajo cuando sea necesario
- Prueba el flujo de firma completo desde la perspectiva de tus clientes

### Configuración de dominio de correo

- Utiliza un subdominio (por ejemplo, `sign.yourcompany.com`) en lugar de tu dominio principal
- Configura todos los registros DNS (SPF, DKIM, DMARC) para una entregabilidad óptima
- Monitoriza las tasas de rebote del correo y ajusta según sea necesario
- Prueba la entrega del correo antes de ir en vivo con los clientes

### Plantillas de correo

- Usa el [endpoint de valores predeterminados](#view-default-templates) para ver las plantillas integradas de Firma como punto de partida
- Incluye siempre `{{signing_link}}` en las plantillas de invitación y recordatorio
- Establece una plantilla a nivel de empresa como base con marca y sobrescríbela por espacio de trabajo cuando sea necesario
- Usa `{{company_logo}}` en las plantillas para incluir el logotipo del espacio de trabajo o de la empresa

### Seguridad

- Genera siempre los tokens JWT en tu backend
- Nunca expongas las claves de API en el código del cliente
- Usa tiempos de expiración de token cortos (recomendado: 1-4 horas)
- Valida los orígenes de postMessage al gestionar eventos de iframe

## Resolución de problemas

### La verificación de propiedad del dominio ha fallado

**Causas posibles**:

- Los registros DNS no se han propagado (espera hasta 48 horas)
- Nombre o valor incorrecto del registro TXT
- Registro TXT añadido a la zona equivocada

**Solución**: comprueba que tu configuración DNS coincide con las `verification_instructions` devueltas al añadir el dominio

### Los registros DNS no se verifican

**Causas posibles**:

- Los registros aún no se han propagado
- Valores incorrectos de los registros
- Faltan registros

**Solución**: llama a `GET /company/domains/{id}` o `verify-dns` para ver qué registros específicos están pendientes

### El iframe de firma no carga

**Causas posibles**:

- ID de usuario de solicitud de firma no válido
- Solicitud de firma expirada o cancelada
- La política de seguridad de contenido bloquea el iframe

**Solución**: verifica el estado de la solicitud de firma y comprueba errores CSP en la consola del navegador

### Token JWT expirado

**Causas posibles**:

- TTL del token demasiado corto para el caso de uso
- Desviación de reloj entre servidores

**Solución**: genera tokens con la expiración adecuada, plantéate refrescar los tokens de forma proactiva

### Los colores no se aplican

**Causas posibles**:

- Formato hexadecimal no válido (debe ser de 6 dígitos, p. ej., `#0066cc`, no de 3 dígitos `#06c`)
- Ajuste aplicado a nivel de empresa pero el espacio de trabajo tiene su propia sobrescritura

**Solución**: comprueba primero los ajustes del espacio de trabajo y luego los de la empresa. Usa `null` a nivel de espacio de trabajo para borrar una sobrescritura.
