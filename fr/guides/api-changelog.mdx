## v1.27.0

**Type de release :** Patch Release

[Télécharger la spécification OpenAPI](/api-reference/v01.27.00/openapi-v01.27.00.json)

### Les champs de paramètres sont désormais documentés dans les réponses

Cette version documente les champs de paramètres que l'API retourne déjà. Toutes les modifications sont additives et rétrocompatibles — aucun comportement d'API n'a changé, seule la spécification OpenAPI a été mise à jour pour correspondre à ce que les endpoints envoient déjà.

#### Paramètres de demande de signature

L'objet de paramètres retourné par les endpoints de demande de signature (`POST /signing-requests`, `POST /signing-requests/create-and-send`, `GET /signing-requests`, `GET /signing-requests/{id}`, `PUT /signing-requests/{id}`, `PATCH /signing-requests/{id}`) documente désormais quatre champs supplémentaires :

| Champ | Type | Description |
|-------|------|-------------|
| `show_qr_code` | boolean ou null | Affiche un QR code sur la page de signature qui permet aux signataires de continuer sur leur téléphone. Hérite du paramètre d'espace de travail ou d'entreprise lorsque `null`. |
| `allow_presigning_download` | boolean ou null | Autorise les signataires à télécharger le document original avant signature. Hérite du paramètre d'espace de travail ou d'entreprise lorsque `null`. |
| `identity_editable_fields` | string[] ou null | Champs d'identité que les signataires peuvent modifier avant signature (par ex. `["name", "company"]`). `null` désactive la modification de l'identité. |
| `notify_identity_change_email` | boolean | Envoie une notification par e-mail lorsqu'un signataire modifie son identité. |

#### Paramètres de modèle

L'objet de paramètres retourné par les endpoints de modèle (`POST /templates`, `PUT /templates/{id}`, `PATCH /templates/{id}`, `GET /templates`, `GET /templates/{id}`, `POST /templates/{id}/replace-document`) documente désormais le champ `show_qr_code`. Les paramètres de modèle couvrent les mêmes options que les paramètres de demande de signature, à l'exception des champs d'identité du signataire (`identity_editable_fields`, `notify_identity_change_email`), qui s'appliquent uniquement aux demandes de signature.

#### Réponses de cancel et resend

| Endpoint | Nouveau champ de réponse | Type | Description |
|----------|--------------------|------|-------------|
| `POST /signing-requests/{id}/cancel` | `emails_sent` | integer | Nombre d'e-mails de notification d'annulation envoyés aux signataires |
| `POST /signing-requests/{id}/resend` | `recipients_failed` | integer | Nombre de destinataires dont la notification de renvoi a échoué |

### Nouveaux schémas

Deux nouveaux schémas séparent les formes de paramètres qui diffèrent selon les endpoints, afin que chaque réponse documente exactement les champs qu'elle retourne :

| Schéma | Description |
|--------|-------------|
| `TemplateSettings` | Paramètres retournés par les endpoints de modèle. Identique à `SigningRequestSettings` moins les champs d'identité du signataire (`identity_editable_fields`, `notify_identity_change_email`), qui s'appliquent uniquement aux demandes de signature. |
| `LegacyDocumentSettings` | Ensemble de paramètres réduit retourné par l'endpoint `POST /documents` déprécié. Les nouvelles intégrations doivent utiliser les endpoints de demande de signature, qui retournent `SigningRequestSettings`. |

### Guide de migration

Aucune modification incompatible. Il s'agit d'ajouts uniquement à la documentation décrivant des champs que l'API retourne déjà. Les intégrations existantes ne sont pas affectées. Si vous analysez des objets de paramètres, vous pouvez désormais vous appuyer sur les champs documentés `show_qr_code`, `allow_presigning_download`, `identity_editable_fields` et `notify_identity_change_email`, et sur `emails_sent` / `recipients_failed` dans les réponses de cancel et resend.

## v1.25.2

**Type de release :** Patch Release

[Télécharger la spécification OpenAPI](/api-reference/v01.25.02/openapi-v01.25.02.json)

### Validation plus stricte pour les surcharges de champs de modèle

Lors de la création d'une demande de signature à partir d'un modèle, tout champ fourni qui ne correspond pas à un champ de ce modèle est désormais rejeté avec **`400` `VALIDATION_ERROR`** au lieu d'être silencieusement ignoré. Auparavant, un tel champ était supprimé et la requête aboutissait tout de même, si bien qu'un identifiant mal saisi signifiait que le champ n'apparaissait tout simplement pas sur le document sans erreur.

Un champ fourni correspond à un champ de modèle par `template_field_id`, `variable_name` ou `variable_defined_name`. L'erreur nomme le ou les champs incriminés :

```json
{
  "error": "The following provided fields do not match any field in the template and cannot be applied: full_name_x. Each provided field must reference an existing template field by 'template_field_id' or 'variable_name'.",
  "code": "VALIDATION_ERROR"
}
```

S'applique à `POST /signing-requests` et `POST /signing-requests/create-and-send`.

**Migration :** assurez-vous que chaque champ que vous envoyez sur une création basée sur un modèle référence un champ de modèle existant. Si vous vous appuyiez sur le fait que les champs supplémentaires/non correspondants soient ignorés, retirez-les.

### Types de champs insensibles à la casse

Les valeurs de `type` de champ sont désormais normalisées sans tenir compte de la casse. `"Initials"`, `"Signature"`, `"TextArea"`, etc. sont résolus vers leur type canonique (`initial`, `signature`, `text_area`) au lieu d'être rejetés comme invalides. Les valeurs en minuscules sont inchangées.

## v1.25.1

**Type de release :** Patch Release

[Télécharger la spécification OpenAPI](/api-reference/v01.25.01/openapi-v01.25.01.json)

### Réponses d'erreur précises pour create-and-send

`POST /signing-requests/create-and-send` retournait auparavant un **500** générique chaque fois que l'étape d'envoi rejetait une requête, masquant la véritable raison. Il retourne désormais le **statut exploitable côté client** (`400`, `402` ou `422`) avec un `code` lisible par machine et le détail pertinent :

- **`400` `VALIDATION_ERROR`** — la validation de la requête ou de l'envoi a échoué (par ex. un signataire manquant d'informations requises) ; la réponse comprend `validation_errors`, `field_errors` ou `details` identifiant ce qu'il faut corriger.
- **`402` `INSUFFICIENT_CREDITS`** — crédits insuffisants pour envoyer ; `current_credits` indique le solde.
- **`422` `RECIPIENT_EMAIL_SUPPRESSED`** *(nouveau)* — l'adresse e-mail d'un destinataire est supprimée (précédemment rebondie ou marquée comme spam) et ne peut pas être utilisée.

Les véritables défaillances côté serveur retournent toujours `500`. Lorsque l'étape d'envoi échoue, la demande de signature partiellement créée est annulée, de sorte qu'aucun brouillon orphelin n'est laissé.

**Guide de migration :** aucune action requise pour les clients qui traitent déjà toute réponse non-2xx comme un échec. Si vous traitiez auparavant de manière particulière le `500` opaque de cet endpoint, passez à un branchement sur le statut HTTP / `code` : `400` et `422` signifient « corrigez la requête, puis réessayez », `402` signifie « ajoutez des crédits ».

## v1.25.0

**Type de release :** Minor Release

[Télécharger la spécification OpenAPI](/api-reference/v01.25.00/openapi-v01.25.00.json)

### Couleurs de marque de l'éditeur

Les paramètres d'apparence incluent désormais cinq couleurs de marque supplémentaires qui thématisent les **éditeurs intégrés** (les éditeurs de modèle et de demande de signature que vous intégrez sur vos propres pages) :

- **`color_accent`** — couleur d'accentuation pour les mises en évidence, les onglets actifs, les commutateurs et les contrôles principaux dans le chrome de l'éditeur.
- **`color_accent_fg`** — couleur de premier plan (texte/icônes) affichée sur les surfaces d'accentuation.
- **`color_canvas`** — la couleur entourant le canevas du document, derrière la page.
- **`color_muted`** — couleur de surface atténuée (séparateurs, états de survol, rails des commutateurs).
- **`color_muted_fg`** — couleur de texte atténuée.

Ces couleurs rejoignent les six champs de couleur existants, de sorte que les endpoints de paramètres portent désormais onze champs `color_*` :

- **`GET`/`PUT /company/settings`** et **`GET`/`PUT /workspace/{workspace_id}/settings`** acceptent et retournent les cinq nouveaux champs. Chacun est une chaîne hexadécimale `#rrggbb` nullable ; `null` signifie hériter (workspace → company → défaut Firma).
- L'objet **`color_palette`** résolu retourné dans les charges utiles d'éditeur intégré contient désormais onze clés (les six existantes plus `accent`, `accent_fg`, `canvas`, `muted`, `muted_fg`). Lorsqu'elles ne sont pas définies, `muted` prend par défaut la couleur `card` résolue et `muted_fg` est dérivée pour assurer la lisibilité, de sorte que la palette est toujours entièrement remplie.

Les valeurs d'espace de travail remplacent les valeurs d'entreprise ; les valeurs non définies retombent sur les défauts Firma, de sorte que les intégrations existantes s'affichent inchangées.

**Guide de migration :** aucune action requise. Les cinq champs sont additifs et nullables. Pour habiller les éditeurs intégrés à votre marque, envoyez n'importe lequel des nouveaux champs `color_*` (hex `#rrggbb` valide, ou `null` pour effacer) à l'endpoint des paramètres company ou workspace.

## v1.24.0

**Type de release :** Minor Release

[Télécharger la spécification OpenAPI](/api-reference/v01.24.00/openapi-v01.24.00.json)

### Clés d'API de test d'espace de travail

Les espaces de travail exposent désormais une **clé d'API en mode test** aux côtés de la clé live existante. Les requêtes authentifiées avec une clé de test ne consomment pas de crédits et produisent des demandes de signature marquées comme test et filigranées.

- **`POST /workspaces`** provisionne désormais une paire de clés live + test et retourne à la fois `api_key` (live) et le nouveau champ `test_api_key`.
- **`GET /workspaces/{id}`** et **`GET /workspaces`** retournent désormais à la fois `api_key` et `test_api_key`.
- **`POST /workspaces/{id}/api-key/regenerate`** et **`/api-key/expire`** acceptent un `key_type` optionnel (`"live"` ou `"test"`, défaut `"live"`). Régénérer un type de clé n'affecte plus l'autre.
- Les espaces de travail existants créés via l'API sont rétro-remplis avec une clé de test.

Toutes les modifications sont additives et rétrocompatibles : les consommateurs qui lisent `api_key` continuent de fonctionner sans changement.

**Guide de migration :** aucune action requise. Pour utiliser le mode test, lisez `test_api_key` depuis toute réponse de création/récupération d'espace de travail et envoyez-le en tant que clé `Authorization` pour les requêtes que vous voulez exécuter en mode test.

## v1.22.1

**Type de release :** Patch Release

[Télécharger la spécification OpenAPI](/api-reference/v01.22.01/openapi-v01.22.01.json)

### Suppression de domaine d'e-mail

`DELETE /company/domains/{id}` et `DELETE /workspace/{workspace_id}/domains/{id}` réussissent désormais lorsque la cible est le domaine principal de l'espace de travail ou son unique domaine. Auparavant, ces requêtes retournaient `400 "Cannot delete primary domain"`. Après suppression, les e-mails sortants basculent sur l'expéditeur par défaut de l'entreprise, ou sur le défaut de la plateforme si aucun n'est défini. Pour utiliser à nouveau un domaine d'envoi personnalisé, ajoutez et vérifiez un nouveau domaine, puis définissez-le comme principal.

### Guide de migration

Aucune modification incompatible. Les réponses `400 "Cannot delete primary domain"` et « cannot delete the only domain » sont supprimées des deux endpoints DELETE de domaine ; ils retournent désormais `200` dans ces cas. Les intégrations qui traitaient ce `400` de manière particulière peuvent supprimer cette gestion.

## v1.22.0

**Type de release :** Minor Release

[Télécharger la spécification OpenAPI](/api-reference/v01.22.00/openapi-v01.22.00.json)

### Endpoint de suppression de demande de signature

Nouvel endpoint `DELETE /signing-requests/{id}` pour supprimer les demandes de signature non envoyées (brouillon). Cet endpoint ne fonctionne que sur les demandes de signature qui n'ont pas encore été envoyées. Pour les demandes de signature envoyées, utilisez plutôt l'endpoint d'annulation existant.

**Réponse :**
- `200` avec `signing_request_id` et `deleted_on` en cas de succès
- `409 ALREADY_SENT` si la demande de signature a été envoyée
- `404` si non trouvée ou déjà supprimée

### Nouvel événement de webhook

- `signing_request.deleted` se déclenche lorsqu'une demande de signature est supprimée via l'API

### Guide de migration

Aucune modification incompatible. La nouvelle méthode `DELETE` sur `/signing-requests/{id}` est purement additive. Les intégrations existantes ne sont pas affectées.

## v1.21.1

**Type de release :** Patch Release

[Télécharger la spécification OpenAPI](/api-reference/v01.21.01/openapi-v01.21.01.json)

### Gestion des webhooks d'espace de travail

L'endpoint `PATCH /workspaces/{id}` prend désormais en charge deux nouveaux champs pour gérer les webhooks au niveau de l'espace de travail via l'API :

- `webhook_enabled` (boolean) - Active ou désactive les webhooks au niveau de l'espace de travail. Lorsqu'il est défini à `true` pour la première fois, un secret de webhook est généré automatiquement.
- `ignore_company_webhooks` (boolean) - Si `true`, les webhooks au niveau de l'entreprise ne se déclencheront pas pour les événements de cet espace de travail.

Auparavant, ces paramètres ne pouvaient être configurés que via l'interface du tableau de bord.

### Modifications de schéma

- Corps de requête **PATCH /workspaces/{id}** : ajout de `webhook_enabled` (boolean, optionnel) et `ignore_company_webhooks` (boolean, optionnel)
- Réponse **PATCH /workspaces/{id}** : inclut désormais `webhook_enabled`, `webhook_secret`, `webhook_secret_rotated_at`, `webhook_secret_created_at` et `ignore_company_webhooks`

### Guide de migration

Aucune modification incompatible. Les configurations existantes de webhooks d'espace de travail ne sont pas affectées. Utilisez `PATCH /workspaces/{id}` avec `webhook_enabled: true` pour activer les webhooks d'espace de travail via l'API au lieu du tableau de bord.

## v1.21.0

**Type de release :** Minor Release

[Télécharger la spécification OpenAPI](/api-reference/v01.21.00/openapi-v01.21.00.json)

### Nouveaux paramètres

- `email_local_part` (string, nullable) - Personnalisez la partie locale (avant le @) des e-mails de signature sortants lorsque vous utilisez un domaine personnalisé vérifié. Disponible à deux niveaux :
  - **Paramètres d'entreprise** (`GET/PATCH /company`) - Définit le défaut pour tous les espaces de travail. Chaîne avec pour défaut `"support"`.
  - **Paramètres d'espace de travail** (`GET/PUT /workspace/{id}/settings`) - Remplace le défaut de l'entreprise. Chaîne nullable (null hérite de l'entreprise).

Les e-mails sont envoyés depuis `{email_local_part}@{your-custom-domain}`. Par exemple, définir `email_local_part` à `"noreply"` envoie les e-mails depuis `noreply@yourdomain.com` au lieu de `support@yourdomain.com`.

Validation : caractères alphanumériques en minuscules, points, tirets et underscores. Doit commencer et se terminer par un caractère alphanumérique. 1 à 64 caractères. Motif : `^[a-z0-9]([a-z0-9._-]*[a-z0-9])?$`

### Nouvelles fonctionnalités

#### Champs d'identité modifiables

Les demandes de signature prennent désormais en charge la modification d'identité par champ et par signataire. Lorsque configurés, les signataires voient une boîte de dialogue de confirmation avant signature où ils peuvent examiner et modifier les champs d'identité spécifiés (nom, entreprise, titre, téléphone, adresse).

Nouvelle propriété de paramètres :

| Propriété | Type | Défaut | Description |
|----------|------|---------|-------------|
| `identity_editable_fields` | `string[]` ou `null` | `null` | Tableau des clés de champs d'identité que les signataires peuvent modifier. `null` = désactivé. |
| `notify_identity_change_email` | `boolean` | `false` | Envoyer un e-mail lorsqu'un signataire modifie son identité |

Clés de champs valides : `name`, `company`, `title`, `phone`, `address`

Remplacement par destinataire via `identity_editable_fields` sur l'objet destinataire :
- `null` - utilise le défaut de la requête
- `[]` - désactivé pour ce signataire
- `["name", "company"]` - champs personnalisés pour ce signataire

Exemple :

```json
{
  "settings": {
    "identity_editable_fields": ["name", "company"]
  },
  "recipients": [
    {
      "name": "John Doe",
      "email": "john@example.com",
      "designation": "signer",
      "order": 1,
      "identity_editable_fields": null
    },
    {
      "name": "Jane Smith",
      "email": "jane@example.com",
      "designation": "signer",
      "order": 2,
      "identity_editable_fields": []
    }
  ]
}
```

Dans cet exemple, John hérite du défaut de la requête (name + company modifiables), tandis que Jane a la modification d'identité explicitement désactivée.

Nouvel événement de webhook : `signing_request.recipient.identity_changed` est désormais disponible en tant qu'événement de webhook standard. Abonnez-vous à celui-ci dans votre configuration de webhook pour recevoir des notifications lorsqu'un signataire modifie son identité pendant la signature.

#### Champs de données préremplies modifiables

Les champs de données préremplies peuvent désormais être optionnellement modifiables par les signataires. Auparavant, tous les champs avec `prefilled_data` défini étaient automatiquement verrouillés en lecture seule. Une nouvelle propriété `prefilled_editable` vous permet de contrôler ce comportement.

Lorsque `prefilled_editable` vaut `true`, la valeur du champ est toujours auto-remplie à partir des données du destinataire, mais le signataire peut la modifier pendant le processus de signature.

| Propriété | Type | Défaut | Description |
|----------|------|---------|-------------|
| `prefilled_editable` | `boolean` | `false` | Lorsque true, les signataires peuvent modifier les valeurs préremplies |

Disponible sur :
- Champs de modèle (`PATCH /templates/{id}`, `PUT /templates/{id}`)
- Champs de demande de signature (`POST /signing-requests`, `POST /signing-requests/create-and-send`)
- Objets de réponse de champ (`GET /signing-requests/{id}`)

Exemple - créer une demande de signature avec un champ prérempli modifiable :

```json
{
  "fields": [
    {
      "type": "text",
      "recipient_id": "temp_1",
      "page_number": 1,
      "x": 10,
      "y": 20,
      "width": 30,
      "height": 5,
      "format_rules": {
        "prefilledData": "full_name",
        "prefilledEditable": true
      }
    }
  ]
}
```

#### Validation de la source de valeur du champ

L'API valide désormais que les champs n'utilisent qu'une seule source de valeur à la fois. Les requêtes qui combinent `read_only_value`, `prefilled_data` et une `value` explicite sur le même champ retourneront un `400 VALIDATION_ERROR`. Cette application concerne :

- `POST /signing-requests`
- `POST /signing-requests/create-and-send`
- `PATCH /templates/{id}` (création et mise à jour de champ)
- `PUT /templates/{id}` (upsert de champ)

#### Précédence des valeurs

Lorsque `prefilled_editable` vaut `true` et que `value` explicite et `prefilled_data` sont tous deux définis sur un champ, la valeur explicite prime sur les données de destinataire auto-remplies.

### Guide de migration

Aucune modification incompatible. Les nouvelles fonctionnalités sont optionnelles :
- La modification d'identité est désactivée par défaut (`identity_editable_fields: null`). Définissez-la sur un tableau de clés de champs pour l'activer.
- Les champs de données préremplies existants restent verrouillés (lecture seule) par défaut. Définissez `prefilled_editable: true` dans `format_rules` pour les rendre modifiables.

#### Téléchargement du document d'une demande de signature

Un nouvel endpoint récupère une URL de téléchargement pour le document d'une demande de signature. Pour les demandes de signature terminées, l'URL pointe vers le PDF signé final. Pour les demandes en cours, annulées, refusées ou expirées, l'URL pointe vers un PDF de capture partielle représentant l'état du document au moment de la dernière action du signataire.

**Endpoint :** `GET /signing-requests/{id}/download`

**Champs de réponse :**

| Champ | Type | Description |
|-------|------|-------------|
| `status` | `string` | Statut de la demande de signature : `finished`, `in_progress`, `cancelled`, `declined` ou `expired` |
| `is_partial` | `boolean` | `true` lorsque le document est une capture partielle (tous les signataires n'ont pas terminé) |
| `download_url` | `string` | URL pré-signée vers le document PDF |
| `generated_at` | `date-time \| null` | Quand le document a été généré pour la dernière fois |
| `expires_at` | `date-time` | Quand le `download_url` expire |

| État de la demande de signature | Réponse |
|-----------------------|----------|
| Pas encore envoyée | `409 Conflict` — téléchargement indisponible |
| Envoyée, génération en cours | `503 Service Unavailable` avec en-tête `Retry-After` |
| Envoyée, au moins un signataire a terminé | `200` avec `is_partial: true` |
| Annulée / Refusée / Expirée | `200` avec `is_partial: true` (capture du dernier état connu) |
| Terminée (tous les signataires ont fini) | `200` avec `is_partial: false` |

Les captures partielles sont générées à la demande et mises en cache. La première requête peut retourner `503` avec un en-tête `Retry-After` pendant la génération du PDF — réessayez après l'intervalle indiqué.

**Expiration de l'URL :** le `download_url` est une URL pré-signée. Interrogez à nouveau l'endpoint après `expires_at` pour obtenir une URL fraîche.

Peut être défini au niveau entreprise (défaut pour tous les espaces de travail) ou remplacé par espace de travail.

**Endpoint :** `PUT /workspace/{workspace_id}/settings`

| Champ | Type | Description |
|-------|------|-------------|
| `show_partial_watermark` | `boolean \| null` | `true` pour activer le filigrane, `false` pour désactiver, `null` pour hériter du défaut d'entreprise |

#### Type de champ Stamp

Nouveau type de champ `stamp` pour placer des tampons d'image préconfigurés ou des sceaux d'entreprise sur les documents. Les tampons sont des images PNG en lecture seule définies par le créateur du modèle ou de la demande de signature, rendues au moment de la signature sans interaction du signataire, et intégrées dans le PDF final.

### Modifications

- Ajout de `stamp` aux enums de type de champ dans tous les schémas liés aux champs (modèles, demandes de signature, create-and-send)

### Guide de migration

**De v01.12.01 à v01.13.00 :**

Il s'agit d'un changement additif non incompatible. Les intégrations existantes ne sont pas affectées.

- Si vous créez des champs via l'API, vous pouvez désormais utiliser `"stamp"` comme type de champ. Les champs de tampon doivent être définis avec `read_only: true` et l'image du tampon en tant que data URL PNG dans `read_only_value`.
- Les champs de tampon sont également pris en charge dans les balises d'ancrage. Lors de l'utilisation de balises d'ancrage, l'image du tampon doit être fournie via `read_only_value` en tant que data URL PNG.

## v01.12.01

**Date de publication :** 2 avril 2026  
[Télécharger la spécification OpenAPI](/api-reference/v01.12.01/openapi-v01.12.01.json)

### Nouvelles fonctionnalités

#### `variable_defined_name` sur les réponses de champs

Tous les endpoints de l'API qui retournent des champs de modèle ou de demande de signature incluent désormais une propriété `variable_defined_name`. C'est le `field_name` lisible par l'humain provenant de la définition de champ personnalisé à laquelle le champ est lié.

Auparavant, les champs liés à des définitions de champs personnalisés n'exposaient que `variable_name`, qui contient l'identifiant interne (par ex. `custom_template_a1b2c3d4-...`). Le nouveau `variable_defined_name` retourne le nom que vous avez initialement défini (par ex. `artist_name`), ce qui facilite le mappage des champs dans votre intégration sans référence croisée à l'endpoint des champs personnalisés.

**Endpoints affectés :**
- `GET /templates` (liste et unique)
- `GET /templates/{id}/fields`
- `POST /templates/{id}/replace-document`
- `GET /signing-requests` (liste)
- `GET /signing-requests/{id}/fields`
- `GET /documents` (héritage)

**Exemple de réponse :**
```json
{
  "variable_name": "custom_template_a1b2c3d4-...",
  "variable_defined_name": "artist_name"
}
```

Pour les champs non liés à une définition de champ personnalisé, `variable_defined_name` vaut `null`.

#### `variable_defined_name` comme entrée pour la correspondance de champs

Lors de la création d'une demande de signature à partir d'un modèle via `POST /signing-requests/create-and-send`, vous pouvez désormais utiliser `variable_defined_name` pour cibler des champs pour des surcharges au lieu du `variable_name` interne :

```json
{
  "fields": [
    { "variable_defined_name": "artist_name", "read_only_value": "John Smith" }
  ]
}
```

`variable_name` et `variable_defined_name` sont tous deux acceptés. `variable_name` prime lorsque les deux sont fournis.

### Aucune modification incompatible

Il s'agit d'une version patch non incompatible. Tout le comportement existant de l'API est inchangé.

## v01.11.00

**Date de publication :** 19 mars 2026  
[Télécharger la spécification OpenAPI](/api-reference/v01.11.00/openapi-v01.11.00.json)

### Nouvelles fonctionnalités

#### Type de champ de téléversement de fichier

Un nouveau type de champ `file` permet aux signataires de téléverser des pièces jointes de fichier. Pris en charge dans les demandes de signature et les modèles.

- Les signataires peuvent téléverser des images (JPG, PNG) et/ou des fichiers PDF selon la configuration
- Les fichiers sont validés par les magic bytes, pas seulement par l'extension
- Taille maximale de fichier : 10 Mo

**Configuration via `format_rules` :**

```json
{
  "type": "file",
  "format_rules": {
    "acceptedFileTypes": "image_and_pdf"
  }
}
```

| Valeur `acceptedFileTypes` | Formats acceptés |
|--------------------------|------------------|
| `image_and_pdf` (défaut) | JPG, PNG, PDF |
| `image` | JPG, PNG uniquement |
| `pdf` | PDF uniquement |

#### Balises d'ancrage pour le placement automatique de champs

Un nouveau tableau `anchor_tags` à la création de la demande de signature permet le placement automatique de champs en utilisant des marqueurs de texte intégrés dans les documents PDF (par ex. `{{SIGN_HERE}}`). Le texte d'ancrage est retiré du PDF après traitement par défaut.

- Jusqu'à 100 balises d'ancrage par requête
- Disponible uniquement pour la création basée sur document (pas basée sur modèle)
- Les champs créés à partir de balises d'ancrage sont ajoutés aux côtés de tout champ spécifié manuellement

**Propriétés clés :**

| Propriété | Type | Défaut | Description |
|----------|------|---------|-------------|
| `anchor_string` | `string` (requis) | — | Texte à rechercher dans le PDF |
| `type` | `string` (requis) | — | Type de champ à placer à l'emplacement de l'ancrage |
| `recipient_id` | `string\|integer` (requis) | — | Destinataire attribué au champ |
| `x_offset` / `y_offset` | `number` | `0` | Décalage par rapport à la position d'ancrage |
| `offset_units` | `percent` \| `pixels` | `percent` | Type d'unité pour les décalages |
| `width` / `height` | `number` | varie selon le type | Dimensions du champ en pourcentage de la page |
| `occurrence` | `integer` | `0` (tous) | Quelle occurrence cibler (0 = toutes) |
| `ignore_if_not_present` | `boolean` | `false` | Ignorer sans erreur si l'ancrage n'est pas trouvé |
| `remove_anchor_text` | `boolean` | `true` | Retirer le texte d'ancrage du PDF |
| `case_sensitive` | `boolean` | `false` | Correspondance sensible à la casse |
| `match_whole_word` | `boolean` | `true` | Correspondance des mots entiers uniquement |

**Exemple :**

```json
{
  "anchor_tags": [
    {
      "anchor_string": "{{SIGN_HERE}}",
      "type": "signature",
      "recipient_id": "temp_1",
      "width": 25,
      "height": 5
    }
  ]
}
```

#### Couleur d'arrière-plan de champ

Tous les types de champs prennent désormais en charge une propriété `background_color` — une chaîne de couleur hexadécimale (par ex. `#FFFDE7`, `#fff`) utile pour mettre en évidence les champs qui nécessitent une attention. Disponible sur les champs standards et les définitions de balises d'ancrage.

| Schéma | Description |
|--------|-------------|
| `FileFormatRules` | Règles de formatage pour les champs de téléversement de fichier (enum `acceptedFileTypes`) |
| `AnchorTag` | Définition de balise d'ancrage pour le placement automatique de champ (~25 propriétés) |

#### Field / TemplateField / SigningRequestField

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `background_color` | `string \| null` | Couleur hex pour l'arrière-plan du champ (par ex. `#FFFDE7`) |

#### Enum de type de champ

- Ajout de `file` à la liste des types de champs acceptés

#### `format_rules`

- Accepte désormais `FileFormatRules` (avec `acceptedFileTypes`) en plus de `DateFormatRules` et du texte d'affichage d'URL

#### Création de demande de signature

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `anchor_tags` | `AnchorTag[]` | Balises d'ancrage pour le placement automatique de champ (100 max) |

### Modifications incompatibles

#### Le défaut de `use_signing_order` a changé

Le défaut du paramètre `use_signing_order` est passé de `false` à `true`. Cela signifie que les nouvelles demandes de signature imposeront l'ordre de signature par défaut. Lorsque `true`, les signataires reçoivent le document en séquence selon leur champ `order`. Définissez `use_signing_order: false` explicitement pour préserver le comportement précédent où tous les signataires reçoivent le document simultanément.

### Guide de migration

Si vous vous appuyez sur le comportement précédent où tous les signataires reçoivent les documents simultanément, définissez explicitement `use_signing_order: false` lors de la création des demandes de signature. Examinez tout code d'intégration qui crée des demandes de signature sans spécifier ce champ.

## v01.10.00

**Date de publication :** 10 mars 2026  
[Télécharger la spécification OpenAPI](/api-reference/v01.10.00/openapi-v01.10.00.json)

### Nouvelles fonctionnalités

#### Logique de champ conditionnelle

Les champs prennent désormais en charge des règles conditionnelles à la fois pour le statut **required** et pour la **visibilité**. Deux nouvelles propriétés nullables sont disponibles sur les objets de champ :

- `required_conditions` — Lorsqu'il est défini, remplace le drapeau `required` statique. Le champ n'est requis que lorsque les conditions évaluent à true en fonction d'autres valeurs de champs.
- `visibility_conditions` — Lorsqu'il est défini, le champ est masqué à moins que les conditions n'évaluent à true. Les champs masqués ne sont pas validés à la soumission.

Les conditions utilisent une structure de groupes imbriquée avec des opérateurs logiques :

| Schéma | Description |
|--------|-------------|
| `ConditionSet` | Conteneur de plus haut niveau avec un opérateur `logic` (`and`/`or`) et un tableau de `groups` |
| `ConditionGroup` | Contient un tableau de `conditions`, combinées en utilisant l'opposé de l'opérateur logique du parent |
| `Condition` | Évalue un champ unique par `field_id`, `operator` et `value` optionnelle |

**Opérateurs pris en charge :** `is_filled`, `is_empty`, `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `less_than`, `greater_than_or_equal`, `less_than_or_equal`

**Exemple — rendre un champ requis uniquement lorsqu'un autre champ est rempli :**
```json
{
  "required_conditions": {
    "logic": "and",
    "groups": [
      {
        "conditions": [
          {
            "field_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
            "operator": "is_filled"
          }
        ]
      }
    ]
  }
}
```

#### Prise en charge des documents DOCX

Tous les endpoints de téléversement de document acceptent désormais les **fichiers DOCX** en plus du PDF. Les fichiers DOCX sont automatiquement convertis en PDF au téléversement. Cela s'applique à :

- Création de modèle (`POST /templates`)
- Remplacement de document de modèle (`POST /templates/{id}/replace-document`)
- Création de demande de signature (`POST /signing-requests`)
- Mises à jour de modèle/demande de signature (PUT et PATCH)

#### Endpoint de piste d'audit

Un nouvel endpoint récupère la piste d'audit complète d'une demande de signature, combinant les actions d'administrateur (created, edited, sent, cancelled) et les actions de signataire (viewed, signed, declined, downloaded). Les événements sont triés chronologiquement.

**Endpoint :** `GET /signing-requests/{id}/audit`

**Champs de réponse :**

| Champ | Type | Description |
|-------|------|-------------|
| `id` | `uuid` | ID de l'événement d'audit |
| `timestamp` | `date-time` | Quand l'événement s'est produit |
| `source` | `string` | `admin` ou `signer` |
| `event` | `string` | Identifiant du type d'événement |
| `description` | `string` | Description de l'événement lisible par l'humain |
| `actor` | `object \| null` | Qui a effectué l'action |
| `ip_address` | `string \| null` | Adresse IP (événements de signataire uniquement) |
| `details` | `object \| null` | Métadonnées supplémentaires spécifiques à l'événement |

#### Contrôle du cadre de signature

Un nouveau paramètre `show_signature_frame` contrôle si un cadre visuel avec un ID de signature est rendu autour des signatures dans les PDF terminés. Disponible aux niveaux entreprise, espace de travail et demande de signature.

| Niveau | Champ | Comportement |
|-------|-------|----------|
| Company | `show_signature_frame` | Définit le défaut (activé par défaut) |
| Workspace Settings | `show_signature_frame` | Remplace le défaut de l'entreprise ; `null` hérite |
| Signing Request Settings | `show_signature_frame` | Remplace le défaut de l'espace de travail ; `null` hérite |

**Valeurs :**
- `true` — Afficher le cadre de signature avec l'ID de signature
- `false` — Masquer le cadre de signature
- `null` — Hériter du niveau parent

#### Suppression forcée des conditions à la suppression d'utilisateur

Lors de la suppression de destinataires dont les champs sont référencés par des conditions dans d'autres champs, un nouveau paramètre booléen `force_remove_conditions` contrôle le comportement :

- `false` (défaut) — La requête est rejetée avec une erreur listant les champs dépendants
- `true` — Retire automatiquement les références de condition et procède à la suppression

Cela s'applique aux opérations de suppression d'utilisateur de modèle et de demande de signature.

### Modifications de schéma

#### Field

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `required_conditions` | `ConditionSet \| null` | Règles conditionnelles pour quand ce champ est requis |
| `visibility_conditions` | `ConditionSet \| null` | Règles conditionnelles pour quand ce champ est visible |

#### Nouveaux schémas

| Schéma | Description |
|--------|-------------|
| `ConditionSet` | Un ensemble de groupes de conditions avec une logique imbriquée (contient `logic` et `groups`) |
| `ConditionGroup` | Un groupe de conditions (contient un tableau `conditions`) |
| `Condition` | Une condition unique évaluant la valeur d'un champ (contient `field_id`, `operator`, `value`) |

#### SigningRequestSettings / WorkspaceSettings / Company

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `show_signature_frame` | `boolean \| null` | Si les cadres de signature doivent être rendus dans les PDF terminés |

#### Suppression d'utilisateur de modèle et de demande de signature

| Paramètre ajouté | Type | Description |
|-----------------|------|-------------|
| `force_remove_conditions` | `boolean` | Force le retrait des références de condition lors de la suppression d'utilisateurs dont les champs sont référencés (défaut : `false`) |

### Nouveaux endpoints

| Endpoint | Méthode | Description |
|----------|--------|-------------|
| `/signing-requests/{id}/audit` | GET | Récupérer la piste d'audit d'une demande de signature |

### Guide de migration

Aucune modification incompatible. Toutes les nouvelles fonctionnalités sont additives :
- Les propriétés de champ conditionnelles ont pour défaut `null` (pas de conditions)
- `show_signature_frame` a pour défaut `null` (hériter, ce qui est activé par défaut)

- La prise en charge DOCX est transparente — les téléversements PDF existants continuent de fonctionner inchangés
- L'endpoint de piste d'audit est purement additif

### Nouvelles fonctionnalités

#### Vérification d'e-mail OTP

Un nouveau paramètre `require_otp_verification` vous permet d'exiger que les signataires vérifient leur adresse e-mail avec un code à usage unique avant d'accéder aux documents de signature. Cela ajoute une couche supplémentaire de vérification d'identité.

**Où il est disponible :**

| Niveau | Champ | Comportement |
|-------|-------|----------|
| Company | `require_otp_verification` | Définit le défaut pour tous les espaces de travail |
| Workspace Settings | `require_otp_verification` | Remplace le défaut de l'entreprise ; `null` hérite de l'entreprise |
| Signing Request Settings | `require_otp_verification` | Remplace le défaut de l'espace de travail ; `null` hérite de l'espace de travail |

**Valeurs :**
- `true` — Exiger la vérification OTP avant l'accès au document
- `false` — Ne pas exiger la vérification OTP
- `null` — Hériter du niveau parent (workspace → company, signing request → workspace)

**Exemple — activer OTP au niveau de l'espace de travail :**
```json
PATCH /workspace-settings/{workspace_id}
{
  "settings": {
    "require_otp_verification": true
  }
}
```

**Exemple — remplacer pour une demande de signature spécifique :**
```json
PATCH /signing-requests/{id}
{
  "settings": {
    "require_otp_verification": false
  }
}
```

#### Remplacement du document d'un modèle

Un nouvel endpoint permet de remplacer le document PDF d'un modèle tout en préservant tous les placements de champs. C'est utile lorsque vous devez mettre à jour un document (par ex. corriger une faute de frappe) sans recréer toutes les configurations de champs.

**Endpoint :** `POST /templates/{id}/replace-document`

**Exigences :**
- Le document de remplacement doit avoir le **même nombre de pages** que l'original
- Les dimensions des pages doivent correspondre à **1pt de tolérance près**
- Le document doit être fourni en tant que **PDF encodé en base64**

**Exemple :**
```json
POST /templates/{template_id}/replace-document
{
  "document": "JVBERi0xLjQKJeLjz9..."
}
```

**Cas d'erreur :**
| Erreur | Cause |
|-------|-------|
| 400 | Nombre de pages différent |
| 400 | Dimensions de page différentes (dépasse la tolérance de 1pt) |
| 400 | Document PDF invalide ou corrompu |

#### SigningRequestSettings

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `require_otp_verification` | `boolean \| null` | Exiger la vérification OTP par e-mail ; `null` hérite de l'espace de travail |

#### WorkspaceSettings

| Champ ajouté | Type | Description |
|-------------|------|-------------|
| `require_otp_verification` | `boolean \| null` | Exiger la vérification OTP par e-mail ; `null` hérite de l'entreprise |

### Nouveaux endpoints

| Endpoint | Méthode | Description |
|----------|--------|-------------|
| `/templates/{id}/replace-document` | POST | Remplacer le PDF du modèle en préservant les champs |

### Guide de migration

Aucune modification incompatible. Le champ `require_otp_verification` a pour défaut `null` (hériter), donc le comportement existant est inchangé. L'endpoint de remplacement de document est purement additif.
