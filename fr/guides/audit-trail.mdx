Chaque demande de signature dans Firma dispose d'une piste d'audit complète. Chaque consultation de document, interaction de champ, signature, refus et action d'administration est enregistré avec des horodatages, l'identité de l'acteur et les adresses IP.

Cas d'usage courants :

- **Conformité et preuve légale** — prouvez qui a signé quoi, quand et depuis où
- **Débogage des problèmes de signataire** — voyez exactement ce qu'un destinataire a fait pendant sa session de signature
- **Reporting interne** — suivez les actions d'admin comme la création, les modifications et les annulations

La piste d'audit capture deux catégories d'événements :

- **Événements de signataire** — consultations de documents, interactions de champs, signatures, refus, téléchargements de certificats
- **Événements d'admin** — création, modifications, envoi, annulation, renvoi (via tableau de bord ou API)

Chaque événement inclut un horodatage, une description lisible par l'humain, l'identité de l'acteur et (pour les événements de signataire) l'adresse IP du signataire.

## Consulter la piste d'audit dans le tableau de bord

Ouvrez n'importe quelle demande de signature dans le tableau de bord Firma et cliquez sur l'onglet **Audit Trail**. Vous verrez une chronologie de tous les événements, avec les actions de signataire et d'admin visuellement distinguées. Les événements répétitifs consécutifs (comme le défilement ou les interactions de champ répétées) sont automatiquement condensés avec un indicateur de comptage.

## Récupérer la piste d'audit via l'API

Utilisez l'endpoint d'audit pour récupérer l'historique complet des événements d'une demande de signature :

```
GET /signing-requests/{id}/audit
```

### curl

```bash
curl -X GET "https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit" \
  -H "Authorization: YOUR_API_KEY"
```

### Node.js

```js
const response = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequestId}/audit`,
  {
    headers: {
      'Authorization': process.env.FIRMA_API_KEY,
    },
  }
);

const { results } = await response.json();
console.log(results);
```

### Python

```python
import os
import requests

response = requests.get(
    f"https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit",
    headers={
        "Authorization": os.environ["FIRMA_API_KEY"],
    },
)

results = response.json()["results"]
```

### Champs de la réponse

| Champ | Type | Description |
|-------|------|-------------|
| `id` | `uuid` | ID de l'événement |
| `timestamp` | `datetime` | Quand l'événement s'est produit |
| `source` | `string` | `"signer"` ou `"admin"` |
| `event` | `string` | Identifiant du type d'événement (voir [référence des événements](#event-reference) ci-dessous) |
| `description` | `string` | Description de l'événement lisible par l'humain |
| `actor` | `object \| null` | Qui a effectué l'action (voir ci-dessous) |
| `ip_address` | `string \| null` | Adresse IP du signataire (événements de signataire uniquement) |
| `details` | `object \| null` | Métadonnées spécifiques à l'événement |
| `condensed_count` | `integer` | Présent lorsque des événements identiques consécutifs du même acteur sont regroupés |

## Comprendre les sources d'événements

Le champ `source` vous indique si un événement provient d'un signataire ou d'un admin, et le champ `actor` a une forme différente pour chacun.

### Événements de signataire

Les événements avec `source: "signer"` sont des actions effectuées par les destinataires pendant le flux de signature. L'objet `actor` contient le `name` et l'`email` du signataire. Le champ `ip_address` est rempli.

```json
{
  "source": "signer",
  "actor": { "name": "Alice Johnson", "email": "alice@example.com" },
  "ip_address": "203.0.113.42"
}
```

### Événements d'admin

Les événements avec `source: "admin"` sont des actions effectuées via le tableau de bord ou l'API. L'objet `actor` contient un champ `type` (`"api_key"` ou `"user"`) et optionnellement un `user_id`. Le champ `ip_address` vaut `null`.

```json
{
  "source": "admin",
  "actor": { "type": "api_key" },
  "ip_address": null
}
```

```json
{
  "source": "admin",
  "actor": { "type": "user", "user_id": "usr_abc123" },
  "ip_address": null
}
```

## Suivi en temps réel via les webhooks

Si vous avez besoin d'une notification en temps réel des événements de signataire plutôt que de sonder l'endpoint d'audit, Firma prend en charge les webhooks. Consultez le [guide des Webhooks](webhooks) pour les instructions de configuration.

## Référence des événements

### Cycle de vie du document

| Événement | Description | A des `details` |
|-------|-------------|---------------|
| `document_opened` | Le document a été ouvert | Non |
| `document_viewed` | Le document a été consulté | Non |
| `document_saved` | Valeurs de champ enregistrées | Non |
| `document_finished` | Signature terminée | Non |
| `document_signed` | Le document a été signé | Non |
| `document_hidden` | A quitté la page | Non |
| `document_visible` | Est revenu sur la page | Non |
| `document_closed` | Le document a été fermé | Non |

### Signature et vérification

| Événement | Description | A des `details` |
|-------|-------------|---------------|
| `signature_finalized` | Signature finalisée | Non |
| `signing_declined` | A refusé de signer | Non |
| `terms_accepted` | A accepté les conditions générales | Non |
| `otp_verified` | Identité vérifiée via OTP | Non |
| `certificate_download` | A téléchargé le certificat | Non |

### Interactions de champs

Les interactions de champs sont suivies en tant qu'événements `field_interaction`. L'objet `details` contient l'action spécifique et les métadonnées du champ.

| Action | Description |
|--------|-------------|
| `field_focused` | Sélection d'un champ |
| `field_completed` | Champ complété pour la première fois |
| `field_modified` | Valeur d'un champ existant modifiée |
| `field_cleared` | Valeur d'un champ effacée |
| `field_blur` | Fin de l'édition d'un champ |

L'objet `details` pour les interactions de champs inclut :

| Champ | Type | Description |
|-------|------|-------------|
| `field_type` | `string` | Type de champ (signature, text, date, etc.) |
| `action` | `string` | L'action spécifique du tableau ci-dessus |
| `interaction_count` | `integer` | Nombre de fois que ce champ a été utilisé |
| `value_length` | `integer` | Longueur de la valeur du champ (lorsque applicable) |
| `time_spent_ms` | `integer` | Temps passé sur le champ en millisecondes (lorsque applicable) |

### Navigation

| Événement | Description | A des `details` |
|-------|-------------|---------------|
| `page_viewed` | Page spécifique consultée | Oui — `page_number` |
| `document_scrolled` | Défilement vers une page | Oui — `page_number` |
| `navigation_action` | Navigation dans le document | Oui — `action` (`next_page`, `prev_page`, `go_to_page`) |
| `zoom_changed` | Niveau de zoom modifié | Non |
| `modal_interaction` | Ouverture ou fermeture d'une boîte de dialogue | Oui — `modal_type`, `action` |

### Activité d'admin

Les événements d'admin utilisent `event: "activity"` avec un champ `description` lisible par l'humain qui décrit l'action effectuée (par ex. « Created signing request via API », « Sent signing request », « Cancelled signing request »).
