- **Accès partitionné :** les utilisateurs, modèles et activités sont limités à un seul espace de travail.
- **Suivi de l'utilisation :** voyez les enveloppes envoyées par espace de travail pour un reporting propre.
- **Sécurité dès la conception :** pas d'exposition des documents ou données entre espaces de travail.

Cela facilite l'attribution à chaque client ou équipe de son propre environnement de signature privé, tout en gérant l'ensemble via la même API.

## Endpoints principaux

Endpoints principaux (voir la référence de l'API pour tous les détails) :

- POST `/workspaces` — créer un espace de travail
- GET `/workspaces/{workspace_id}` — récupérer les détails d'un espace de travail
- PUT `/workspaces/{workspace_id}` — mettre à jour un espace de travail
- GET `/workspaces` — lister les espaces de travail (pagination)
- GET `/workspace/{workspace_id}/settings` — récupérer les paramètres d'un espace de travail
- PUT `/workspace/{workspace_id}/settings` — mettre à jour les paramètres d'un espace de travail

<Note>
  Consultez le guide sur les [Limites de débit](rate-limits).
</Note>


Vous trouverez également des pages de référence dédiées sous `api-reference/v01.15.00/workspaces` :

- `api-reference/v01.15.00/workspaces/create-a-new-workspace`
- `api-reference/v01.15.00/workspaces/list-workspaces`
- `api-reference/v01.15.00/workspaces/update-a-workspace`

Authentification

Toutes les requêtes doivent inclure un en-tête Authorization utilisant votre clé API :

```
Authorization: Bearer YOUR_API_KEY
```

Note : n'exposez jamais votre clé API principale aux navigateurs des utilisateurs finaux. Pour les opérations par espace de travail qui doivent être lancées depuis le frontend, appelez votre backend qui détient la clé.

## Créer un espace de travail

Endpoint : POST `/workspaces`

Corps requis : `{ "name": "string" }`

Exemple (curl) :

```bash
curl -X POST "https://api.firma.dev/functions/v1/signing-request-api/workspaces" \
  -H "Authorization: Bearer $FIRMA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Acme Corp Workspace" }'
```

Une réponse réussie (201) retourne la ressource `Workspace`, incluant les clés `api_key` (live) et `test_api_key` de l'espace de travail. Exemple de forme :

```json
{
  "id": "workspace_123",
  "name": "Acme Corp Workspace",
  "protected": false,
  "api_key": "firma_live_…",
  "test_api_key": "firma_test_…",
  "created_date": "2025-09-04T12:00:00Z",
  "updated_date": "2025-09-04T12:00:00Z"
}
```

`api_key` est la clé live de l'espace de travail et `test_api_key` est sa clé de test. Consultez [Authentification](authentication) pour savoir comment fonctionne le mode test.

Exemple serveur — Node (fetch) :

```js
import fetch from 'node-fetch'

export async function createWorkspace(name) {
  const resp = await fetch('https://api.firma.dev/functions/v1/signing-request-api/workspaces', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ name })
  })
  if (!resp.ok) throw new Error(`create workspace failed: ${resp.status}`)
  return resp.json()
}
```

Exemple serveur — Python (requests) :

```py
import os
import requests

def create_workspace(name):
    url = 'https://api.firma.dev/functions/v1/signing-request-api/workspaces'
    resp = requests.post(url, headers={
        'Authorization': f"Bearer {os.environ['FIRMA_API_KEY']}",
        'Content-Type': 'application/json'
    }, json={ 'name': name })
    resp.raise_for_status()
    return resp.json()
```

## Récupérer un espace de travail

Endpoint : GET `/workspaces/{workspace_id}`

Exemple (curl) :

```bash
curl "https://api.firma.dev/functions/v1/signing-request-api/workspaces/workspace_123" \
  -H "Authorization: Bearer $FIRMA_API_KEY"
```

Réponse : la ressource `Workspace` (voir exemple ci-dessus).

## Mettre à jour un espace de travail

Endpoint : PUT `/workspaces/{workspace_id}` — remplacement complet de la ressource. Corps requis identique à `CreateWorkspace` (par ex., `{ "name": "New name" }`).

Exemple (curl) :

```bash
curl -X PUT "https://api.firma.dev/functions/v1/signing-request-api/workspaces/workspace_123" \
  -H "Authorization: Bearer $FIRMA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Acme Corp - East Coast" }'
```

Exemple serveur — Node (fetch) :

```js
export async function updateWorkspace(workspaceId, name) {
  const resp = await fetch(`https://api.firma.dev/functions/v1/signing-request-api/workspaces/${workspaceId}`, {
    method: 'PUT',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ name })
  })
  if (!resp.ok) throw new Error(`update workspace failed: ${resp.status}`)
  return resp.json()
}
```

## Lister les espaces de travail

Endpoint : GET `/workspaces`

Prend en charge les paramètres de requête `page` et `page_size`. Exemple :

```bash
curl "https://api.firma.dev/functions/v1/signing-request-api/workspaces?page=1&page_size=20" \
  -H "Authorization: Bearer $FIRMA_API_KEY"
```

La forme de la réponse contient `results` (tableau de `Workspace`) et des métadonnées `pagination`.

## Bonnes pratiques RBAC et clé API

- Utilisez des clés API à portée limitée lorsque possible : conservez une clé API maître côté serveur uniquement et créez des clés API par espace de travail pour les partenaires d'intégration si votre plateforme le prend en charge.
- Ne renvoyez jamais de clés API au navigateur. Pour les opérations initiées par les utilisateurs finaux, appelez votre backend qui effectue l'appel d'API à Firma.
- Utilisez l'idempotence lorsqu'elle est prise en charge (pour les endpoints qui acceptent `Idempotency-Key`) pour éviter la double création due aux tentatives.

## Cas limites et dépannage

- Noms en double : les noms d'espace de travail sont conçus pour l'affichage et peuvent ne pas être uniques ; utilisez l'`id` pour les références programmatiques.
- Espaces de travail protégés : le drapeau `protected` indique les espaces de travail détenus par le système qui ne peuvent pas être supprimés ou modifiés par les flux normaux.
- Limites de débit et erreurs : inspectez les réponses non-2xx pour `{ error, code }` et faites remonter des messages significatifs aux opérateurs.
- **Limite de débit dépassée (429)** : surveillez l'en-tête `X-RateLimit-Remaining` et implémentez un backoff exponentiel pour les nouvelles tentatives.
