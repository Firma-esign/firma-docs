# Guida al versioning dell'API

L'API di Firma utilizza il versioning basato su header tramite l'header `X-API-Version`. Questo ti permette di specificare quale versione dell'API vuoi utilizzare, con avvisi di deprecazione automatici e un processo di sunset strutturato.

## Come specificare una versione

Includi l'header `X-API-Version` nella tua richiesta:

<CodeGroup>

```bash cURL
curl https://api.firma.dev/functions/v1/signing-request-api/signing-requests \
  -H "X-API-Version: 1" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```javascript JavaScript
const response = await fetch(
  'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
  {
    headers: {
      'X-API-Version': '1',
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`
    }
  }
);
```

```python Python
import requests

headers = {
    'X-API-Version': '1',
    'Authorization': f'Bearer {os.environ["FIRMA_API_KEY"]}'
}

response = requests.get(
    'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
    headers=headers
)
```

</CodeGroup>

<Note>
Se l'header `X-API-Version` viene omesso, l'API utilizza per impostazione predefinita la versione stabile corrente (attualmente **1**).
</Note>

## Ciclo di vita delle versioni

Ogni versione dell'API attraversa tre fasi:

| Stato | Descrizione | Comportamento |
|--------|-------------|----------|
| **active** | Versione completamente supportata | Risposte normali con l'header `X-API-Version` |
| **deprecated** | Ancora funzionante ma programmata per la rimozione | Le risposte includono header di avviso di deprecazione |
| **sunset** | Rimossa dal servizio | Restituisce un errore `410 Gone` |

## Header di risposta

### Per le versioni attive

```
X-API-Version: 1
```

### Per le versioni deprecate

Quando una versione è deprecata, i seguenti header vengono inclusi in ogni risposta:

```
X-API-Version: 1
Deprecation: 2025-06-01
Sunset: 2025-12-01
X-Deprecated-Message: This API version is deprecated and will be removed on 2025-12-01. Please migrate to version 2.
```

| Header | Descrizione |
|--------|-------------|
| `Deprecation` | Conforme a RFC 8594 - Data in cui la versione è stata deprecata |
| `Sunset` | Conforme a RFC 8594 - Data in cui la versione sarà rimossa |
| `X-Deprecated-Message` | Messaggio di avviso leggibile |

## Risposte di errore

### Header di versione non valido

```json
// 400 Bad Request
{
  "error": "Invalid X-API-Version header. Must be a positive integer.",
  "code": "INVALID_VERSION"
}
```

### Versione non supportata

```json
// 400 Bad Request
{
  "error": "API version 99 is not supported. Supported versions: 1",
  "code": "UNSUPPORTED_VERSION"
}
```

### Versione sunset (rimossa)

```json
// 410 Gone
{
  "error": "API version 1 has been removed as of 2025-12-01. Please migrate to version 2.",
  "code": "VERSION_SUNSET"
}
```

## Politica sulle breaking change

<Info>
Le breaking change vengono introdotte solo nell'incremento delle versioni maggiori (ad es. v1 → v2). Le modifiche non breaking, come nuovi campi opzionali o nuovi endpoint, non richiedono una nuova versione.
</Info>

Quando viene rilasciata una nuova versione maggiore:

1. **La versione precedente viene marcata come deprecata**
2. **Riceverai header di avviso** per un periodo di deprecazione consigliato di 6 mesi
3. **Dopo la data di sunset**, la vecchia versione restituisce `410 Gone`

### Cosa costituisce una breaking change?

Le breaking change includono:
- Rimuovere o rinominare campi esistenti
- Cambiare i tipi di campo o le regole di validazione
- Rimuovere endpoint
- Modificare i requisiti di autenticazione
- Modificare significativamente i rate limit

Le modifiche non breaking includono:
- Aggiungere nuovi campi opzionali
- Aggiungere nuovi endpoint
- Aggiungere nuovi valori enum (se il tuo client gestisce con eleganza i valori sconosciuti)
- Migliorare i messaggi di errore

## Best practice

<AccordionGroup>

<Accordion title="Controlla sempre gli header di deprecazione">
Implementa controlli automatici nel codice del tuo client API per rilevare gli header `Deprecation` e `Sunset`. Registra dei warning quando questi header sono presenti.

```javascript
const response = await fetch(url, options);

if (response.headers.get('Deprecation')) {
  console.warn('API Deprecation Warning:', {
    deprecation: response.headers.get('Deprecation'),
    sunset: response.headers.get('Sunset'),
    message: response.headers.get('X-Deprecated-Message')
  });
}
```
</Accordion>

<Accordion title="Specifica esplicitamente la versione">
Invece di affidarti alla versione predefinita, specifica sempre esplicitamente l'header `X-API-Version`. Questo previene breaking change inaspettate quando la versione predefinita cambia.

```javascript
// Consigliato - versione esplicita
headers: { 'X-API-Version': '1' }

// Da evitare - fa affidamento sulla predefinita
headers: { }
```
</Accordion>

<Accordion title="Iscriviti agli aggiornamenti del changelog">
Rimani informato sulle prossime modifiche iscrivendoti al changelog o agli annunci sull'API. Questo ti dà un preavviso su:
- Rilasci di nuove versioni
- Programmi di deprecazione
- Breaking change
- Guide alla migrazione
</Accordion>

<Accordion title="Testa in anticipo le nuove versioni">
Quando viene annunciata una nuova versione, testa la tua integrazione con essa ben prima della data di sunset della vecchia versione. Questo ti dà il tempo di:
- Identificare le breaking change nella tua implementazione
- Aggiornare il codice e le dipendenze
- Testare a fondo in staging
- Effettuare il deploy con fiducia prima della scadenza
</Accordion>

</AccordionGroup>

## Stato della versione corrente

La versione attiva corrente è **v1** senza deprecazione programmata al momento.

<Note>
Quando vengono rilasciate nuove versioni o vengono annunciate scadenze di deprecazione, questa guida verrà aggiornata con date specifiche e informazioni sulla migrazione.
</Note>
