## v1.27.0

**Typ wydania:** Wydanie poprawkowe

[Pobierz specyfikację OpenAPI](/api-reference/v01.27.00/openapi-v01.27.00.json)

### Pola ustawień teraz udokumentowane w odpowiedziach

To wydanie dokumentuje pola ustawień, które API już zwraca. Wszystkie zmiany są addytywne i wstecznie kompatybilne — żadne zachowanie API nie zmieniło się, zaktualizowano tylko specyfikację OpenAPI, aby pasowała do tego, co endpointy już wysyłają.

#### Ustawienia żądania podpisu

Obiekt settings zwracany przez endpointy żądań podpisu (`POST /signing-requests`, `POST /signing-requests/create-and-send`, `GET /signing-requests`, `GET /signing-requests/{id}`, `PUT /signing-requests/{id}`, `PATCH /signing-requests/{id}`) dokumentuje teraz cztery dodatkowe pola:

| Pole | Typ | Opis |
|-------|------|-------------|
| `show_qr_code` | boolean lub null | Pokaż kod QR na stronie podpisywania, który pozwala sygnatariuszom kontynuować na telefonie. Dziedziczy z ustawienia przestrzeni roboczej lub firmy, gdy `null`. |
| `allow_presigning_download` | boolean lub null | Pozwól sygnatariuszom pobrać oryginalny dokument przed podpisaniem. Dziedziczy z ustawienia przestrzeni roboczej lub firmy, gdy `null`. |
| `identity_editable_fields` | string[] lub null | Pola tożsamości, które sygnatariusze mogą edytować przed podpisaniem (np. `["name", "company"]`). `null` wyłącza edycję tożsamości. |
| `notify_identity_change_email` | boolean | Wyślij powiadomienie e-mail, gdy sygnatariusz zmieni swoją tożsamość. |

#### Ustawienia szablonu

Obiekt settings zwracany przez endpointy szablonów (`POST /templates`, `PUT /templates/{id}`, `PATCH /templates/{id}`, `GET /templates`, `GET /templates/{id}`, `POST /templates/{id}/replace-document`) dokumentuje teraz pole `show_qr_code`. Ustawienia szablonu obejmują te same opcje co ustawienia żądania podpisu z wyjątkiem pól tożsamości sygnatariusza (`identity_editable_fields`, `notify_identity_change_email`), które mają zastosowanie tylko do żądań podpisu.

#### Odpowiedzi cancel i resend

| Endpoint | Nowe pole odpowiedzi | Typ | Opis |
|----------|--------------------|------|-------------|
| `POST /signing-requests/{id}/cancel` | `emails_sent` | integer | Liczba wysłanych e-maili z powiadomieniem o anulowaniu do sygnatariuszy |
| `POST /signing-requests/{id}/resend` | `recipients_failed` | integer | Liczba odbiorców, których powiadomienie o ponownym wysłaniu nie zostało wysłane |

### Nowe schematy

Dwa nowe schematy dzielą kształty ustawień, które różnią się między endpointami, dzięki czemu każda odpowiedź dokumentuje dokładnie te pola, które zwraca:

| Schemat | Opis |
|--------|-------------|
| `TemplateSettings` | Ustawienia zwracane przez endpointy szablonów. Pasuje do `SigningRequestSettings` minus pola tożsamości sygnatariusza (`identity_editable_fields`, `notify_identity_change_email`), które mają zastosowanie tylko do żądań podpisu. |
| `LegacyDocumentSettings` | Zredukowany zestaw ustawień zwracany przez przestarzały endpoint `POST /documents`. Nowe integracje powinny używać endpointów żądań podpisu, które zwracają `SigningRequestSettings`. |

### Przewodnik migracji

Brak zmian łamiących zgodność. Są to dodatki tylko do dokumentacji opisujące pola, które API już zwraca. Istniejące integracje nie są dotknięte. Jeśli parsujesz obiekty ustawień, możesz teraz polegać na udokumentowanych polach `show_qr_code`, `allow_presigning_download`, `identity_editable_fields` i `notify_identity_change_email` oraz na `emails_sent` / `recipients_failed` w odpowiedziach cancel i resend.

## v1.25.2

**Typ wydania:** Wydanie poprawkowe

[Pobierz specyfikację OpenAPI](/api-reference/v01.25.02/openapi-v01.25.02.json)

### Ściślejsza walidacja nadpisań pól szablonu

Podczas tworzenia żądania podpisu z szablonu każde podane pole, które nie pasuje do pola w tym szablonie, jest teraz odrzucane z **`400` `VALIDATION_ERROR`** zamiast cicho ignorowane. Wcześniej takie pole było usuwane, a żądanie nadal się udawało, więc błędnie wpisany identyfikator oznaczał, że pole po prostu nigdy nie pojawiło się na dokumencie bez błędu.

Podane pole pasuje do pola szablonu przez `template_field_id`, `variable_name` lub `variable_defined_name`. Błąd wskazuje problematyczne pole(a):

```json
{
  "error": "The following provided fields do not match any field in the template and cannot be applied: full_name_x. Each provided field must reference an existing template field by 'template_field_id' or 'variable_name'.",
  "code": "VALIDATION_ERROR"
}
```

Ma zastosowanie do `POST /signing-requests` i `POST /signing-requests/create-and-send`.

**Migracja:** upewnij się, że każde pole wysyłane w utworzeniu opartym na szablonie odwołuje się do istniejącego pola szablonu. Jeśli polegałeś na ignorowaniu dodatkowych/niepasujących pól, usuń je.

### Typy pól bez uwzględniania wielkości liter

Wartości `type` pól są teraz normalizowane bez uwzględniania wielkości liter. `"Initials"`, `"Signature"`, `"TextArea"` itp. są rozwiązywane na ich kanoniczny typ (`initial`, `signature`, `text_area`) zamiast być odrzucane jako nieprawidłowe. Wartości pisane małymi literami są niezmienione.

## v1.25.1

**Typ wydania:** Wydanie poprawkowe

[Pobierz specyfikację OpenAPI](/api-reference/v01.25.01/openapi-v01.25.01.json)

### Dokładne odpowiedzi błędów dla create-and-send

`POST /signing-requests/create-and-send` wcześniej zwracał ogólny **500**, gdy krok wysyłania odrzucał żądanie, ukrywając prawdziwy powód. Teraz zwraca **status z możliwością działania po stronie klienta** (`400`, `402` lub `422`) z odczytywalnym maszynowo `code` i odpowiednimi szczegółami:

- **`400` `VALIDATION_ERROR`** — walidacja żądania lub wysyłania nie powiodła się (np. sygnatariusz z brakującymi wymaganymi informacjami); odpowiedź zawiera `validation_errors`, `field_errors` lub `details` identyfikujące, co naprawić.
- **`402` `INSUFFICIENT_CREDITS`** — niewystarczająca liczba kredytów do wysłania; `current_credits` raportuje saldo.
- **`422` `RECIPIENT_EMAIL_SUPPRESSED`** *(nowy)* — adres e-mail odbiorcy jest tłumiony (wcześniej odbity lub oznaczony jako spam) i nie można do niego wysłać.

Prawdziwe niepowodzenia po stronie serwera nadal zwracają `500`. Gdy krok wysyłania zawodzi, częściowo utworzone żądanie podpisu jest cofane, więc nie pozostaje osierocona wersja robocza.

**Przewodnik migracji:** Nie jest wymagana żadna akcja dla klientów, którzy już traktują dowolną odpowiedź inną niż 2xx jako niepowodzenie. Jeśli wcześniej obsługiwałeś specjalnie nieprzejrzysty `500` z tego endpointu, przełącz się na rozgałęzianie na status HTTP / `code`: `400` i `422` oznaczają „napraw żądanie, następnie ponów", `402` oznacza „dodaj kredyty".

## v1.25.0

**Typ wydania:** Wydanie mniejsze

[Pobierz specyfikację OpenAPI](/api-reference/v01.25.00/openapi-v01.25.00.json)

### Kolory marki edytora

Ustawienia wyglądu zawierają teraz pięć dodatkowych kolorów marki, które nadają motyw **osadzonym edytorom** (edytory szablonów i żądań podpisu, które osadzasz na własnych stronach):

- **`color_accent`** — kolor akcentu dla podświetleń, aktywnych zakładek, przełączników i głównych kontrolek w chrome edytora.
- **`color_accent_fg`** — kolor pierwszego planu (tekst/ikony) pokazany na powierzchniach akcentu.
- **`color_canvas`** — otoczenie płótna dokumentu za stroną.
- **`color_muted`** — wyciszony kolor powierzchni (separatory, stany hover, ścieżki przełączników).
- **`color_muted_fg`** — wyciszony kolor tekstu.

Dołączają one do istniejących sześciu pól kolorów, więc endpointy ustawień noszą teraz jedenaście pól `color_*`:

- **`GET`/`PUT /company/settings`** i **`GET`/`PUT /workspace/{workspace_id}/settings`** akceptują i zwracają wszystkie pięć nowych pól. Każde jest nullable łańcuchem hex `#rrggbb`; `null` oznacza dziedziczenie (przestrzeń robocza → firma → domyślne Firmy).
- Rozwiązany obiekt **`color_palette`** zwracany w payloadach osadzonych edytorów zawiera teraz jedenaście kluczy (sześć istniejących plus `accent`, `accent_fg`, `canvas`, `muted`, `muted_fg`). Gdy nie ustawione, `muted` domyślnie ustawia się na rozwiązany kolor `card`, a `muted_fg` jest wyprowadzony dla czytelności, więc paleta jest zawsze w pełni wypełniona.

Wartości przestrzeni roboczej nadpisują wartości firmy; nieustawione wartości wracają do domyślnych Firmy, więc istniejące integracje renderują się bez zmian.

**Przewodnik migracji:** Nie jest wymagana żadna akcja. Wszystkie pięć pól jest addytywnych i nullable. Aby nadać markę osadzonym edytorom, wyślij dowolne z nowych pól `color_*` (prawidłowy hex `#rrggbb` lub `null`, aby wyczyścić) do endpointu ustawień firmy lub przestrzeni roboczej.

## v1.24.0

**Typ wydania:** Wydanie mniejsze

[Pobierz specyfikację OpenAPI](/api-reference/v01.24.00/openapi-v01.24.00.json)

### Testowe klucze API przestrzeni roboczej

Przestrzenie robocze udostępniają teraz **klucz API w trybie testowym** obok istniejącego klucza live. Żądania uwierzytelnione kluczem testowym nie zużywają kredytów i produkują oznaczone jako testowe, opatrzone znakiem wodnym żądania podpisu.

- **`POST /workspaces`** teraz udostępnia parę kluczy live + test i zwraca zarówno `api_key` (live), jak i nowe pole `test_api_key`.
- **`GET /workspaces/{id}`** i **`GET /workspaces`** zwracają teraz zarówno `api_key`, jak i `test_api_key`.
- **`POST /workspaces/{id}/api-key/regenerate`** i **`/api-key/expire`** akceptują opcjonalny `key_type` (`"live"` lub `"test"`, domyślnie `"live"`). Regeneracja jednego typu klucza nie wpływa już na drugi.
- Istniejące przestrzenie robocze utworzone przez API są uzupełniane kluczem testowym.

Wszystkie zmiany są addytywne i wstecznie kompatybilne: konsumenci, którzy odczytują `api_key`, nadal działają bez zmian.

**Przewodnik migracji:** Nie jest wymagana żadna akcja. Aby użyć trybu testowego, odczytaj `test_api_key` z dowolnej odpowiedzi tworzenia/pobierania przestrzeni roboczej i wyślij go jako klucz `Authorization` dla żądań, które chcesz uruchomić w trybie testowym.

## v1.23.0

**Typ wydania:** Wydanie mniejsze

[Pobierz specyfikację OpenAPI](/api-reference/v01.23.00/openapi-v01.23.00.json)

### Warunki sygnatariusza

Nowe endpointy Signer Terms pozwalają zarządzać niestandardowymi oświadczeniami zgody sygnatariusza na poziomie firmy i przestrzeni roboczej:

- Endpointy `GET` listują warunki sygnatariusza firmy i przestrzeni roboczej
- Endpointy `PUT` tworzą lub aktualizują warunki sygnatariusza firmy i przestrzeni roboczej dla każdego języka
- Endpointy `DELETE` usuwają warunki sygnatariusza firmy i przestrzeni roboczej

### Ustawienia firmy

Nowe endpointy `GET /company/settings` i `PUT /company/settings` udostępniają ustawienia firmy, w tym bramkę akceptacji warunków.

### Zmiany schematu

- **Ustawienia firmy** schemat: dodano `require_terms_acceptance` (boolean)
- **Ustawienia przestrzeni roboczej** schemat: dodano `require_terms_acceptance` (boolean, nullable). Użyj `null`, aby dziedziczyć ustawienie firmy.

### Przewodnik migracji

Brak zmian łamiących zgodność. To wydanie jest addytywne i nie jest wymagana żadna akcja dla istniejących integracji.

## v1.22.1

**Typ wydania:** Wydanie poprawkowe

[Pobierz specyfikację OpenAPI](/api-reference/v01.22.01/openapi-v01.22.01.json)

### Usuwanie domeny e-mail

`DELETE /company/domains/{id}` i `DELETE /workspace/{workspace_id}/domains/{id}` teraz kończą się sukcesem, gdy cel jest główną domeną przestrzeni roboczej lub jej jedyną domeną. Wcześniej zwracały `400 "Cannot delete primary domain"`. Po usunięciu wychodząca poczta wraca do domyślnego nadawcy firmy lub domyślnego platformy, jeśli nie jest ustawiony. Aby ponownie użyć niestandardowej domeny wysyłkowej, dodaj i zweryfikuj nową domenę, a następnie ustaw ją jako główną.

### Przewodnik migracji

Brak zmian łamiących zgodność. Odpowiedzi `400 "Cannot delete primary domain"` i „cannot delete the only domain" są usuwane z obu endpointów `DELETE` domeny; teraz zwracają `200` w tych przypadkach. Integracje, które obsługiwały specjalnie ten `400`, mogą usunąć tę obsługę.

## v1.18.0

**Data wydania:** 5 maja 2026

### Nowe funkcje

#### Edytowalne pola tożsamości

Żądania podpisu obsługują teraz edycję tożsamości dla każdego pola, dla każdego sygnatariusza. Po skonfigurowaniu sygnatariusze widzą okno dialogowe potwierdzenia przed podpisaniem, gdzie mogą przeglądać i edytować określone pola tożsamości (imię, firma, tytuł, telefon, adres).

Nowa właściwość ustawień:

| Właściwość | Typ | Domyślne | Opis |
|----------|------|---------|-------------|
| `identity_editable_fields` | `string[]` lub `null` | `null` | Tablica kluczy pól tożsamości, które sygnatariusze mogą edytować. `null` = wyłączone. |
| `notify_identity_change_email` | `boolean` | `false` | Wyślij e-mail, gdy sygnatariusz zmieni swoją tożsamość |

Prawidłowe klucze pól: `name`, `company`, `title`, `phone`, `address`

Nadpisanie dla każdego odbiorcy przez `identity_editable_fields` w obiekcie odbiorcy:
- `null` - użyj domyślnego żądania
- `[]` - wyłączone dla tego sygnatariusza
- `["name", "company"]` - niestandardowe pola dla tego sygnatariusza

Przykład:

```json
{
  "settings": {
    "identity_editable_fields": ["name", "company"]
  },
  "recipients": [
    {
      "name": "John Doe",
      "email": "john@example.com",
      "designation": "signer",
      "order": 1,
      "identity_editable_fields": null
    },
    {
      "name": "Jane Smith",
      "email": "jane@example.com",
      "designation": "signer",
      "order": 2,
      "identity_editable_fields": []
    }
  ]
}
```

W tym przykładzie John dziedziczy domyślne żądania (name + company edytowalne), podczas gdy Jane ma edycję tożsamości wyraźnie wyłączoną.

Nowe zdarzenie webhook: `signing_request.recipient.identity_changed` jest teraz dostępne jako standardowe zdarzenie webhook. Zasubskrybuj je w konfiguracji webhooka, aby otrzymywać powiadomienia, gdy sygnatariusz modyfikuje swoją tożsamość podczas podpisywania.

#### Edytowalne wstępnie wypełnione pola danych

Wstępnie wypełnione pola danych mogą być teraz opcjonalnie edytowalne przez sygnatariuszy. Wcześniej wszystkie pola z ustawionym `prefilled_data` były automatycznie blokowane jako tylko do odczytu. Nowa właściwość `prefilled_editable` pozwala kontrolować to zachowanie.

Gdy `prefilled_editable` jest `true`, wartość pola jest nadal automatycznie wypełniana z danych odbiorcy, ale sygnatariusz może ją zmodyfikować podczas procesu podpisywania.

| Właściwość | Typ | Domyślne | Opis |
|----------|------|---------|-------------|
| `prefilled_editable` | `boolean` | `false` | Gdy true, sygnatariusze mogą edytować wstępnie wypełnione wartości |

Dostępne w:
- Polach szablonu (`PATCH /templates/{id}`, `PUT /templates/{id}`)
- Polach żądania podpisu (`POST /signing-requests`, `POST /signing-requests/create-and-send`)
- Obiektach odpowiedzi pól (`GET /signing-requests/{id}`)

Przykład - utwórz żądanie podpisu z edytowalnym wstępnie wypełnionym polem:

```json
{
  "fields": [
    {
      "type": "text",
      "recipient_id": "temp_1",
      "page_number": 1,
      "x": 10,
      "y": 20,
      "width": 30,
      "height": 5,
      "format_rules": {
        "prefilledData": "full_name",
        "prefilledEditable": true
      }
    }
  ]
}
```

#### Walidacja źródła wartości pola

API weryfikuje teraz, że pola używają tylko jednego źródła wartości naraz. Żądania łączące `read_only_value`, `prefilled_data` i wyraźną `value` w tym samym polu zwrócą `400 VALIDATION_ERROR`. To wymuszenie ma zastosowanie do:

- `POST /signing-requests`
- `POST /signing-requests/create-and-send`
- `PATCH /templates/{id}` (tworzenie i aktualizacja pola)
- `PUT /templates/{id}` (upsert pola)

#### Pierwszeństwo wartości

Gdy `prefilled_editable` jest `true`, a zarówno wyraźna `value`, jak i `prefilled_data` są ustawione w polu, wyraźna wartość ma pierwszeństwo nad automatycznie wypełnionymi danymi odbiorcy.

### Przewodnik migracji

Brak zmian łamiących zgodność. Nowe funkcje są opcjonalne:
- Edycja tożsamości jest domyślnie wyłączona (`identity_editable_fields: null`). Ustaw na tablicę kluczy pól, aby włączyć.
- Istniejące wstępnie wypełnione pola danych pozostają domyślnie zablokowane (tylko do odczytu). Ustaw `prefilled_editable: true` w `format_rules`, aby uczynić je edytowalnymi.

### Nowe funkcje

#### Pobieranie dokumentu żądania podpisu

Nowy endpoint pobiera URL do pobrania dokumentu żądania podpisu. Dla ukończonych żądań podpisu URL wskazuje na końcowy podpisany PDF. Dla żądań w toku, anulowanych, odrzuconych lub wygasłych URL wskazuje na częściowy PDF snapshot uchwycający stan dokumentu w momencie ostatniej akcji sygnatariusza.

**Endpoint:** `GET /signing-requests/{id}/download`

**Pola odpowiedzi:**

| Pole | Typ | Opis |
|-------|------|-------------|
| `status` | `string` | Status żądania podpisu: `finished`, `in_progress`, `cancelled`, `declined` lub `expired` |
| `is_partial` | `boolean` | `true`, gdy dokument jest częściowym snapshotem (nie wszyscy sygnatariusze ukończyli) |
| `download_url` | `string` | Wstępnie podpisany URL do dokumentu PDF |
| `generated_at` | `date-time \| null` | Kiedy dokument został ostatnio wygenerowany |
| `expires_at` | `date-time` | Kiedy `download_url` wygasa |

**Zachowanie według statusu:**

| Stan żądania podpisu | Odpowiedź |
|-----------------------|----------|
| Jeszcze nie wysłane | `409 Conflict` — pobieranie niedostępne |
| Wysłane, generowanie w toku | `503 Service Unavailable` z nagłówkiem `Retry-After` |
| Wysłane, co najmniej jeden sygnatariusz ukończył | `200` z `is_partial: true` |
| Anulowane / Odrzucone / Wygasłe | `200` z `is_partial: true` (snapshot ostatniego znanego stanu) |
| Ukończone (wszyscy sygnatariusze ukończyli) | `200` z `is_partial: false` |

Częściowe snapshoty są generowane na żądanie i buforowane. Pierwsze żądanie może zwrócić `503` z nagłówkiem `Retry-After` podczas generowania PDF — ponów po wskazanym odstępie.

**Wygaśnięcie URL:** `download_url` jest wstępnie podpisanym URL-em. Pobierz endpoint ponownie po `expires_at`, aby uzyskać świeży URL.

#### Ustawienie znaku wodnego dla częściowego pobierania

Nowe ustawienie przestrzeni roboczej `show_partial_watermark` kontroluje, czy częściowe pobrania PDF zawierają ukośny znak wodny „IN PROGRESS — NOT YET EXECUTED". Domyślnie wyłączone. Można ustawić na poziomie firmy (domyślne dla wszystkich przestrzeni roboczych) lub nadpisać dla każdej przestrzeni roboczej.

**Endpoint:** `PUT /workspace/{workspace_id}/settings`

| Pole | Typ | Opis |
|-------|------|-------------|
| `show_partial_watermark` | `boolean \| null` | `true`, aby włączyć znak wodny, `false`, aby wyłączyć, `null`, aby dziedziczyć domyślne firmy |

### Przewodnik migracji

**Z v1.15.3 do v1.16.0:**

To jest niełamiąca zgodności addytywna zmiana. Istniejące integracje nadal działają bez modyfikacji.

- Endpoint pobierania jest czysto opcjonalny. Żadne istniejące zachowanie się nie zmienia.
- Wywołaj `GET /signing-requests/{id}/download` — obsługuj `503` z `Retry-After`, aż dokument będzie gotowy.

#### `variable_defined_name` w odpowiedziach pól

Wszystkie endpointy API, które zwracają pola szablonu lub żądania podpisu, zawierają teraz właściwość `variable_defined_name`. Jest to czytelna dla ludzi `field_name` z definicji pola niestandardowego, z którą pole jest połączone.

Wcześniej pola połączone z definicjami pól niestandardowych udostępniały tylko `variable_name`, który zawiera wewnętrzny identyfikator (np. `custom_template_a1b2c3d4-...`). Nowy `variable_defined_name` zwraca oryginalnie zdefiniowaną nazwę (np. `artist_name`), ułatwiając mapowanie pól w integracji bez odwoływania się do endpointu pól niestandardowych.

- `GET /templates/{id}/fields`
- `POST /templates/{id}/replace-document`
- `GET /signing-requests` (lista)
- `GET /signing-requests/{id}/fields`
- `GET /documents` (przestarzały)

**Przykład odpowiedzi:**
```json
{
  "variable_name": "custom_template_a1b2c3d4-...",
  "variable_defined_name": "artist_name"
}
```

Dla pól niepołączonych z definicją pola niestandardowego, `variable_defined_name` jest `null`.

#### `variable_defined_name` jako wejście do dopasowywania pól

Podczas tworzenia żądania podpisu z szablonu przez `POST /signing-requests/create-and-send` możesz teraz używać `variable_defined_name` do celowania w pola dla nadpisań zamiast wewnętrznego `variable_name`:

```json
{
  "fields": [
    { "variable_defined_name": "artist_name", "read_only_value": "John Smith" }
  ]
}
```

Zarówno `variable_name`, jak i `variable_defined_name` są akceptowane. `variable_name` ma pierwszeństwo, gdy oba są podane.

#### Logika pól warunkowych

Pola obsługują teraz warunkowe reguły dla statusu **required** i **visibility**. Dwie nowe nullable właściwości są dostępne w obiektach pól:

- `required_conditions` — Gdy ustawione, nadpisuje statyczną flagę `required`. Pole jest wymagane tylko wtedy, gdy warunki oceniają się na true na podstawie wartości innych pól.
- `visibility_conditions` — Gdy ustawione, pole jest ukryte, chyba że warunki oceniają się na true. Ukryte pola nie są walidowane przy wysyłaniu.

Warunki używają zagnieżdżonej struktury grupowej z operatorami logicznymi:

| Schemat | Opis |
|--------|-------------|
| `ConditionSet` | Kontener najwyższego poziomu z operatorem `logic` (`and`/`or`) i tablicą `groups` |
| `ConditionGroup` | Zawiera tablicę `conditions`, łączonych używając przeciwieństwa operatora logic rodzica |
| `Condition` | Ocenia pojedyncze pole przez `field_id`, `operator` i opcjonalną `value` |

**Obsługiwane operatory:** `is_filled`, `is_empty`, `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `less_than`, `greater_than_or_equal`, `less_than_or_equal`

**Przykład — Uczyń pole wymaganym tylko wtedy, gdy inne pole jest wypełnione:**
```json
{
  "required_conditions": {
    "logic": "and",
    "groups": [
      {
        "conditions": [
          {
            "field_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
            "operator": "is_filled"
          }
        ]
      }
    ]
  }
}
```

Nowe ustawienie `show_signature_frame` kontroluje, czy wizualna ramka z ID podpisu jest renderowana wokół podpisów w ukończonych PDF-ach. Dostępne na poziomach firmy, przestrzeni roboczej i żądania podpisu.

- `false` — Ukryj ramkę podpisu
- `null` — Dziedzicz z poziomu nadrzędnego

#### Wymuszanie usuwania warunków przy usuwaniu użytkownika

```json
  "field": {
    "id": "field-uuid",
    "x": 120,
    "y": 220
  }
}
```

#### Dopasowywanie użytkownika szablonu dla odbiorców

Podczas tworzenia żądań podpisu z szablonów możesz teraz używać `template_user_id`, aby wyraźnie dopasować odbiorców do użytkowników szablonu:

```json
{
  "template_id": "template-uuid",
  "recipients": [
    {
      "template_user_id": "template-user-uuid",
      "first_name": "Jane",
      "last_name": "Smith",
      "email": "jane@example.com"
    }
  ]
}
```

- `template_user_id` jest preferowaną metodą dopasowywania
- `order` pozostaje dostępny jako fallback

### Zmiany schematu

#### Enum typu pola
Zaktualizowano z:
```
["text", "signature", "date", "checkbox", "initials", "dropdown", "radio"]
```

Do:
```
["text", "signature", "date", "checkbox", "initials", "dropdown", "radio_buttons", "textarea", "url"]
```

#### Schemat Recipient
- Tworzenie żądania podpisu używa teraz odniesienia do współdzielonego schematu `Recipient`
- Dodano właściwość `template_user_id` dla wyraźnego dopasowywania użytkownika szablonu
