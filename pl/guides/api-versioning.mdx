# Przewodnik po wersjonowaniu API

API Firma używa wersjonowania opartego na nagłówkach za pośrednictwem nagłówka `X-API-Version`. Pozwala to określić, której wersji API chcesz użyć, z automatycznymi ostrzeżeniami o wycofywaniu i uporządkowanym procesem wygaszania.

## Jak określić wersję

Dołącz nagłówek `X-API-Version` do żądania:

<CodeGroup>

```bash cURL
curl https://api.firma.dev/functions/v1/signing-request-api/signing-requests \
  -H "X-API-Version: 1" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```javascript JavaScript
const response = await fetch(
  'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
  {
    headers: {
      'X-API-Version': '1',
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`
    }
  }
);
```

```python Python
import requests

headers = {
    'X-API-Version': '1',
    'Authorization': f'Bearer {os.environ["FIRMA_API_KEY"]}'
}

response = requests.get(
    'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
    headers=headers
)
```

</CodeGroup>

<Note>
Jeśli nagłówek `X-API-Version` zostanie pominięty, API domyślnie użyje aktualnie stabilnej wersji (obecnie **1**).
</Note>

## Cykl życia wersji

Każda wersja API przechodzi przez trzy etapy:

| Status | Opis | Zachowanie |
|--------|-------------|----------|
| **active** | W pełni obsługiwana wersja | Normalne odpowiedzi z nagłówkiem `X-API-Version` |
| **deprecated** | Nadal działa, ale zaplanowana do usunięcia | Odpowiedzi zawierają nagłówki ostrzegające o wycofaniu |
| **sunset** | Usunięta z usługi | Zwraca błąd `410 Gone` |

## Nagłówki odpowiedzi

### Dla aktywnych wersji

```
X-API-Version: 1
```

### Dla wersji wycofanych

Gdy wersja jest wycofywana, w każdej odpowiedzi znajdują się następujące nagłówki:

```
X-API-Version: 1
Deprecation: 2025-06-01
Sunset: 2025-12-01
X-Deprecated-Message: This API version is deprecated and will be removed on 2025-12-01. Please migrate to version 2.
```

| Nagłówek | Opis |
|--------|-------------|
| `Deprecation` | Zgodny z RFC 8594 — data wycofania wersji |
| `Sunset` | Zgodny z RFC 8594 — data usunięcia wersji |
| `X-Deprecated-Message` | Czytelna dla człowieka wiadomość ostrzegawcza |

## Odpowiedzi z błędami

### Nieprawidłowy nagłówek wersji

```json
// 400 Bad Request
{
  "error": "Invalid X-API-Version header. Must be a positive integer.",
  "code": "INVALID_VERSION"
}
```

### Nieobsługiwana wersja

```json
// 400 Bad Request
{
  "error": "API version 99 is not supported. Supported versions: 1",
  "code": "UNSUPPORTED_VERSION"
}
```

### Wersja wygaszona (usunięta)

```json
// 410 Gone
{
  "error": "API version 1 has been removed as of 2025-12-01. Please migrate to version 2.",
  "code": "VERSION_SUNSET"
}
```

## Zasady zmian łamiących kompatybilność

<Info>
Zmiany łamiące kompatybilność wprowadzane są tylko w przypadku podniesienia głównej wersji (np. v1 → v2). Zmiany niełamiące kompatybilności, takie jak nowe opcjonalne pola lub nowe endpointy, nie wymagają nowej wersji.
</Info>

Po wydaniu nowej głównej wersji:

1. **Poprzednia wersja zostaje oznaczona jako wycofana**
2. **Będziesz otrzymywać nagłówki ostrzegawcze** przez zalecany 6-miesięczny okres wycofywania
3. **Po dacie wygaszenia** stara wersja zwraca `410 Gone`

### Co stanowi zmianę łamiącą kompatybilność?

Zmiany łamiące kompatybilność obejmują:
- Usunięcie lub zmianę nazwy istniejących pól
- Zmianę typów pól lub reguł walidacji
- Usunięcie endpointów
- Zmianę wymagań uwierzytelniania
- Znaczące zmiany limitów zapytań

Zmiany niełamiące kompatybilności obejmują:
- Dodawanie nowych opcjonalnych pól
- Dodawanie nowych endpointów
- Dodawanie nowych wartości enum (jeśli Twój klient poprawnie obsługuje nieznane wartości)
- Ulepszanie komunikatów o błędach

## Najlepsze praktyki

<AccordionGroup>

<Accordion title="Zawsze sprawdzaj nagłówki wycofania">
Zaimplementuj automatyczne sprawdzanie w kodzie klienta API, aby wykrywać nagłówki `Deprecation` i `Sunset`. Loguj ostrzeżenia, gdy te nagłówki są obecne.

```javascript
const response = await fetch(url, options);

if (response.headers.get('Deprecation')) {
  console.warn('API Deprecation Warning:', {
    deprecation: response.headers.get('Deprecation'),
    sunset: response.headers.get('Sunset'),
    message: response.headers.get('X-Deprecated-Message')
  });
}
```
</Accordion>

<Accordion title="Jawnie określaj wersję">
Zamiast polegać na wersji domyślnej, zawsze jawnie określaj nagłówek `X-API-Version`. Zapobiega to nieoczekiwanym zmianom łamiącym kompatybilność, gdy zmienia się wersja domyślna.

```javascript
// Dobrze — jawna wersja
headers: { 'X-API-Version': '1' }

// Unikaj — poleganie na wartości domyślnej
headers: { }
```
</Accordion>

<Accordion title="Subskrybuj aktualizacje changeloga">
Bądź na bieżąco z nadchodzącymi zmianami, subskrybując changelog lub ogłoszenia dotyczące API. Zapewnia to wczesne powiadomienia o:
- Wydaniach nowych wersji
- Harmonogramach wycofywania
- Zmianach łamiących kompatybilność
- Przewodnikach migracji
</Accordion>

<Accordion title="Testuj nowe wersje wcześnie">
Gdy ogłoszona zostanie nowa wersja, przetestuj integrację z nią z dużym wyprzedzeniem przed datą wygaszenia starej wersji. Daje to czas na:
- Zidentyfikowanie zmian łamiących kompatybilność w implementacji
- Aktualizację kodu i zależności
- Dokładne testy w środowisku staging
- Pewne wdrożenie przed terminem
</Accordion>

</AccordionGroup>
