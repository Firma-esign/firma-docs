# Osadzany edytor żądań podpisu

Osadź edytor żądań podpisu Firma wewnątrz swojej aplikacji, używając uwierzytelniania JWT dla bezpiecznego, ograniczonego czasowo dostępu. Jest to idealne dla integracji white-label i aplikacji wielodostępowych.

## Przypadki użycia

- **Przepływy podpisywania white-label**: zarządzaj żądaniami podpisu pod własną marką
- **Aplikacje wielodostępowe**: bezpieczny dostęp per podpisujący bez ujawniania kluczy API
- **Osadzone przepływy dokumentów**: bezproblemowe zarządzanie żądaniami podpisu wewnątrz produktu
- **Dostęp ograniczony czasowo**: tokeny automatycznie wygasają dla bezpieczeństwa (7-dniowe wygaśnięcie)

## Jak to działa

1. Twój serwer żąda tokenu JWT z API Firma przy użyciu klucza API
2. Firma zwraca krótkotrwały token JWT z ID żądania podpisu
3. Twój frontend osadza edytor z tokenem JWT
4. Token automatycznie wygasa po 7 dniach

<Note>
**Limit zapytań**: endpointy JWT obsługują **100 zapytań na minutę na klucz API** dla aplikacji o dużym natężeniu.
</Note>

## Uwierzytelnianie JWT

### Wygeneruj token JWT

Wygeneruj token JWT dla konkretnego żądania podpisu przy użyciu endpointu `/jwt/generate-signing-request`.

**Endpoint**: `POST /jwt/generate-signing-request`

**Ciało żądania**:
```json
{
  "companies_workspaces_signing_requests_id": "123e4567-e89b-12d3-a456-426614174000"
}
```

**Odpowiedź (200 OK)**:
```json
{
  "jwt": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "jwt_id": "jwt123-e89b-12d3-a456-426614174000",
  "expires_at": "2024-04-27T10:00:00Z",
  "signing_request_id": "123e4567-e89b-12d3-a456-426614174000",
  "created_at": "2024-04-20T10:00:00Z"
}
```

**Nagłówki limitu zapytań**:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 99
X-RateLimit-Reset: 1729180800
```

## Przewodnik implementacji

<Warning>
**Bezpieczeństwo**: nigdy nie ujawniaj klucza API w kodzie po stronie klienta. Zawsze generuj tokeny JWT z bezpiecznego backendu.
</Warning>

### Backend: wygeneruj token JWT

Wywołaj API Firma z backendu, aby wygenerować token JWT. Twój endpoint backendu powinien przyjmować ID żądania podpisu i zwracać JWT do frontendu.

**Przykład Node.js**:
```js
// Przykład: prosta funkcja backendu do generowania JWT
async function generateSigningRequestToken(signingRequestId) {
  const response = await fetch('https://api.firma.dev/functions/v1/signing-request-api/jwt/generate-signing-request', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      companies_workspaces_signing_requests_id: signingRequestId
    })
  })
  
  const data = await response.json()
  return data.jwt
}
```

**Przykład Python**:
```py
import os
import requests

def generate_signing_request_token(signing_request_id):
    response = requests.post(
        'https://api.firma.dev/functions/v1/signing-request-api/jwt/generate-signing-request',
        headers={
            'Authorization': f'Bearer {os.getenv("FIRMA_API_KEY")}',
            'Content-Type': 'application/json'
        },
        json={
            'companies_workspaces_signing_requests_id': signing_request_id
        }
    )
    data = response.json()
    return data['jwt']
```

### Implementacja frontendu — HTML / czysty JavaScript

```html
<!-- Załaduj bibliotekę Firma Signing Request Editor -->
<script src="https://api.firma.dev/functions/v1/embed-proxy/signing-request-editor.js"></script>

<!-- Utwórz kontener dla edytora -->
<div id="signing-request-editor-container"></div>

<script>
  // Zakładając, że pobrałeś już JWT z backendu
  const jwt = 'eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...';
  const signingRequestId = '08f72e3b-79a8-4eac-a268-b3f9efaf6573';
  
  // Inicjalizuj edytor żądań podpisu
  const editor = new FirmaSigningRequestEditor({
    container: '#signing-request-editor-container',
    jwt: jwt,
    signingRequestId: signingRequestId,
    showCloseButton: true, // Pokaż przycisk zamykania w nagłówku
    onSave: (data) => {
      console.log('Signing request saved:', data);
    },
    onSend: (data) => {
      console.log('Signing request sent:', data);
    },
    onClose: (data) => {
      if (data.hasUnsavedChanges) {
        // Poproś użytkownika o zapisanie przed wyjściem
      }
      console.log('Editor closed, unsaved changes:', data.hasUnsavedChanges);
    },
    onError: (error) => {
      console.error('Editor error:', error);
    },
    onLoad: (signingRequest) => {
      console.log('Editor loaded successfully:', signingRequest);
    }
  });
</script>
```

### Implementacja frontendu — React

```jsx
import { useEffect, useRef, useState } from 'react';

function SigningRequestEditorComponent({ signingRequestId, jwt }) {
  const containerRef = useRef<HTMLDivElement>(null);
  const editorRef = useRef<any>(null);
  const [isLoaded, setIsLoaded] = useState(false);

  // Załaduj bibliotekę Firma Signing Request Editor
  useEffect(() => {
    const script = document.createElement('script');
    script.src = 'https://api.firma.dev/functions/v1/embed-proxy/signing-request-editor.js';
    script.async = true;
    
    script.onload = () => setIsLoaded(true);
    document.body.appendChild(script);
    
    return () => {
      if (document.body.contains(script)) {
        document.body.removeChild(script);
      }
    };
  }, []);

  // Inicjalizuj edytor, gdy biblioteka i JWT są gotowe
  useEffect(() => {
    if (!isLoaded || !containerRef.current || !jwt) return;

    editorRef.current = new window.FirmaSigningRequestEditor({
      container: containerRef.current,
      jwt: jwt,
      signingRequestId: signingRequestId,
      showCloseButton: true, // Pokaż przycisk zamykania w nagłówku
      onSave: (data) => console.log('Saved:', data),
      onSend: (data) => console.log('Sent:', data),
      onClose: (data) => {
        if (data.hasUnsavedChanges) {
          // Poproś użytkownika o zapisanie przed wyjściem
        }
      },
      onError: (error) => console.error('Error:', error),
      onLoad: (signingRequest) => console.log('Loaded:', signingRequest)
    });

    return () => {
      if (editorRef.current?.destroy) {
        editorRef.current.destroy();
      }
    };
  }, [isLoaded, jwt, signingRequestId]);

  return <div ref={containerRef} className="w-full h-full" />;
}

export default SigningRequestEditorComponent;
```

### Opcje konfiguracji

| Opcja | Typ | Opis |
|--------|------|-------------|
| `container` | HTMLElement | Element DOM do zamontowania edytora (wymagane) |
| `jwt` | string | Token uwierzytelniający z backendu (wymagane) |
| `signingRequestId` | string | Identyfikator żądania podpisu (wymagane) |
| `theme` | `'dark'` \| `'light'` | Motyw edytora (domyślnie: `'dark'`) |
| `readOnly` | boolean | Włącz tryb tylko do odczytu (domyślnie: `false`) |
| `height` | string | Wysokość kontenera (domyślnie: `'100vh'`) |
| `width` | string | Szerokość kontenera (domyślnie: `'100%'`) |
| `showCloseButton` | boolean | Pokaż przycisk zamykania w nagłówku edytora (domyślnie: `false`) |
| `onSave` | function | Callback wywoływany po zapisaniu żądania podpisu |
| `onSend` | function | Callback wywoływany po wysłaniu żądania podpisu |
| `onClose` | function | Callback wywoływany po kliknięciu przycisku zamykania lub wywołaniu `triggerClose()`. Otrzymuje `{ hasUnsavedChanges: boolean }` |
| `onError` | function | Callback do obsługi błędów |
| `onLoad` | function | Callback wywoływany po załadowaniu edytora z danymi żądania podpisu |

### Metody instancji

| Metoda | Opis |
|--------|-------------|
| `triggerClose()` | Programistycznie uruchamia przepływ zamykania. Sprawdza niezapisane zmiany i wywołuje `onClose` z `{ hasUnsavedChanges: boolean }` |

Użyj `triggerClose()`, gdy aplikacja hosta musi zamknąć edytor z własnego UI — na przykład zdarzenie nawigacji rodzica lub przycisk „wstecz":

```js
// Zamknij edytor z UI aplikacji
document.getElementById('my-back-button').addEventListener('click', () => {
  editor.triggerClose();
});
```

## Zdarzenia postMessage (edytor → host)

Edytor żądań podpisu Firma emituje zdarzenia postMessage dla ważnych akcji cyklu życia. Poniżej znajduje się zalecany, minimalny schemat zdarzeń, który możesz zaimplementować, aby reagować na zapisy i wysyłki edytora.

<Tip>
Użyj tych zdarzeń do śledzenia aktywności edytora bez wykonywania dodatkowych wywołań API, pomagając Ci pozostać w limitach zapytań.
</Tip>

Koperta zdarzenia (payload window.postMessage):

```json
{
	"type": "editor.event",
	"event": "signing_request.saved", // lub signing_request.sent, signing_request.closed
	"payload": {
		"signing_request_id": "sr_123",
		"updated_at": "2025-09-04T12:34:56Z",
		"status": "draft" // lub "sent"
	}
}
```

### Przykład listenera klienta (czysty JS)

```js
window.addEventListener('message', (ev) => {
	// 1) Waliduj origin
	if (ev.origin !== 'https://app.firma.dev') return
	// 2) Waliduj kształt
	const data = ev.data || {}
	if (data.type !== 'editor.event') return

	switch (data.event) {
		case 'signing_request.saved':
			console.log('Signing request saved', data.payload)
			// Opcjonalnie pobierz najnowsze żądanie podpisu przez API
			break
		case 'signing_request.sent':
			console.log('Signing request sent', data.payload)
			// Uruchom webhook lub powiadomienie
			break
		case 'signing_request.closed':
			console.log('Editor closed')
			break
		default:
			break
	}
})
```

## Zarządzanie cyklem życia tokenu

Tokeny JWT są generowane z 7-dniowym czasem wygaśnięcia dla typowych przepływów edycji. Edytor automatycznie obsługuje wygaśnięcie tokenu.

### Automatyczne wygaśnięcie

Tokeny JWT wygasają automatycznie po 7 dniach na podstawie znacznika czasu `expires_at`. Po wygaśnięciu:
- Osadzony edytor odrzuci token
- Aby kontynuować, należy zażądać nowego tokenu
- Nie jest potrzebne żadne wywołanie API — tokeny wygasają biernie

### Odświeżanie tokenu (opcjonalne)

Dla długotrwałych sesji edycji możesz odświeżyć token przed wygaśnięciem, używając metody `updateJWT()`:

```js
// Wygeneruj nowy JWT z backendu
const newJwt = await fetch('/your-backend/generate-token').then(r => r.json());

// Zaktualizuj edytor nowym tokenem
editor.updateJWT(newJwt.jwt);
```

## Limity zapytań

Endpointy JWT obsługują **100 zapytań na minutę na klucz API**.

**Nagłówki limitu zapytań**:
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1729180800
```

Jeśli przekroczysz limit zapytań (odpowiedź 429):
- Buforuj tokeny i używaj ich ponownie do wygaśnięcia (7 dni)
- Implementuj wykładniczy backoff dla ponowień
- Monitoruj nagłówek `X-RateLimit-Remaining`
- Generuj tokeny na żądanie, nie z wyprzedzeniem

## Najlepsze praktyki bezpieczeństwa

<Warning>
**Nigdy nie ujawniaj klucza API w kodzie po stronie klienta.** Zawsze generuj tokeny JWT z bezpiecznego endpointu serwera.
</Warning>

### ✅ Rób

- ✅ Generuj tokeny z serwera backendu
- ✅ Monitoruj limity zapytań (100 zapytań/minutę)
- ✅ Używaj HTTPS dla wszystkich żądań API
- ✅ Używaj `readOnly: true` dla dostępu tylko do wyświetlania

### ❌ Nie rób

- ❌ Nie ujawniaj kluczy API w kodzie frontendu
- ❌ Nie używaj ponownie tokenów między sesjami lub podpisującymi
- ❌ Nie loguj tokenów JWT (ryzyko bezpieczeństwa)
- ❌ Nie współdziel tokenów między różnymi żądaniami podpisu

## Rozwiązywanie problemów

### Błąd wygaśniętego tokenu

**Objaw**: edytor pokazuje „Token expired" lub błąd uwierzytelniania

**Rozwiązanie**: 
- Implementuj odświeżanie tokenu przed wygaśnięciem (7-dniowe okno)
- Wygeneruj nowy token i wywołaj `editor.updateJWT(newToken)`
- Sprawdź synchronizację zegara systemowego

### 401 Unauthorized

**Objaw**: generowanie JWT kończy się błędem 401

**Możliwe przyczyny**:
- Nieprawidłowy lub brakujący klucz API
- Klucz API nie ma wymaganych uprawnień
- Klucz API jest wyłączony

**Rozwiązanie**: zweryfikuj klucz API w panelu i sprawdź uprawnienia

### 404 Not Found

**Objaw**: żądanie podpisu nie znaleziono podczas generowania JWT

**Możliwe przyczyny**:
- ID żądania podpisu nie istnieje
- Żądanie podpisu należy do innego workspace'a
- Żądanie podpisu zostało usunięte

**Rozwiązanie**: zweryfikuj ID żądania podpisu i dostęp do workspace'a

### Przekroczenie limitu zapytań

**Objaw**: 429 Too Many Requests

**Rozwiązanie**:
- Implementuj buforowanie tokenów (7-dniowe okno wygaśnięcia jest hojne)
- Poczekaj na reset limitu zapytań (sprawdź nagłówek `X-RateLimit-Reset`)
- Implementuj logikę ponowień z wykładniczym backoffem

### Edytor nie ładuje się

**Objaw**: kontener jest pusty lub w nieskończoność pokazuje spinner ładowania

**Możliwe przyczyny**:
- Skrypt nie załadowany (sprawdź zdarzenie `script.onload`)
- Nieprawidłowy JWT lub ID żądania podpisu
- Problemy z CORS (sprawdź konsolę przeglądarki)

**Rozwiązanie**:
- Zweryfikuj, że skrypt jest ładowany z `https://app.firma.dev/embed/signing-request-editor.js`
- Sprawdź, czy JWT jest prawidłowy i nie wygasł
- Upewnij się, że backend zwraca poprawne nagłówki CORS
