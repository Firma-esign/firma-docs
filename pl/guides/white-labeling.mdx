Firma zapewnia kompleksowe możliwości white-labeling, które pozwalają stworzyć w pełni brandowane doświadczenie e-podpisów. Od niestandardowych logo i motywów kolorów po osadzone interfejsy — możesz zapewnić, że Twoi klienci wchodzą w interakcję z Twoją marką przez cały proces podpisywania.

## Przegląd

White labeling w Firmie obejmuje kilka komponentów:

1. **Niestandardowy branding** - Wgrywaj loga, ustawiaj motywy kolorystyczne i całkowicie ukrywaj branding Firmy
2. **Niestandardowe warunki sygnatariusza** - Wymagaj od sygnatariuszy akceptacji własnych warunków usług
3. **Etykiety przycisków podpisywania** - Dostosuj tekst na przyciskach podpisywania dla każdego języka
4. **Niestandardowe domeny e-mail** - Wysyłaj e-maile z żądaniami podpisu z własnej domeny
5. **Niestandardowy adres nadawcy e-maili** - Kontroluj adres „from" we wszystkich wychodzących e-mailach
6. **Niestandardowe szablony e-maili** - Personalizuj treść i branding e-maili z powiadomieniami o podpisywaniu
7. **Wyłączenie e-maili Firmy** - Wyłącz automatyczne e-maile dla każdego żądania podpisu i wysyłaj własne
8. **Osadzone doświadczenia** - Osadzaj edytory podpisywania i szablonów bezpośrednio w swojej aplikacji

<Note>
  Wszystkie przykłady API używają klucza API bezpośrednio w nagłówku `Authorization`. Nie używaj prefiksu `Bearer`.
</Note>

## Niestandardowy branding

Dostosuj loga, kolory i widoczność brandingu na poziomie firmy i przestrzeni roboczej. Ustawienia przestrzeni roboczej nadpisują ustawienia firmy, umożliwiając tworzenie odrębnych brandowanych doświadczeń dla każdego z klientów.

### Logo firmy

Wgraj logo dla całego konta Firmy. To logo pojawia się w e-mailach podpisywania i w doświadczeniu podpisywania.

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/logo.png"
```

**Wymagania:**
- Format: tylko PNG lub JPEG
- Maksymalny rozmiar: 2 MB

Aby usunąć logo firmy:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Logo przestrzeni roboczej

Nadpisz logo firmy dla konkretnej przestrzeni roboczej:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY" \
  -F "file=@/path/to/workspace-logo.png"
```

Aby usunąć logo przestrzeni roboczej (powraca do logo firmy):

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspaces/{workspace_id}/logo \
  -H "Authorization: YOUR_API_KEY"
```

### Motywy kolorystyczne

Dostosuj paletę kolorów doświadczenia podpisywania, używając wartości kolorów hex. Kolory można ustawić zarówno na poziomie firmy, jak i przestrzeni roboczej.

| Ustawienie koloru | Opis | Domyślne Firmy |
|-------------------|------|----------------|
| `color_primary` | Główny kolor akcentu (przyciski, linki) | `#c285ff` |
| `color_primary_fg` | Kolor tekstu na elementach głównych | `#ffffff` |
| `color_background` | Tło strony | `#1c1c21` |
| `color_foreground` | Główny kolor tekstu | `#ffffff` |
| `color_card` | Tło karty/panelu | `#22222a` |
| `color_border` | Kolor obramowania | `#3b3b3b` |
| `color_accent` | Drugorzędny kolor akcentu | `#c285ff` |
| `color_accent_fg` | Kolor tekstu na elementach akcentu | `#ffffff` |
| `color_canvas` | Tło płótna/podglądu dokumentu | `#1c1c21` |
| `color_muted` | Wyciszone/subtelne obszary tła | `#22222a` |
| `color_muted_fg` | Kolor tekstu na obszarach wyciszonych | `#a1a1aa` |

**Ustaw kolory na poziomie firmy:**

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#0066cc",
    "color_primary_fg": "#ffffff",
    "color_background": "#f5f5f5",
    "color_foreground": "#1a1a1a",
    "color_card": "#ffffff",
    "color_border": "#e0e0e0"
  }'
```

**Ustaw kolory na poziomie przestrzeni roboczej** (nadpisuje kolory firmy dla tej przestrzeni roboczej):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "color_primary": "#ff6600",
    "color_primary_fg": "#ffffff"
  }'
```

Kolory muszą być 6-cyfrowymi wartościami hex (np. `#0066cc`). Ustaw kolor na `null`, aby go wyczyścić i wrócić do następnego poziomu w hierarchii.

### Ukryj branding Firmy

Usuń cały branding Firmy z doświadczenia podpisywania, włączając `show_custom_branding_only` na poziomie firmy:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "show_custom_branding_only": true
  }'
```

### Dodatkowe ustawienia wyświetlania

Dopracuj doświadczenie podpisywania tymi ustawieniami, dostępnymi na poziomie firmy i przestrzeni roboczej:

| Ustawienie | Opis | Domyślne |
|------------|------|----------|
| `show_signature_frame` | Pokaż ramkę/obramowanie rysowania podpisu | `true` |
| `show_partial_watermark` | Pokaż znak wodny na częściowo podpisanych dokumentach | `false` |
| `show_qr_code` | Dołącz kod QR linkujący do strony podpisywania w e-mailach | `false` |
| `require_otp_verification` | Wymagaj weryfikacji OTP e-maila przed podpisaniem | `false` |
| `require_terms_acceptance` | Wymagaj od sygnatariuszy akceptacji warunków przed podpisaniem | `true` |
| `disable_guided_navigation` | Wyłącz nawigację polami krok po kroku | `false` |
| `allow_presigning_download` | Pozwól sygnatariuszom pobrać dokument przed podpisaniem | `false` |

Na poziomie przestrzeni roboczej ustaw je na `null`, aby dziedziczyć ustawienie na poziomie firmy.

### Hierarchia ustawień

Ustawienia brandingu podążają za kaskadową hierarchią:

1. **Ustawienie przestrzeni roboczej** (najwyższy priorytet)
2. **Ustawienie firmy**
3. **Domyślne Firmy** (najniższy priorytet)

Pozwala to ustawić domyślne wartości dla całej firmy i nadpisać je dla każdej przestrzeni roboczej. Na poziomie przestrzeni roboczej ustawienie wartości na `null` oznacza „dziedzicz z firmy".

Zobacz [referencję API ustawień przestrzeni roboczej](/api-reference/v01.27.00/workspace-settings/get-workspace-settings), aby uzyskać pełną listę konfigurowalnych ustawień.

## Niestandardowe warunki sygnatariusza

Wymagaj od sygnatariuszy zaakceptowania Twoich własnych warunków usług lub umowy prawnej przed podpisaniem. Warunki są konfigurowane dla każdego języka i podążają za tą samą kaskadą firma/przestrzeń robocza co inne ustawienia.

### Jak to działa

Gdy `require_terms_acceptance` jest włączone (domyślnie), sygnatariusze widzą pole wyboru akceptacji warunków przed możliwością podpisania. Możesz dostosować tekst oświadczenia i opcjonalnie połączyć z pełnym dokumentem warunków.

Warunki są przechowywane dla każdego języka, więc sygnatariusze widzą warunki w języku swojej przestrzeni roboczej, wracając przez kaskadę:

1. **Warunki przestrzeni roboczej** dla języka sygnatariusza (najwyższy priorytet)
2. **Warunki firmy** dla języka sygnatariusza
3. **Domyślne warunki Firmy** (najniższy priorytet)

### Ustaw warunki na poziomie firmy

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acme.com/terms\">Terms of Service</a> and acknowledge that my electronic signature is legally binding.",
    "terms_url": "https://acme.com/terms"
  }'
```

### Ustaw warunki na poziomie przestrzeni roboczej

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "statement_text": "I agree to the <a href=\"https://acmecorp.com/signing-terms\">Signing Terms</a>.",
    "terms_url": "https://acmecorp.com/signing-terms"
  }'
```

**Pola:**
- `statement_text` (wymagane): Tekst HTML pokazany obok pola wyboru akceptacji. Podstawowe tagi HTML są dozwolone (`<a>`, `<b>`, `<i>`, `<br>`).
- `terms_url` (opcjonalne): URL do pełnego dokumentu warunków.

### Lista warunków

Pobierz wszystkie skonfigurowane warunki dla firmy lub przestrzeni roboczej:

```bash
# Company terms
curl https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms \
  -H "Authorization: YOUR_API_KEY"

# Workspace terms
curl https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms \
  -H "Authorization: YOUR_API_KEY"
```

### Usuń warunki

Usuń niestandardowe warunki dla konkretnego języka, aby wrócić do następnego poziomu w kaskadzie:

```bash
# Delete company terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/company/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"

# Delete workspace terms for English
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/signer-terms/en \
  -H "Authorization: YOUR_API_KEY"
```

### Wyłącz akceptację warunków

Aby całkowicie wyłączyć wymóg akceptacji warunków:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "require_terms_acceptance": false
  }'
```

Obsługiwane języki: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

Zobacz [referencję API Signer Terms](/api-reference/v01.27.00/signer-terms/list-company-signer-terms), aby uzyskać kompletną dokumentację endpointów.

## Niestandardowe etykiety przycisków podpisywania

Dostosuj tekst na przyciskach podpisywania dla każdej przestrzeni roboczej i języka. Pozwala to dopasować etykiety przycisków do terminologii lub lokalizacji Twojego produktu.

### Ustaw nadpisania etykiet przycisków

Etykiety przycisków są ustawiane jako obiekt JSONB w ustawieniach przestrzeni roboczej, kluczowany według języka, a następnie według klucza przycisku:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": {
      "en": {
        "sign": "Approve and Sign",
        "next": "Continue",
        "finish": "Complete Signing"
      },
      "es": {
        "sign": "Aprobar y Firmar",
        "next": "Continuar",
        "finish": "Completar Firma"
      }
    }
  }'
```

### Wyczyść nadpisania etykiet przycisków

Ustaw na `null`, aby przywrócić domyślne etykiety przycisków Firmy:

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "signing_button_label_overrides": null
  }'
```

**Walidacja:**
- Każda etykieta musi być łańcuchem znaków, maks. 500 znaków
- Klucze języków muszą być prawidłowymi kodami języków (maks. 10 znaków)
- Tagi HTML i znaki kontrolne są automatycznie usuwane

<Note>
  Nadpisania etykiet przycisków to ustawienie tylko na poziomie przestrzeni roboczej. Nie ma nadpisania na poziomie firmy.
</Note>

## Niestandardowe domeny e-mail

Domyślnie e-maile z żądaniami podpisu są wysyłane z domeny Firmy. Dzięki niestandardowym domenom e-mail e-maile wydają się pochodzić bezpośrednio z Twojej firmy lub firm Twoich klientów.

### Domeny e-mail na poziomie konta (firmy)

Skonfiguruj niestandardową domenę e-mail dla całego konta Firmy. Wszystkie przestrzenie robocze domyślnie będą używać tej domeny, chyba że zostanie nadpisana.

#### Krok 1: dodaj swoją domenę

Użyj API, aby dodać niestandardową domenę e-mail:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "acme.com"
  }'
```

**Odpowiedź (201 Created)**:

```json
{
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 0,
    "domain_status": 0,
    "is_primary": false,
    "verification_token": "firma-verify=abc123xyz",
    "date_created": "2024-01-15T10:30:00Z"
  },
  "verification_instructions": {
    "record_type": "TXT",
    "record_name": "_firma-verification.acme.com",
    "record_value": "firma-verify=abc123xyz",
    "next_step": "Add this TXT record to your DNS, then call POST /company/domains/{id}/verify-ownership"
  }
}
```

#### Krok 2: dodaj rekord weryfikacyjny TXT

Dodaj rekord weryfikacyjny TXT do swojego DNS:

| Typ | Nazwa                          | Wartość                  |
| --- | ------------------------------ | ------------------------ |
| TXT | `_firma-verification.acme.com` | `firma-verify=abc123xyz` |

<Note>
  Propagacja DNS zwykle zajmuje kilka minut, ale może potrwać do 48 godzin.
</Note>

#### Krok 3: zweryfikuj własność domeny

Po dodaniu rekordu TXT zweryfikuj własność:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-ownership \
  -H "Authorization: YOUR_API_KEY"
```

**Odpowiedź**:

```json
{
  "message": "Domain ownership verified",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 1,
    "domain_status": 0
  },
  "next_step": "Call POST /company/domains/{id}/finalize to complete domain setup and receive DNS records for email sending"
}
```

#### Krok 4: sfinalizuj konfigurację domeny

Po weryfikacji własności sfinalizuj domenę, aby otrzymać rekordy DNS do wysyłania e-maili:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/finalize \
  -H "Authorization: YOUR_API_KEY"
```

**Odpowiedź**:

```json
{
  "message": "Domain finalized. Add the following DNS records to enable email sending.",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 0
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "ttl": "Auto", "status": "pending" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "ttl": "Auto", "status": "pending" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "ttl": "Auto", "status": "pending" }
  ],
  "next_step": "Add these DNS records, then call POST /company/domains/{id}/verify-dns to complete verification"
}
```

#### Krok 5: dodaj rekordy DNS

Dodaj wszystkie trzy rekordy DNS do swojej domeny:

| Typ   | Nazwa               | Wartość                             |
| ----- | ------------------- | ----------------------------------- |
| TXT   | `@`                 | `v=spf1 include:amazonses.com ~all` |
| CNAME | `resend._domainkey` | `resend._domainkey.amazonses.com`   |
| TXT   | `_dmarc`            | `v=DMARC1; p=none;`                 |

#### Krok 6: zweryfikuj rekordy DNS

Po dodaniu rekordów DNS zweryfikuj je:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/verify-dns \
  -H "Authorization: YOUR_API_KEY"
```

**Odpowiedź (wszystko zweryfikowane)**:

```json
{
  "verified": true,
  "message": "Domain is fully verified and ready to send emails",
  "domain": {
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "domain": "acme.com",
    "verification_status": 2,
    "domain_status": 1,
    "is_primary": true
  },
  "dns_records": [
    { "type": "TXT", "name": "@", "value": "v=spf1 include:amazonses.com ~all", "status": "verified" },
    { "type": "CNAME", "name": "resend._domainkey", "value": "resend._domainkey.amazonses.com", "status": "verified" },
    { "type": "TXT", "name": "_dmarc", "value": "v=DMARC1; p=none;", "status": "verified" }
  ]
}
```

#### Krok 7: ustaw jako główną domenę (opcjonalne)

Jeśli masz wiele domen, ustaw jedną jako domyślną:

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/company/domains/{domain_id}/set-primary \
  -H "Authorization: YOUR_API_KEY"
```

### Domeny e-mail na poziomie przestrzeni roboczej

Dla aplikacji SaaS wielotenantowych możesz skonfigurować różne domeny e-mail dla każdej przestrzeni roboczej. Domeny przestrzeni roboczej nadpisują domenę na poziomie firmy.

Przepływ pracy jest identyczny jak dla domen firmy, ale używa endpointów o zakresie przestrzeni roboczej:

| Akcja            | Endpoint                                                       |
| ---------------- | -------------------------------------------------------------- |
| Lista domen      | `GET /workspace/{workspace_id}/domains`                        |
| Dodaj domenę     | `POST /workspace/{workspace_id}/domains`                       |
| Pobierz domenę   | `GET /workspace/{workspace_id}/domains/{id}`                   |
| Usuń domenę      | `DELETE /workspace/{workspace_id}/domains/{id}`                |
| Zweryfikuj własność | `POST /workspace/{workspace_id}/domains/{id}/verify-ownership` |
| Sfinalizuj konfigurację | `POST /workspace/{workspace_id}/domains/{id}/finalize`         |
| Zweryfikuj DNS   | `POST /workspace/{workspace_id}/domains/{id}/verify-dns`       |
| Ustaw jako główną | `POST /workspace/{workspace_id}/domains/{id}/set-primary`      |

**Przykład: dodaj domenę przestrzeni roboczej**

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/domains \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "sign.acmecorp.com"
  }'
```

Zobacz [referencję API domen e-mail](/api-reference/v01.27.00/email-domains/list-company-domains), aby uzyskać kompletną dokumentację endpointów.

## Niestandardowy adres nadawcy e-maili

Kontroluj adres „from" we wszystkich wychodzących e-mailach, łącząc niestandardową domenę z niestandardową częścią lokalną.

### Jak są rozwiązywane adresy e-mail

Gdy Firma wysyła e-mail, adres nadawcy jest budowany z trzech komponentów:

```
{sender_name} <{local_part}@{domain}>
```

Każdy komponent jest rozwiązywany przez łańcuch awaryjny:

**Rozwiązywanie domeny:**
1. Zweryfikowana domena przestrzeni roboczej (preferowana główna)
2. Zweryfikowana domena firmy
3. `updates.firma.dev` (domyślna)

**Rozwiązywanie części lokalnej** (dotyczy tylko podczas używania niestandardowej domeny):
1. Ustawienie `email_local_part` przestrzeni roboczej
2. Ustawienie `email_local_part` firmy
3. `support` (domyślne)

**Nazwa nadawcy** to nazwa użytkownika, który wysłał żądanie podpisu.

Na przykład, jeśli ustawisz `email_local_part` na `noreply`, a Twoja zweryfikowana domena to `sign.acmecorp.com`, e-maile będą wysyłane z:

```
Jane Smith <noreply@sign.acmecorp.com>
```

### Ustawianie części lokalnej e-maila

**Poziom firmy** (domyślny dla wszystkich przestrzeni roboczych):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "noreply"
  }'
```

**Poziom przestrzeni roboczej** (nadpisuje ustawienie firmy):

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/settings \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email_local_part": "signing"
  }'
```

**Reguły walidacji:**
- 1-64 znaków, małe litery, cyfry, kropki, podkreślenia i myślniki
- Musi zaczynać się i kończyć literą lub cyfrą
- Bez kolejnych kropek
- Wartości zastrzeżone (`postmaster`, `abuse`, `mailer-daemon`) nie są dozwolone

Ustaw na `null` na poziomie przestrzeni roboczej, aby dziedziczyć z ustawienia firmy.

## Niestandardowe szablony e-maili

Dostosuj e-maile z powiadomieniami, które Firma wysyła w Twoim imieniu, aby pasowały do głosu i stylu Twojej marki. Możesz ustawić szablony zarówno na poziomie firmy, jak i przestrzeni roboczej, przy czym szablony przestrzeni roboczej nadpisują domyślne szablony firmy.

### Konfigurowalne typy e-maili

| Typ e-maila | Kiedy wysyłany |
|-------------|----------------|
| `signing_invite` | Gdy żądanie podpisu jest wysyłane do odbiorcy |
| `next_signer` | Gdy nadchodzi kolej następnego sygnatariusza w kolejności podpisywania |
| `resend_notification` | Gdy nadawca ręcznie wysyła ponownie zaproszenie do podpisu |
| `reminder_default` | Gdy wysyłane jest automatyczne przypomnienie o oczekującym podpisie |
| `signing_expired` | Gdy żądanie podpisu wygasa |
| `signing_cancelled` | Gdy żądanie podpisu jest anulowane |
| `signing_declined` | Gdy sygnatariusz odmawia (wysyłane do innych sygnatariuszy) |
| `signing_declined_admin` | Gdy sygnatariusz odmawia (wysyłane do nadawcy, zawiera powód odmowy) |
| `signing_completed` | Gdy wszyscy sygnatariusze ukończą podpisywanie |
| `signer_identity_changed` | Gdy sygnatariusz zmienia swoje imię podczas podpisywania |

### Dostępne symbole zastępcze

Szablony obsługują ciała HTML z dynamicznymi symbolami zastępczymi przy użyciu składni `{{placeholder}}`.

**Symbole zastępcze sygnatariusza:**

| Symbol zastępczy | Opis |
|------------------|------|
| `{{signer_first_name}}` | Imię sygnatariusza |
| `{{signer_last_name}}` | Nazwisko sygnatariusza |
| `{{signer_name}}` | Pełne imię sygnatariusza |
| `{{signer_email}}` | Adres e-mail sygnatariusza |
| `{{signer_title}}` | Tytuł zawodowy sygnatariusza |
| `{{signer_company}}` | Nazwa firmy sygnatariusza |

**Symbole zastępcze dokumentu:**

| Symbol zastępczy | Opis |
|------------------|------|
| `{{signing_request_name}}` | Nazwa żądania podpisu |
| `{{signing_link}}` | URL do podpisu dla sygnatariusza |
| `{{expiration_date}}` | Kiedy żądanie podpisu wygasa |
| `{{download_link}}` | Link do pobrania podpisanego dokumentu |
| `{{decliner_name}}` | Nazwa sygnatariusza, który odmówił |
| `{{decline_reason}}` | Powód podany podczas odmowy |
| `{{signing_qr_code}}` | Obraz kodu QR linkującego do strony podpisywania |

**Symbole zastępcze zespołu:**

| Symbol zastępczy | Opis |
|------------------|------|
| `{{team_name}}` | Nazwa przestrzeni roboczej/zespołu |
| `{{team_email}}` | Kontaktowy e-mail przestrzeni roboczej |
| `{{company_name}}` | Nazwa firmy |
| `{{company_logo}}` | Obraz logo firmy |

Możesz również pobrać tę listę programowo:

```bash
curl https://api.firma.dev/functions/v1/signing-request-api/email-templates/placeholders \
  -H "Authorization: YOUR_API_KEY"
```

### Wyświetl domyślne szablony

Pobierz wbudowane domyślne szablony Firmy dla dowolnego obsługiwanego języka, aby użyć jako punktu wyjścia:

```bash

  -H "Authorization: YOUR_API_KEY"
```

Obsługiwane języki: `en`, `es`, `it`, `pt`, `fr`, `de`, `el`, `ru`, `pl`.

### Ustaw szablon e-maila przestrzeni roboczej

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "subject": "Action required: Please sign {{signing_request_name}}",
    "body": "<p>Hi {{signer_name}},</p><p>Please review and sign {{signing_request_name}}.</p><p><a href=\"{{signing_link}}\">Sign now</a></p><p>- The {{team_name}} Team</p>"
  }'
```

**Walidacja:**
- `subject`: wymagane, maks. 500 znaków
- `body`: wymagane, maks. 50 000 znaków

### Ustaw szablon e-maila firmy

```bash
curl -X PUT https://api.firma.dev/functions/v1/signing-request-api/company/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY" \
```

### Usuń niestandardowy szablon

Usuń niestandardowy szablon, aby wrócić do następnego poziomu w hierarchii:

```bash
curl -X DELETE https://api.firma.dev/functions/v1/signing-request-api/workspace/{workspace_id}/email-templates/signing_invite \
  -H "Authorization: YOUR_API_KEY"
```

### Hierarchia szablonów

Szablony e-maili podążają za kaskadową hierarchią:

1. **Szablon przestrzeni roboczej** (najwyższy priorytet)
2. **Szablon firmy**
3. **Wbudowany domyślny** dla języka przestrzeni roboczej (najniższy priorytet)

Pozwala to ustawić brandowany szablon dla całej firmy i nadpisać go dla konkretnych przestrzeni roboczych, gdy to potrzebne. Usunięcie szablonu na dowolnym poziomie powoduje powrót do następnego poziomu.

<Warning>
  Zwracane jest ostrzeżenie, jeśli ciało szablonu nie zawiera symbolu zastępczego `{{signing_link}}`, ponieważ odbiorcy potrzebują linku, aby uzyskać dostęp do przepływu podpisywania.
</Warning>

Zobacz [referencję API szablonów e-maili](/api-reference/v01.27.00/email-templates/list-workspace-email-templates), aby uzyskać kompletną dokumentację endpointów.

## Wyłączanie e-maili Firmy

Aby uzyskać pełną kontrolę nad komunikacją z klientami, możesz wyłączyć automatyczne e-maile Firmy dla każdego żądania podpisu. Pozwala to:

- Wysyłać linki żądań podpisu przez własny system e-mail
- Integrować z istniejącymi przepływami powiadomień
- Dostosowywać czas e-maili i sekwencje follow-up
- Używać własnej infrastruktury dostarczania e-maili

### Ustawienia e-maili w żądaniach podpisu

Każde żądanie podpisu ma ustawienia kontrolujące, które e-maile są wysyłane:

| Ustawienie                | Opis                                                     | Domyślne |
| ------------------------- | -------------------------------------------------------- | -------- |
| `send_signing_email`      | Wysyłaj e-maile z powiadomieniami o żądaniach podpisu do sygnatariuszy | `true`   |
| `send_finish_email`       | Wysyłaj e-mail z ukończeniem, gdy wszyscy sygnatariusze ukończą | `true`   |
| `send_expiration_email`   | Wysyłaj powiadomienie o wygaśnięciu, gdy żądanie wygasa  | `true`   |
| `send_cancellation_email` | Wysyłaj powiadomienie o anulowaniu, gdy żądanie jest anulowane | `true`   |

### Utwórz żądanie podpisu z wyłączonymi e-mailami

```bash
curl -X POST https://api.firma.dev/functions/v1/signing-request-api/signing-requests \
  -H "Authorization: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace_id": "workspace_123",
    "template_id": "template_456",
    "settings": {
      "send_signing_email": false,
      "send_finish_email": false,
      "send_expiration_email": false,
      "send_cancellation_email": false
    },
    "recipients": [
      {
        "first_name": "Jane",
        "last_name": "Smith",
        "email": "jane@example.com",
        "designation": "Signer",
        "order": 1
      }
    ]
  }'
```

### Pobierz URL-e podpisywania do ręcznej dystrybucji

Gdy e-maile są wyłączone, pobierz URL-e podpisywania z API i wyślij je przez własne kanały:

```js
// Create signing request with emails disabled
const response = await fetch(
  'https://api.firma.dev/functions/v1/signing-request-api/signing-requests',
  {
    method: 'POST',
    headers: {
      'Authorization': API_KEY,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      workspace_id: workspaceId,
      template_id: templateId,
      settings: {
        send_signing_email: false,
        send_finish_email: false
      },
      recipients: [
        { first_name: 'Jane', last_name: 'Smith', email: 'jane@example.com', designation: 'Signer', order: 1 }
      ]
    })
  }
)
const signingRequest = await response.json()

// Get signing URLs for each recipient
const usersResponse = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequest.id}/users`,
  {
    headers: { 'Authorization': API_KEY }
  }
)
const { results: users } = await usersResponse.json()

// Send emails through your own system
for (const user of users) {
  const signingUrl = `https://app.firma.dev/signing/${user.id}`
  await yourEmailService.send({
    to: user.email,
    subject: 'Please sign your document',
    body: `Click here to sign: ${signingUrl}`
  })
}
```

## Najlepsze praktyki

### Branding

- Używaj spójnej palety kolorów w logach, szablonach e-maili i osadzonych doświadczeniach
- Wgrywaj loga w formacie PNG z przezroczystym tłem dla najlepszych rezultatów
- Włącz `show_custom_branding_only`, aby całkowicie usunąć branding Firmy
- Ustaw branding na poziomie firmy jako punkt odniesienia, następnie nadpisz dla każdej przestrzeni roboczej w razie potrzeby
- Przetestuj pełny przepływ podpisywania z perspektywy swoich klientów

### Konfiguracja domeny e-mail

- Używaj subdomeny (np. `sign.yourcompany.com`) zamiast głównej domeny
- Skonfiguruj wszystkie rekordy DNS (SPF, DKIM, DMARC) dla optymalnej dostarczalności
- Monitoruj wskaźniki odrzuceń e-maili i dostosowuj w razie potrzeby
- Przetestuj dostarczanie e-maili przed wejściem na produkcję z klientami

### Szablony e-maili

- Użyj [endpointu domyślnych](#view-default-templates), aby zobaczyć wbudowane szablony Firmy jako punkt wyjścia
- Zawsze uwzględniaj `{{signing_link}}` w szablonach zaproszeń i przypomnień
- Ustaw szablon na poziomie firmy jako brandowany punkt odniesienia, następnie nadpisz dla każdej przestrzeni roboczej w razie potrzeby
- Używaj `{{company_logo}}` w szablonach, aby dołączyć logo przestrzeni roboczej lub firmy

### Bezpieczeństwo

- Zawsze generuj tokeny JWT na swoim backendzie
- Nigdy nie ujawniaj kluczy API w kodzie po stronie klienta
- Używaj krótkich czasów wygasania tokenów (zalecane: 1-4 godziny)
- Waliduj origin postMessage podczas obsługi zdarzeń iframe

## Rozwiązywanie problemów

### Weryfikacja własności domeny nie powiodła się

**Możliwe przyczyny**:

- Rekordy DNS nie zostały propagowane (poczekaj do 48 godzin)
- Nieprawidłowa nazwa lub wartość rekordu TXT
- Rekord TXT dodany do niewłaściwej strefy

**Rozwiązanie**: Sprawdź, czy Twoja konfiguracja DNS pasuje do `verification_instructions` zwróconych podczas dodawania domeny

### Rekordy DNS nie weryfikują się

**Możliwe przyczyny**:

- Rekordy jeszcze nie propagowane
- Nieprawidłowe wartości rekordów
- Brakujące rekordy

**Rozwiązanie**: Wywołaj `GET /company/domains/{id}` lub `verify-dns`, aby zobaczyć, które konkretne rekordy są w toku

### Iframe podpisywania nie ładuje się

**Możliwe przyczyny**:

- Nieprawidłowe ID użytkownika żądania podpisu
- Wygasłe lub anulowane żądanie podpisu
- Content Security Policy blokuje iframe

**Rozwiązanie**: Zweryfikuj status żądania podpisu i sprawdź konsolę przeglądarki pod kątem błędów CSP

### Token JWT wygasł

**Możliwe przyczyny**:

- TTL tokenu zbyt krótki dla przypadku użycia
- Przesunięcie zegara między serwerami

**Rozwiązanie**: Generuj tokeny z odpowiednim wygaśnięciem, rozważ proaktywne odświeżanie tokenów

### Kolory nie są stosowane

**Możliwe przyczyny**:

- Nieprawidłowy format hex (musi być 6-cyfrowy, np. `#0066cc`, nie 3-cyfrowy `#06c`)
- Ustawienie zastosowane na poziomie firmy, ale przestrzeń robocza ma własne nadpisanie

**Rozwiązanie**: Sprawdź najpierw ustawienia przestrzeni roboczej, następnie ustawienia firmy. Użyj `null` na poziomie przestrzeni roboczej, aby wyczyścić nadpisanie.
