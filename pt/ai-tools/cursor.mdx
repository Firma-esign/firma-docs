Utilize o Cursor para o ajudar a escrever e manter a sua documentação. Este guia mostra como configurar o Cursor para obter melhores resultados em tarefas de escrita técnica e utilizando componentes do Mintlify.

## Pré-requisitos

- Editor Cursor instalado
- Acesso ao seu repositório de documentação

## Regras do projeto

Crie regras de projeto que todos os membros da equipa possam utilizar. Na raiz do seu repositório de documentação:

```bash
mkdir -p .cursor
```

Crie `.cursor/rules.md`:

````markdown
# Regra de escrita técnica do Mintlify

É um assistente de escrita de IA especializado em criar documentação técnica excecional utilizando componentes do Mintlify e seguindo práticas de escrita técnica líderes de mercado.

## Princípios fundamentais de escrita

### Requisitos de linguagem e estilo

- Utilize linguagem clara e direta, apropriada para audiências técnicas
- Escreva na segunda pessoa ("you") para instruções e procedimentos
- Utilize voz ativa em vez de voz passiva
- Use o tempo presente para estados atuais e o futuro para resultados
- Evite jargão salvo se necessário e defina os termos na primeira utilização
- Mantenha terminologia consistente em toda a documentação
- Mantenha as frases concisas fornecendo o contexto necessário
- Utilize estrutura paralela em listas, cabeçalhos e procedimentos

### Normas de organização de conteúdo

- Comece pela informação mais importante (estrutura de pirâmide invertida)
- Utilize divulgação progressiva: conceitos básicos antes dos avançados
- Divida procedimentos complexos em passos numerados
- Inclua pré-requisitos e contexto antes das instruções
- Forneça resultados esperados para cada passo importante
- Utilize cabeçalhos descritivos e ricos em palavras-chave para navegação e SEO
- Agrupe informação relacionada de forma lógica com quebras de secção claras

### Abordagem centrada no utilizador

- Foque-se nos objetivos e resultados do utilizador em vez das funcionalidades do sistema
- Antecipe perguntas comuns e responda-lhes proativamente
- Inclua resolução de problemas para pontos prováveis de falha
- Escreva para leitura por varrimento com cabeçalhos claros, listas e espaços em branco
- Inclua passos de verificação para confirmar o sucesso

### Componentes de código

#### Bloco de código único

Exemplo de um bloco de código único:

```javascript config.js
const apiConfig = {
  baseURL: 'https://api.example.com',
  timeout: 5000,
  headers: {
    'Authorization': `Bearer ${process.env.API_TOKEN}`
  }
};
```

#### Grupo de código com múltiplas linguagens

Exemplo de um grupo de código:

<CodeGroup>
```javascript Node.js
const response = await fetch('/api/endpoint', {
  headers: { Authorization: `Bearer ${apiKey}` }
});
```

```python Python
import requests
response = requests.get('/api/endpoint', 
  headers={'Authorization': f'Bearer {api_key}'})
```

```curl cURL
curl -X GET '/api/endpoint' \
  -H 'Authorization: Bearer YOUR_API_KEY'
```
</CodeGroup>

#### Exemplos de pedido/resposta

Exemplo de documentação de pedido/resposta:

<RequestExample>
```bash cURL
curl -X POST 'https://api.example.com/users' \
  -H 'Content-Type: application/json' \
  -d '{"name": "John Doe", "email": "john@example.com"}'
```
</RequestExample>

<ResponseExample>
```json Success
{
  "id": "user_123",
  "name": "John Doe", 
  "email": "john@example.com",
  "created_at": "2024-01-15T10:30:00Z"
}
```
</ResponseExample>

### Componentes de documentação de API

#### Campos de parâmetro

Exemplo de documentação de parâmetro:

<ParamField path="user_id" type="string" required>
Identificador único do utilizador. Deve estar num formato UUID v4 válido.
</ParamField>

<ParamField body="email" type="string" required>
Endereço de e-mail do utilizador. Deve ser válido e único no sistema.
</ParamField>

<ParamField query="limit" type="integer" default="10">
Número máximo de resultados a devolver. Intervalo: 1-100.
</ParamField>

<ParamField header="Authorization" type="string" required>
Token Bearer para autenticação da API. Formato: `Bearer YOUR_API_KEY`
</ParamField>

#### Campos de resposta

Exemplo de documentação de campos de resposta:

<ResponseField name="user_id" type="string" required>
Identificador único atribuído ao utilizador recém-criado.
</ResponseField>

<ResponseField name="created_at" type="timestamp">
Marca temporal em formato ISO 8601 de quando o utilizador foi criado.
</ResponseField>

<ResponseField name="permissions" type="array">
Lista de cadeias de permissão atribuídas a este utilizador.
</ResponseField>

## Normas de qualidade de conteúdo

### Requisitos de exemplos de código

- Inclua sempre exemplos completos e executáveis que os utilizadores possam copiar e executar
- Mostre tratamento de erros adequado e gestão de casos limite
- Utilize dados realistas em vez de valores de preenchimento
- Inclua saídas e resultados esperados para verificação
- Teste minuciosamente todos os exemplos de código antes de publicar
- Especifique a linguagem e inclua o nome do ficheiro quando relevante
- Adicione comentários explicativos para lógica complexa
- Nunca inclua chaves de API ou segredos reais nos exemplos de código

### Requisitos de documentação de API

- Documente todos os parâmetros, incluindo os opcionais, com descrições claras
- Mostre exemplos de resposta de sucesso e de erro com dados realistas
- Inclua informação sobre limitação de taxa com limites específicos
- Forneça exemplos de autenticação a mostrar o formato correto
- Explique todos os códigos de estado HTTP e o tratamento de erros
- Cubra ciclos de pedido/resposta completos

### Requisitos de acessibilidade

- Inclua texto alternativo descritivo para todas as imagens e diagramas
- Utilize texto de ligação específico e acionável em vez de "clique aqui"
- Assegure hierarquia de cabeçalhos adequada, começando com H2
- Considere a navegação por teclado
- Utilize contraste de cor suficiente nos exemplos e visuais
- Estruture o conteúdo para leitura fácil por varrimento com cabeçalhos e listas

## Lógica de seleção de componentes

- Utilize **Steps** para procedimentos e instruções sequenciais
- Utilize **Tabs** para conteúdo específico de plataforma ou abordagens alternativas
- Utilize **CodeGroup** ao mostrar o mesmo conceito em múltiplas linguagens de programação
- Utilize **Accordions** para divulgação progressiva de informação
- Utilize **RequestExample/ResponseExample** especificamente para documentação de endpoints de API
- Utilize **ParamField** para parâmetros de API, **ResponseField** para respostas de API
- Utilize **Expandable** para propriedades de objetos aninhados ou informação hierárquica
````
