## v1.27.0

**Tipo de lançamento:** Patch Release

[Descarregar especificação OpenAPI](/api-reference/v01.27.00/openapi-v01.27.00.json)

### Campos de definições agora documentados nas respostas

Este lançamento documenta campos de definições que a API já devolve. Todas as alterações são aditivas e retrocompatíveis — nenhum comportamento da API foi alterado; apenas a especificação OpenAPI foi atualizada para corresponder ao que os endpoints já enviam.

#### Definições de signing request

O objeto settings devolvido pelos endpoints de signing request (`POST /signing-requests`, `POST /signing-requests/create-and-send`, `GET /signing-requests`, `GET /signing-requests/{id}`, `PUT /signing-requests/{id}`, `PATCH /signing-requests/{id}`) documenta agora quatro campos adicionais:

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `show_qr_code` | boolean or null | Mostrar um código QR na página de assinatura que permite aos signatários continuar no telemóvel. Herda a definição do workspace ou da company quando `null`. |
| `allow_presigning_download` | boolean or null | Permitir aos signatários descarregar o documento original antes de assinar. Herda a definição do workspace ou da company quando `null`. |
| `identity_editable_fields` | string[] or null | Campos de identidade que os signatários podem editar antes de assinar (por exemplo, `["name", "company"]`). `null` desativa a edição da identidade. |
| `notify_identity_change_email` | boolean | Enviar uma notificação por e-mail quando um signatário altera a sua identidade. |

#### Definições de template

O objeto settings devolvido pelos endpoints de template (`POST /templates`, `PUT /templates/{id}`, `PATCH /templates/{id}`, `GET /templates`, `GET /templates/{id}`, `POST /templates/{id}/replace-document`) documenta agora o campo `show_qr_code`. As definições de template cobrem as mesmas opções que as definições de signing request, exceto os campos de identidade do signatário (`identity_editable_fields`, `notify_identity_change_email`), que se aplicam apenas a signing requests.

#### Respostas de cancel e resend

| Endpoint | Novo campo de resposta | Tipo | Descrição |
|----------|------------------------|------|-----------|
| `POST /signing-requests/{id}/cancel` | `emails_sent` | integer | Número de e-mails de notificação de cancelamento enviados aos signatários |
| `POST /signing-requests/{id}/resend` | `recipients_failed` | integer | Número de destinatários cuja notificação de reenvio falhou |

### Novos esquemas

Dois novos esquemas dividem os formatos de definições que diferem entre endpoints, para que cada resposta documente exatamente os campos que devolve:

| Esquema | Descrição |
|---------|-----------|
| `TemplateSettings` | Definições devolvidas pelos endpoints de template. Corresponde ao `SigningRequestSettings` menos os campos de identidade do signatário (`identity_editable_fields`, `notify_identity_change_email`), que se aplicam apenas a signing requests. |
| `LegacyDocumentSettings` | Conjunto reduzido de definições devolvido pelo endpoint descontinuado `POST /documents`. As novas integrações devem utilizar os endpoints de signing request, que devolvem `SigningRequestSettings`. |

### Guia de migração

Sem alterações incompatíveis. Trata-se apenas de acréscimos de documentação que descrevem campos que a API já devolve. As integrações existentes não são afetadas. Se analisa objetos de definições, pode agora confiar nos campos documentados `show_qr_code`, `allow_presigning_download`, `identity_editable_fields` e `notify_identity_change_email`, bem como em `emails_sent` / `recipients_failed` nas respostas de cancel e resend.

```json
{
  "en": {
    "common.finish": "Confirm",
    "signing.decline.confirm": "I Refuse"
  },
  "de": {
    "common.finish": "Bestätigen"
  }
}
```

## v1.25.2

**Tipo de lançamento:** Patch Release

[Descarregar especificação OpenAPI](/api-reference/v01.25.02/openapi-v01.25.02.json)

### Validação mais estrita para substituições de campos de template

Ao criar um signing request a partir de um template, qualquer campo fornecido que não corresponda a um campo desse template é agora rejeitado com **`400` `VALIDATION_ERROR`** em vez de ser silenciosamente ignorado. Anteriormente, tal campo era descartado e o pedido continuava a ser bem-sucedido, pelo que um identificador mal escrito significava simplesmente que o campo nunca aparecia no documento sem erro.

Um campo fornecido corresponde a um campo de template por `template_field_id`, `variable_name` ou `variable_defined_name`. O erro identifica o(s) campo(s) infrator(es):

```json
{
  "error": "The following provided fields do not match any field in the template and cannot be applied: full_name_x. Each provided field must reference an existing template field by 'template_field_id' or 'variable_name'.",
  "code": "VALIDATION_ERROR"
}
```

Aplica-se a `POST /signing-requests` e `POST /signing-requests/create-and-send`.

**Migração:** assegure que cada campo que envia numa criação baseada em template referencia um campo de template existente. Se dependia de campos extra/não correspondidos serem ignorados, remova-os.

### Tipos de campo sem distinção entre maiúsculas e minúsculas

Os valores de `type` de campo são agora normalizados sem distinção entre maiúsculas e minúsculas. `"Initials"`, `"Signature"`, `"TextArea"`, etc., resolvem para o seu tipo canónico (`initial`, `signature`, `text_area`) em vez de serem rejeitados como inválidos. Valores em minúsculas permanecem inalterados.

## v1.25.1

**Tipo de lançamento:** Patch Release

[Descarregar especificação OpenAPI](/api-reference/v01.25.01/openapi-v01.25.01.json)

### Respostas de erro precisas para create-and-send

`POST /signing-requests/create-and-send` devolvia anteriormente um genérico **500** sempre que o passo de send rejeitava um pedido, ocultando o motivo real. Devolve agora o **estado acionável pelo cliente** (`400`, `402` ou `422`) com um `code` legível por máquina e o detalhe relevante:

- **`400` `VALIDATION_ERROR`** — validação do pedido ou do envio falhou (por exemplo, um signatário sem informação obrigatória); a resposta inclui `validation_errors`, `field_errors` ou `details` a identificar o que corrigir.
- **`402` `INSUFFICIENT_CREDITS`** — créditos insuficientes para enviar; `current_credits` reporta o saldo.
- **`422` `RECIPIENT_EMAIL_SUPPRESSED`** *(novo)* — o endereço de e-mail de um destinatário está suprimido (anteriormente devolvido ou marcado como spam) e não pode receber envios.

Falhas genuínas do servidor continuam a devolver `500`. Quando o passo de send falha, o signing request parcialmente criado é revertido, pelo que não fica nenhum rascunho órfão.

**Guia de migração:** nenhuma ação necessária para clientes que já tratam qualquer resposta não-2xx como falha. Se anteriormente tinha lógica especial para o `500` opaco deste endpoint, passe a ramificar com base no estado HTTP / `code`: `400` e `422` significam "corrigir o pedido e voltar a tentar", `402` significa "adicionar créditos".

## v1.25.0

**Tipo de lançamento:** Minor Release

[Descarregar especificação OpenAPI](/api-reference/v01.25.00/openapi-v01.25.00.json)

### Cores de marca do editor

As definições de aparência incluem agora cinco cores de marca adicionais que tematizam os **editores incorporados** (o template e os editores de signing-request que incorpora nas suas próprias páginas):

- **`color_accent`** — cor de acento para destaques, separadores ativos, comutadores e controlos primários no chrome do editor.
- **`color_accent_fg`** — cor de primeiro plano (texto/ícone) apresentada em superfícies de acento.
- **`color_canvas`** — cor de contorno da tela do documento por trás da página.
- **`color_muted`** — cor de superfície esbatida (separadores, estados hover, faixas de comutador).
- **`color_muted_fg`** — cor de texto esbatida.

Estas juntam-se aos seis campos de cor existentes, pelo que os endpoints de definições transportam agora onze campos `color_*`:

- **`GET`/`PUT /company/settings`** e **`GET`/`PUT /workspace/{workspace_id}/settings`** aceitam e devolvem todos os cinco novos campos. Cada um é uma cadeia hexadecimal `#rrggbb` anulável; `null` significa herdar (workspace → company → padrão Firma).
- O objeto **`color_palette`** resolvido devolvido nos payloads de editores incorporados contém agora onze chaves (as seis existentes mais `accent`, `accent_fg`, `canvas`, `muted`, `muted_fg`). Quando não definido, `muted` assume por defeito a cor `card` resolvida e `muted_fg` é derivada para legibilidade, pelo que a paleta é sempre totalmente preenchida.

Os valores do workspace substituem os valores da company; valores não definidos recorrem aos padrões do Firma, pelo que as integrações existentes são renderizadas sem alterações.

**Guia de migração:** nenhuma ação necessária. Todos os cinco campos são aditivos e anuláveis. Para aplicar a marca aos editores incorporados, envie qualquer um dos novos campos `color_*` (hexadecimal `#rrggbb` válido, ou `null` para limpar) ao endpoint de definições da company ou do workspace.

## v1.24.0

**Tipo de lançamento:** Minor Release

[Descarregar especificação OpenAPI](/api-reference/v01.24.00/openapi-v01.24.00.json)

### Chaves de API de teste do workspace

Os workspaces expõem agora uma **chave de API em modo de teste** ao lado da chave live existente. Os pedidos autenticados com uma chave de teste não consomem créditos e produzem signing requests marcados como de teste e com marca de água.

- **`POST /workspaces`** cria agora um par de chaves live + test e devolve tanto `api_key` (live) como o novo campo `test_api_key`.
- **`GET /workspaces/{id}`** e **`GET /workspaces`** devolvem agora tanto `api_key` como `test_api_key`.
- **`POST /workspaces/{id}/api-key/regenerate`** e **`/api-key/expire`** aceitam um `key_type` opcional (`"live"` ou `"test"`, por defeito `"live"`). Regenerar um tipo de chave já não afeta o outro.
- Os workspaces existentes criados via API são preenchidos retroativamente com uma chave de teste.

Todas as alterações são aditivas e retrocompatíveis: consumidores que leem `api_key` continuam a funcionar sem alterações.

**Guia de migração:** nenhuma ação necessária. Para utilizar o modo de teste, leia `test_api_key` de qualquer resposta de criação/leitura de workspace e envie-a como chave `Authorization` para os pedidos que pretende executar em modo de teste.

## v1.22.1

**Tipo de lançamento:** Patch Release

[Descarregar especificação OpenAPI](/api-reference/v01.22.01/openapi-v01.22.01.json)

### Eliminação de domínio de e-mail

`DELETE /company/domains/{id}` e `DELETE /workspace/{workspace_id}/domains/{id}` são agora bem-sucedidos quando o alvo é o domínio principal do workspace ou o seu único domínio. Anteriormente estes devolviam `400 "Cannot delete primary domain"`. Após a eliminação, o envio de e-mail recorre ao remetente por defeito da company, ou ao padrão da plataforma se não estiver definido nenhum. Para utilizar novamente um domínio de envio personalizado, adicione e verifique um novo domínio e depois defina-o como principal.

### Guia de migração

Sem alterações incompatíveis. As respostas `400 "Cannot delete primary domain"` e "cannot delete the only domain" foram removidas de ambos os endpoints `DELETE` de domínio; nestes casos devolvem agora `200`. Integrações que tratavam esse `400` de forma especial podem retirar essa lógica.

## v1.22.0

**Tipo de lançamento:** Minor Release

[Descarregar especificação OpenAPI](/api-reference/v01.22.00/openapi-v01.22.00.json)

### Endpoint Delete Signing Request

Novo endpoint `DELETE /signing-requests/{id}` para remover signing requests não enviados (rascunho). Este endpoint funciona apenas em signing requests que ainda não foram enviados. Para signing requests enviados, utilize o endpoint cancel existente.

**Resposta:**
- `200` com `signing_request_id` e `deleted_on` em caso de sucesso
- `409 ALREADY_SENT` se o signing request tiver sido enviado
- `404` se não encontrado ou já eliminado

### Novo evento de webhook

- `signing_request.deleted` dispara quando um signing request é eliminado via API

### Guia de migração

Sem alterações incompatíveis. O novo método `DELETE` em `/signing-requests/{id}` é puramente aditivo. As integrações existentes não são afetadas.

## v1.21.1

**Tipo de lançamento:** Patch Release

[Descarregar especificação OpenAPI](/api-reference/v01.21.01/openapi-v01.21.01.json)

### Gestão de webhooks do workspace

O endpoint `PATCH /workspaces/{id}` suporta agora dois novos campos para gerir webhooks ao nível do workspace via API:

- `webhook_enabled` (boolean) — ativar ou desativar webhooks ao nível do workspace. Quando definido como `true` pela primeira vez, é gerado automaticamente um segredo de webhook.
- `ignore_company_webhooks` (boolean) — se `true`, os webhooks ao nível da company não disparam para eventos neste workspace.

Anteriormente, estas definições só podiam ser configuradas através da UI do painel.

### Alterações de esquema

- Corpo do pedido **PATCH /workspaces/{id}**: adicionados `webhook_enabled` (boolean, opcional) e `ignore_company_webhooks` (boolean, opcional)
- Resposta **PATCH /workspaces/{id}**: inclui agora `webhook_enabled`, `webhook_secret`, `webhook_secret_rotated_at`, `webhook_secret_created_at` e `ignore_company_webhooks`

### Guia de migração

Sem alterações incompatíveis. As configurações de webhook de workspace existentes não são afetadas. Utilize `PATCH /workspaces/{id}` com `webhook_enabled: true` para ativar webhooks de workspace via API em vez de utilizar o painel.

## v1.21.0

**Tipo de lançamento:** Minor Release

[Descarregar especificação OpenAPI](/api-reference/v01.21.00/openapi-v01.21.00.json)

### Novas definições

- `email_local_part` (string, anulável) — Personalize a parte local (antes do @) dos e-mails de assinatura enviados quando utiliza um domínio personalizado verificado. Disponível em dois níveis:
  - **Definições da company** (`GET/PATCH /company`) — Define o valor por defeito para todos os workspaces. String com valor por defeito `"support"`.
  - **Definições do workspace** (`GET/PUT /workspace/{id}/settings`) — Substitui o valor por defeito da company. String anulável (null herda da company).

Os e-mails são enviados a partir de `{email_local_part}@{your-custom-domain}`. Por exemplo, definir `email_local_part` como `"noreply"` envia e-mails a partir de `noreply@yourdomain.com` em vez de `support@yourdomain.com`.

Validação: alfanuméricos em minúsculas, pontos, hífenes e underscores. Deve começar e terminar com um carácter alfanumérico. 1-64 caracteres. Padrão: `^[a-z0-9]([a-z0-9._-]*[a-z0-9])?$`

### Alterações de esquema

- Esquema **Company**: adicionado `email_local_part` (string, por defeito `"support"`)
- Esquema **Workspace settings**: adicionado `email_local_part` (string, anulável)

### Guia de migração

Sem alterações incompatíveis. As configurações existentes continuam a enviar e-mails a partir de `support@{domain}` por defeito. Defina `email_local_part` ao nível da company ou do workspace para personalizar o endereço de remetente.

## v1.20.0

**Tipo de lançamento:** Minor Release

### Novas definições

- `disable_guided_navigation` (boolean, anulável) — Desativar o deslocamento automático para o próximo campo obrigatório durante a assinatura. Disponível em quatro níveis:
  - **Definições da company** (`GET/PUT /company/settings`) — Define o valor por defeito para todos os workspaces. Boolean (não anulável).
  - **Definições do workspace** (`GET/PUT /workspace/{id}/settings`) — Substitui o valor por defeito da company. Boolean ou null (null herda da company).
  - **Template** (`POST/PATCH /templates`) — Copiado para signing requests criados a partir deste template. Boolean ou null.
  - **Signing request** (`POST/PATCH /signing-requests`) — Substituição de maior prioridade. Boolean ou null (null herda do workspace/company).

A definição segue o mesmo padrão em cascata que `require_otp_verification`: o nível signing request tem prioridade, depois workspace, depois definições da company.

## v1.18.0

**Data de lançamento:** 5 de maio de 2026

### Novas funcionalidades

#### Campos de identidade editáveis

Os signing requests suportam agora edição de identidade por campo e por signatário. Quando configurado, os signatários veem uma caixa de diálogo de confirmação antes de assinar onde podem rever e editar campos de identidade especificados (nome, empresa, título, telefone, morada).

Nova propriedade de definições:

| Propriedade | Tipo | Padrão | Descrição |
|-------------|------|--------|-----------|
| `identity_editable_fields` | `string[]` ou `null` | `null` | Matriz de chaves de campo de identidade que os signatários podem editar. `null` = desativado. |
| `notify_identity_change_email` | `boolean` | `false` | Enviar um e-mail quando um signatário altera a sua identidade |

Chaves de campo válidas: `name`, `company`, `title`, `phone`, `address`

Substituição por destinatário via `identity_editable_fields` no objeto do destinatário:
- `null` — utilizar o valor por defeito do pedido
- `[]` — desativado para este signatário
- `["name", "company"]` — campos personalizados para este signatário

Exemplo:

```json
{
  "settings": {
    "identity_editable_fields": ["name", "company"]
  },
  "recipients": [
    {
      "name": "John Doe",
      "email": "john@example.com",
      "designation": "signer",
      "order": 1,
      "identity_editable_fields": null
    },
    {
      "name": "Jane Smith",
      "email": "jane@example.com",
      "designation": "signer",
      "order": 2,
      "identity_editable_fields": []
    }
  ]
}
```

Neste exemplo, o John herda o valor por defeito do pedido (nome + empresa editáveis), enquanto a Jane tem a edição de identidade explicitamente desativada.

Novo evento de webhook: `signing_request.recipient.identity_changed` está agora disponível como evento de webhook padrão. Subscreva-o na sua configuração de webhook para receber notificações quando um signatário modifica a sua identidade durante a assinatura.

#### Campos de dados pré-preenchidos editáveis

Os campos com dados pré-preenchidos podem agora ser opcionalmente editáveis pelos signatários. Anteriormente, todos os campos com `prefilled_data` definido eram automaticamente bloqueados como apenas de leitura. Uma nova propriedade `prefilled_editable` permite controlar este comportamento.

Quando `prefilled_editable` é `true`, o valor do campo continua a ser preenchido automaticamente a partir dos dados do destinatário, mas o signatário pode modificá-lo durante o processo de assinatura.

| Propriedade | Tipo | Padrão | Descrição |
|-------------|------|--------|-----------|
| `prefilled_editable` | `boolean` | `false` | Quando true, os signatários podem editar valores pré-preenchidos |

Disponível em:
- Campos de template (`PATCH /templates/{id}`, `PUT /templates/{id}`)
- Campos de signing request (`POST /signing-requests`, `POST /signing-requests/create-and-send`)
- Objetos de resposta de campo (`GET /signing-requests/{id}`)

Exemplo — criar um signing request com um campo pré-preenchido editável:

```json
{
  "fields": [
    {
      "type": "text",
      "recipient_id": "temp_1",
      "page_number": 1,
      "x": 10,
      "y": 20,
      "width": 30,
      "height": 5,
      "format_rules": {
        "prefilledData": "full_name",
        "prefilledEditable": true
      }
    }
  ]
}
```

#### Validação da fonte de valor do campo

A API valida agora que os campos utilizam apenas uma fonte de valor de cada vez. Pedidos que combinem `read_only_value`, `prefilled_data` e `value` explícito no mesmo campo devolverão um `400 VALIDATION_ERROR`. Esta imposição aplica-se a:

- `POST /signing-requests`
- `POST /signing-requests/create-and-send`
- `PATCH /templates/{id}` (criação e atualização de campo)
- `PUT /templates/{id}` (upsert de campo)

#### Precedência de valor

Quando `prefilled_editable` é `true` e tanto um `value` explícito como `prefilled_data` estão definidos num campo, o valor explícito tem precedência sobre os dados do destinatário preenchidos automaticamente.

### Guia de migração

Sem alterações incompatíveis. As novas funcionalidades são opt-in:
- A edição de identidade está desativada por defeito (`identity_editable_fields: null`). Defina como uma matriz de chaves de campo para ativar.
- Os campos com dados pré-preenchidos existentes permanecem bloqueados (apenas de leitura) por defeito. Defina `prefilled_editable: true` em `format_rules` para os tornar editáveis.

## v1.16.0

### Novas funcionalidades

#### Descarregar documento de signing request

Um novo endpoint obtém um URL de download para o documento de um signing request. Para signing requests concluídos, o URL aponta para o PDF final assinado. Para pedidos em curso, cancelados, recusados ou expirados, o URL aponta para um PDF de instantâneo parcial que captura o estado do documento no momento da última ação do signatário.

**Endpoint:** `GET /signing-requests/{id}/download`

**Campos de resposta:**

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `status` | `string` | Estado do signing request: `finished`, `in_progress`, `cancelled`, `declined` ou `expired` |
| `is_partial` | `boolean` | `true` quando o documento é um instantâneo parcial (nem todos os signatários concluíram) |
| `download_url` | `string` | URL pré-assinado para o documento PDF |
| `generated_at` | `date-time \| null` | Quando o documento foi gerado pela última vez |
| `expires_at` | `date-time` | Quando o `download_url` expira |

**Comportamento por estado:**

| Estado do signing request | Resposta |
|---------------------------|----------|
| Ainda não enviado | `409 Conflict` — download indisponível |
| Enviado, geração em curso | `503 Service Unavailable` com cabeçalho `Retry-After` |
| Enviado, pelo menos um signatário concluiu | `200` com `is_partial: true` |
| Cancelado / Recusado / Expirado | `200` com `is_partial: true` (instantâneo do último estado conhecido) |
| Concluído (todos os signatários terminaram) | `200` com `is_partial: false` |

Os instantâneos parciais são gerados a pedido e colocados em cache. O primeiro pedido pode devolver `503` com um cabeçalho `Retry-After` enquanto o PDF está a ser gerado — tente novamente após o intervalo indicado.

**Expiração do URL:** O `download_url` é um URL pré-assinado. Chame o endpoint novamente após `expires_at` para obter um URL atualizado.

#### Definição de marca de água de download parcial

Uma nova definição de workspace `show_partial_watermark` controla se os downloads parciais de PDF incluem uma marca de água diagonal "IN PROGRESS — NOT YET EXECUTED". Desativada por defeito. Pode ser definida ao nível da company (padrão para todos os workspaces) ou substituída por workspace.

**Endpoint:** `PUT /workspace/{workspace_id}/settings`

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `show_partial_watermark` | `boolean \| null` | `true` para ativar a marca de água, `false` para desativar, `null` para herdar o padrão da company |

### Guia de migração

**De v1.15.3 para v1.16.0:**

Esta é uma alteração aditiva não incompatível. As integrações existentes continuam a funcionar sem modificações.

- O endpoint de download é totalmente opt-in. Nenhum comportamento existente é alterado.
- Chame `GET /signing-requests/{id}/download` — trate `503` com `Retry-After` até o documento estar pronto.

Para campos não ligados a uma definição de campo personalizado, `variable_defined_name` é `null`.

#### `variable_defined_name` como entrada para correspondência de campo

Ao criar um signing request a partir de um template via `POST /signing-requests/create-and-send`, pode agora utilizar `variable_defined_name` para direcionar substituições de campo em vez do `variable_name` interno:

```json
    { "variable_defined_name": "artist_name", "read_only_value": "John Smith" }
  ]
}
```

Tanto `variable_name` como `variable_defined_name` são aceites. `variable_name` tem precedência quando ambos são fornecidos.

### Sem alterações incompatíveis

Este é um patch release não incompatível. Todo o comportamento existente da API está inalterado.

#### Lógica de campo condicional

Os campos suportam agora regras condicionais tanto para o estado **required** como para a **visibility**. Estão disponíveis duas novas propriedades anuláveis em objetos de campo:

- `required_conditions` — quando definido, substitui a flag `required` estática. O campo é obrigatório apenas quando as condições são avaliadas como verdadeiras com base noutros valores de campo.
- `visibility_conditions` — quando definido, o campo é ocultado a menos que as condições sejam avaliadas como verdadeiras. Campos ocultos não são validados na submissão.

As condições utilizam uma estrutura de grupos aninhada com operadores lógicos:

| Esquema | Descrição |
|---------|-----------|
| `ConditionSet` | Contentor de nível superior com operador `logic` (`and`/`or`) e uma matriz de `groups` |
| `ConditionGroup` | Contém uma matriz de `conditions`, combinada utilizando o oposto do operador lógico do pai |
| `Condition` | Avalia um único campo por `field_id`, `operator` e `value` opcional |

**Operadores suportados:** `is_filled`, `is_empty`, `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `less_than`, `greater_than_or_equal`, `less_than_or_equal`

**Exemplo — tornar um campo obrigatório apenas quando outro campo está preenchido:**
```json
{
  "required_conditions": {
    "logic": "and",
    "groups": [
      {
        "conditions": [
          {
            "field_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",

          }
        ]
      }
    ]
  }
}
```

- Seis novos campos de cor anuláveis (`color_primary`, `color_primary_fg`, `color_background`, `color_foreground`, `color_card`, `color_border`) em Company e Workspace Settings
- Novo esquema `ColorPalette` para cores resolvidas da experiência de assinatura
- As cores seguem uma cascata de 3 níveis: padrões Firma -> Company -> Workspace

### Migrar de v1.19.0 para v1.20.0
- Sem alterações incompatíveis
- Nova definição `disable_guided_navigation` disponível aos níveis de company, workspace, template e signing request
- Segue o mesmo padrão em cascata que `require_otp_verification`: signing request -> workspace -> company (primeiro não nulo vence)
- Assume por defeito `false` (deslocamento automático ativado), preservando o comportamento existente

### Migrar de v1.20.0 para v1.21.0
- Sem alterações incompatíveis
- Nova definição `email_local_part` em Company (`GET/PATCH /company`) e Workspace Settings (`GET/PUT /workspace/{id}/settings`)
- O valor do workspace tem prioridade sobre o valor da company; o padrão é `"support"`
- Só se aplica quando está configurado um domínio personalizado verificado
