Todos os signing requests no Firma têm um audit trail completo. Cada visualização de documento, interação com campo, assinatura, recusa e ação de admin é registada com timestamps, identidade do ator e endereços IP.

Casos de uso comuns:

- **Conformidade e evidência legal** — provar quem assinou o quê, quando e a partir de onde
- **Depuração de problemas do signatário** — ver exatamente o que um destinatário fez durante a sua sessão de assinatura
- **Reporte interno** — acompanhar ações de admin como criação, edições e cancelamentos

O audit trail captura duas categorias de eventos:

- **Eventos de signatário** — visualizações de documento, interações com campos, assinatura, recusa, downloads de certificado
- **Eventos de admin** — criação, edições, envio, cancelamento, reenvio (via painel ou API)

Cada evento inclui um timestamp, uma descrição legível por humanos, a identidade do ator e (para eventos de signatário) o endereço IP do signatário.

## Ver o audit trail no painel

Abra qualquer signing request no painel Firma e clique no separador **Audit Trail**. Verá uma linha temporal cronológica de todos os eventos, com ações de signatário e ações de admin visualmente distinguidas. Eventos repetitivos consecutivos (como scrolls ou interações repetidas com campos) são automaticamente condensados com um indicador de contagem.

## Obter o audit trail via API

Utilize o endpoint audit para obter o histórico completo de eventos de um signing request:

```
GET /signing-requests/{id}/audit
```

### curl

```bash
curl -X GET "https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit" \
  -H "Authorization: YOUR_API_KEY"
```

### Node.js

```js
const response = await fetch(
  `https://api.firma.dev/functions/v1/signing-request-api/signing-requests/${signingRequestId}/audit`,
  {
    headers: {
      'Authorization': process.env.FIRMA_API_KEY,
    },
  }
);

const { results } = await response.json();
console.log(results);
```

### Python

```python
import os
import requests

response = requests.get(
    f"https://api.firma.dev/functions/v1/signing-request-api/signing-requests/{signing_request_id}/audit",
    headers={
        "Authorization": os.environ["FIRMA_API_KEY"],
    },
)

results = response.json()["results"]
```

### Campos de resposta

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `id` | `uuid` | ID do evento |
| `timestamp` | `datetime` | Quando o evento ocorreu |
| `source` | `string` | `"signer"` ou `"admin"` |
| `event` | `string` | Identificador do tipo de evento (ver [referência de eventos](#event-reference) abaixo) |
| `description` | `string` | Descrição do evento legível por humanos |
| `actor` | `object \| null` | Quem executou a ação (ver abaixo) |
| `ip_address` | `string \| null` | Endereço IP do signatário (apenas eventos de signatário) |
| `details` | `object \| null` | Metadados específicos do evento |
| `condensed_count` | `integer` | Presente quando eventos idênticos consecutivos do mesmo ator estão agrupados |

## Compreender as origens dos eventos

O campo `source` indica se um evento provém de um signatário ou de um admin, e o campo `actor` tem uma forma diferente em cada caso.

### Eventos de signatário

Eventos com `source: "signer"` são ações executadas por destinatários durante o fluxo de assinatura. O objeto `actor` contém o `name` e o `email` do signatário. O campo `ip_address` é preenchido.

```json
{
  "source": "signer",
  "actor": { "name": "Alice Johnson", "email": "alice@example.com" },
  "ip_address": "203.0.113.42"
}
```

### Eventos de admin

Eventos com `source: "admin"` são ações executadas via painel ou API. O objeto `actor` contém um campo `type` (`"api_key"` ou `"user"`) e, opcionalmente, um `user_id`. O campo `ip_address` é `null`.

```json
{
  "source": "admin",
  "actor": { "type": "api_key" },
  "ip_address": null
}
```

```json
{
  "source": "admin",
  "actor": { "type": "user", "user_id": "usr_abc123" },
  "ip_address": null
}
```

## Acompanhamento em tempo real via webhooks

Se precisa de notificação em tempo real de eventos de signatário em vez de consultar o endpoint audit, o Firma suporta webhooks. Consulte o [guia de Webhooks](webhooks) para instruções de configuração.

## Referência de eventos

### Ciclo de vida do documento

| Evento | Descrição | Tem `details` |
|--------|-----------|---------------|
| `document_opened` | Abriu o documento | Não |
| `document_viewed` | Visualizou o documento | Não |
| `document_saved` | Guardou valores de campo | Não |
| `document_finished` | Concluiu a assinatura | Não |
| `document_signed` | Assinou o documento | Não |
| `document_hidden` | Saiu da página | Não |
| `document_visible` | Regressou à página | Não |
| `document_closed` | Fechou o documento | Não |

### Assinatura e verificação

| Evento | Descrição | Tem `details` |
|--------|-----------|---------------|
| `signature_finalized` | Assinatura finalizada | Não |
| `signing_declined` | Recusou assinar | Não |
| `terms_accepted` | Aceitou os termos e condições | Não |
| `otp_verified` | Verificou a identidade via OTP | Não |
| `certificate_download` | Descarregou o certificado | Não |

### Interações com campos

As interações com campos são registadas como eventos `field_interaction`. O objeto `details` contém a ação específica e os metadados do campo.

| Ação | Descrição |
|------|-----------|
| `field_focused` | Selecionou um campo |
| `field_completed` | Preencheu um campo pela primeira vez |
| `field_modified` | Modificou um valor de campo existente |
| `field_cleared` | Limpou o valor de um campo |
| `field_blur` | Terminou de editar um campo |

O objeto `details` para interações com campos inclui:

| Campo | Tipo | Descrição |
|-------|------|-----------|
| `field_type` | `string` | Tipo de campo (signature, text, date, etc.) |
| `action` | `string` | A ação específica indicada na tabela acima |
| `interaction_count` | `integer` | Número de vezes que este campo foi interagido |
| `value_length` | `integer` | Comprimento do valor do campo (quando aplicável) |
| `time_spent_ms` | `integer` | Tempo despendido no campo em milissegundos (quando aplicável) |

### Navegação

| Evento | Descrição | Tem `details` |
|--------|-----------|---------------|
| `page_viewed` | Visualizou uma página específica | Sim — `page_number` |
| `document_scrolled` | Fez scroll para uma página | Sim — `page_number` |
| `navigation_action` | Navegou no documento | Sim — `action` (`next_page`, `prev_page`, `go_to_page`) |
| `zoom_changed` | Alterou o nível de zoom | Não |
| `modal_interaction` | Abriu ou fechou uma caixa de diálogo | Sim — `modal_type`, `action` |

### Atividade de admin

Eventos de admin utilizam `event: "activity"` com um campo `description` legível por humanos que descreve a ação realizada (por exemplo, "Created signing request via API", "Sent signing request", "Cancelled signing request").
