- **Розподілений доступ:** Користувачі, шаблони та активність обмежені окремим робочим простором.
- **Відстеження використання:** Дивіться конверти, надіслані на робочий простір, для чистого звітування.
- **Безпека за задумом:** Відсутність міжпросторового доступу до документів або даних.

Це полегшує надання кожному клієнту або команді власного приватного середовища підпису, тоді як ви керуєте всім через один API.

## Основні ендпоінти

Основні ендпоінти (див. API-довідник для повних деталей):

- POST `/workspaces` — створити робочий простір
- GET `/workspaces/{workspace_id}` — отримати деталі робочого простору
- PUT `/workspaces/{workspace_id}` — оновити робочий простір
- GET `/workspaces` — отримати список робочих просторів (пагінація)
- GET `/workspace/{workspace_id}/settings` — отримати налаштування робочого простору
- PUT `/workspace/{workspace_id}/settings` — оновити налаштування робочого простору

<Note>
  Дивіться посібник з [лімітів запитів](rate-limits).
</Note>


Ви також можете знайти окремі довідкові сторінки в `api-reference/v01.15.00/workspaces`:

- `api-reference/v01.15.00/workspaces/create-a-new-workspace`
- `api-reference/v01.15.00/workspaces/list-workspaces`
- `api-reference/v01.15.00/workspaces/update-a-workspace`

Автентифікація

Усі запити мають містити заголовок Authorization з вашим API-ключем:

```
Authorization: Bearer YOUR_API_KEY
```

Примітка: ніколи не розкривайте ваш основний API-ключ у браузерах кінцевих користувачів. Для операцій робочого простору, які потрібно ініціювати з frontend, викликайте ваш backend, який містить ключ.

## Створення робочого простору

Ендпоінт: POST `/workspaces`

Обов’язкове тіло: `{ "name": "string" }`

Приклад (curl):

```bash
curl -X POST "https://api.firma.dev/functions/v1/signing-request-api/workspaces" \
  -H "Authorization: Bearer $FIRMA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Acme Corp Workspace" }'
```

Успішна відповідь (201) повертає ресурс `Workspace`, включаючи `api_key` (live) та `test_api_key`. Приклад структури:

```json
{
  "id": "workspace_123",
  "name": "Acme Corp Workspace",
  "protected": false,
  "api_key": "firma_live_…",
  "test_api_key": "firma_test_…",
  "created_date": "2025-09-04T12:00:00Z",
  "updated_date": "2025-09-04T12:00:00Z"
}
```

`api_key` — це live-ключ робочого простору, а `test_api_key` — його test-ключ. Дивіться [Автентифікацію](authentication) для інформації про тестовий режим.

Приклад сервера — Node (fetch):

```js
import fetch from 'node-fetch'

export async function createWorkspace(name) {
  const resp = await fetch('https://api.firma.dev/functions/v1/signing-request-api/workspaces', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ name })
  })
  if (!resp.ok) throw new Error(`create workspace failed: ${resp.status}`)
  return resp.json()
}
```

Приклад сервера — Python (requests):

```py
import os
import requests

def create_workspace(name):
    url = 'https://api.firma.dev/functions/v1/signing-request-api/workspaces'
    resp = requests.post(url, headers={
        'Authorization': f"Bearer {os.environ['FIRMA_API_KEY']}",
        'Content-Type': 'application/json'
    }, json={ 'name': name })
    resp.raise_for_status()
    return resp.json()
```

## Отримання робочого простору

Ендпоінт: GET `/workspaces/{workspace_id}`

Приклад (curl):

```bash
curl "https://api.firma.dev/functions/v1/signing-request-api/workspaces/workspace_123" \
  -H "Authorization: Bearer $FIRMA_API_KEY"
```

Відповідь: ресурс `Workspace` (див. приклад вище).

## Оновлення робочого простору

Ендпоінт: PUT `/workspaces/{workspace_id}` — повна заміна ресурсу. Обов’язкове тіло, як у `CreateWorkspace` (наприклад, `{ "name": "New name" }`).

Приклад (curl):

```bash
curl -X PUT "https://api.firma.dev/functions/v1/signing-request-api/workspaces/workspace_123" \
  -H "Authorization: Bearer $FIRMA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "Acme Corp - East Coast" }'
```

Приклад сервера — Node (fetch):

```js
export async function updateWorkspace(workspaceId, name) {
  const resp = await fetch(`https://api.firma.dev/functions/v1/signing-request-api/workspaces/${workspaceId}`, {
    method: 'PUT',
    headers: {
      'Authorization': `Bearer ${process.env.FIRMA_API_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ name })
  })
  if (!resp.ok) throw new Error(`update workspace failed: ${resp.status}`)
  return resp.json()
}
```

## Список робочих просторів

Ендпоінт: GET `/workspaces`

Підтримує параметри запиту `page` та `page_size`. Приклад:

```bash
curl "https://api.firma.dev/functions/v1/signing-request-api/workspaces?page=1&page_size=20" \
  -H "Authorization: Bearer $FIRMA_API_KEY"
```

Структура відповіді містить `results` (масив `Workspace`) та метадані `pagination`.

## RBAC та найкращі практики для API-ключів

- Використовуйте API-ключі з обмеженою областю, де можливо: тримайте головний API-ключ лише на сервері та створюйте API-ключі для кожного робочого простору для інтеграційних партнерів, якщо ваша платформа це підтримує.
- Ніколи не повертайте API-ключі до браузера. Для операцій, ініційованих кінцевими користувачами, викликайте ваш backend, який виконує API-виклик до Firma.
- Використовуйте ідемпотентність, де підтримується (для ендпоінтів, що приймають `Idempotency-Key`), щоб уникнути подвійного створення від повторних спроб.

## Граничні випадки та усунення несправностей

- Дубльовані назви: назви робочих просторів зручні для відображення, але можуть не бути унікальними; використовуйте `id` для програмних посилань.
- Захищені робочі простори: прапорець `protected` вказує на робочі простори, що належать системі, які не можна видалити або змінити через звичайні потоки.
- Ліміти запитів та помилки: аналізуйте не-2xx відповіді за `{ error, code }` і показуйте змістовні повідомлення операторам.
- **Перевищення ліміту запитів (429)**: Моніторте заголовок `X-RateLimit-Remaining` і реалізовуйте експоненціальне відкладання для повторних спроб.
